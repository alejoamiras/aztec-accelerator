1. **Title** — Monolithic `.setup` bootstrap.
2. **Smell** — **Long Method**. Fowler bloater: the Tauri `.setup` closure is a single 200+ line routine that also contains multiple nested closures and unrelated lifecycle responsibilities.
3. **Maintenance impact** — **structural**; blast radius spans tray, windowing, crash recovery, HTTPS, HTTP server startup, diagnostics, and updater polling across this bootstrap path. This is a hot change surface because any desktop lifecycle change lands here.

## Finding 2 — Bootstrap Knows the Server’s Internal Shape
1. **Title** — Desktop bootstrap is tightly coupled to server-core internals.
2. **Smell** — **Inappropriate Intimacy**. The Tauri layer directly knows and fills the server core’s public state fields and callback slots instead of depending on a constructor/factory boundary.
3. **Maintenance impact** — **architectural**; blast radius reaches the Tauri bootstrap, Tauri commands, and headless server bootstrap. Any change to the core state contract ripples across multiple packages.
4. **Concrete evidence** — The server core exposes field-level state and callback slots publicly at [packages/accelerator/core/src/server.rs:84-109](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/server.rs:84). The Tauri bootstrap constructs that shape directly at [packages/accelerator/src-tauri/src/main.rs:345-367](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/main.rs:345), filling `bundled_version`, `app_version`, `https_bound`, `config`, `auth_manager`, `prove_semaphore`, `on_status`, `on_versions_changed`, and `show_auth_popup`. The command layer then keeps a full shared `AppState` alias at [packages/accelerator/src-tauri/src/commands.rs:22-24](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/commands.rs:22) and clones the whole state to start HTTPS at [commands.rs:139-166](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/commands.rs:139). A second production bootstrap knows the same internal construction pattern in the headless binary at [packages/accelerator/server/src/main.rs:62-75](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/server/src/main.rs:62).
5. **Why it harms future change** — If the core server evolves its state model, ownership model, or callback contract, that change cannot stay local to core. Desktop startup, command glue, and headless startup all need synchronized edits because they know the exact field layout and which optional slots must be populated.
6. **Smallest safe refactoring** — **Move Function** / constructor extraction: introduce core-side factories such as `HeadlessState::new(...)` and `AppState::with_gui_callbacks(...)`, or a builder that accepts purpose-level dependencies instead of exposing every field.
7. **What disappears** — Field-by-field knowledge of server internals in UI entrypoints, plus part of the cross-package ripple when `AppState` or `HeadlessState` changes.
8. **Instances** — [packages/accelerator/core/src/server.rs:84-109](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/server.rs:84), [packages/accelerator/src-tauri/src/main.rs:345-367](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/main.rs:345), [main.rs:379](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/main.rs:379), [packages/accelerator/src-tauri/src/commands.rs:22-24](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/commands.rs:22), [commands.rs:139-166](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/commands.rs:139), [packages/accelerator/server/src/main.rs:62-75](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/server/src/main.rs:62).

## Finding 3 — One Command Module Changes for Many Reasons

2. **Smell** — **Divergent Change**. This module changes for config persistence, OS autostart, authorization UI response, verified-site lookup, Safari/TLS enablement, and updater prompt flow.

5. **Why it harms future change** — A change to one concern lands in the same file and import surface as unrelated command families, so the command boundary becomes a catch-all. 7. **What disappears** — A single file with many unrelated reasons to change, mixed imports/state aliases, and a growing all-purpose command boundary.
