HighF-02 An approved website address is just loose text — and one door skips the lock
Primitive Obsessionfound by bothconfidence: highfix: days

What it is. When a website asks the accelerator to make a proof, the app checks that site’s address (its “origin”, like https://app.example.com) against an approved list. First it tidies each address into one standard spelling so comparisons are reliable. But approved addresses are stored as plain text, and the rule “these are always tidied first” lives only in code comments. One entry point — the headless server used in CI — skips the tidying step and stores whatever it’s handed.

What gets harder to change. Every new way an address can enter the system is a new chance to forget the tidy-up step, after which approvals silently mismatch. A reviewer can’t trust the value; they have to trace every use and ask “is this one already tidied?” The compiler can’t help — it’s all just text.

The fix. Make a dedicated CanonicalOrigin type whose only way to exist runs the tidy-up. Then holding an un-tidied address becomes impossible, the compiler enforces what comments currently beg for, and a whole one-off “repair” routine deletes itself.

Technical trace

Scope: the bypass is in the headless server binary (a shipped tarball, but documented for CI/e2e); the desktop app loads origins via the canonicalizing config::load. So this is a type-invariant / future-footgun gap, not an attacker hole in the GUI product.

Instances: core/src/authorization.rs:21-58, 101-145 · core/src/config.rs:41-55, 83-130 · core/src/server/auth.rs:35-52 · server/src/main.rs:43-57 (raw ALLOWED_ORIGINS env strings → approved_origins, no url::Url pass).

Refactor: Replace Primitive with Object — CanonicalOrigin(String) + #[serde(try_from="String")]; store Vec<CanonicalOrigin>. Deletes migrate_approved_origins and the headless bypass. Note: finder C3 flagged this; finder C5 dismissed it — both rebuttals + a ground-truth read sided with C3.

HighF-01 The server’s shared settings are a “fill-in-the-blanks” form everyone copies by hand
Data Clumps · nullable bagfound by bothconfidence: highfix: days

What it is. The server’s shared state (a concurrency lock, the version, some callbacks) lives in a structure where almost every slot is “optional — may be empty.” Each of the three places that start the server fills this form in by hand, slot by slot, then uses a “…and default the rest” shortcut.

What gets harder to change. Add one new required setting and you must edit every startup spot and several tests. Miss one, and the shortcut quietly leaves it empty — so it fails much later at runtime instead of immediately at compile time.

The fix. Add real constructors (headless(…), desktop(…)) that build the form correctly in one place, and make the always-needed slots non-optional. New setting → one edit, and the compiler checks every caller for you. (The same already-extracted helpers exist elsewhere in this repo — this just extends the habit.)

Technical trace

Prod instances: struct def core/src/server.rs:83-119 · server/src/main.rs:62-75 · src-tauri/src/main.rs:345-367.

Corroborating (tests): ~5 hand-assembly sites in the server.rs test module (>L210) — the missing constructor taxes tests too. (The coordinator first listed these as prod debt; the verifier relabeled them.)

Refactor: Extract Factory + make required deps non-Option; isolate GUI callbacks into a small desktop-extras type. Deletes repeated Some(Arc::new(Semaphore::new(1))) / env!("CARGO_PKG_VERSION") literals and the ..Default::default() omission trap.

HighF-05 The SDK’s instruction manual is hand-copied in 5 places — and they disagree
Duplicate Code · Divergent Changefound by bothconfidence: highfix: hours · cheapest

What it is. The public “shape” of the SDK (its types and methods) is written out by hand in five places — the code, the export list, the README, a migration guide, and a bundled skill doc. They’ve drifted apart. Most importantly, the README still documents the old status type that was deliberately replaced — the new one prevents a whole category of mistakes, and the README is currently teaching the mistake.

What gets harder to change. Change the API once and you must update five documents in lockstep. Miss one and your users get contradictory instructions — or they follow the README and write code that breaks.

The fix. Make the export list the single source of truth, correct today’s drift, and add a small test that fails the build if the docs and the code disagree. This is the cheapest win on the list and it’s the most user-visible one.

Technical trace

Drift, concretely: README:88-101 shows the obsolete flat interface AcceleratorStatus vs the shipped discriminated union accelerator-prover.ts:56-92; the barrel index.ts:1-8 omits AcceleratorProtocol (MIGRATION says it’s exported → the documented import fails); README’s method table omits setForceLocal (which the playground actually calls).

Refactor: barrel as canonical surface + export/doc-sync test (or generated API snippet). Minor sub-item (lower confidence): the skill’s phase table omits a denied case of the separate AcceleratorPhase type.

HighF-03 The desktop app’s startup is one 203-line do-everything block
Long Method · Divergent Changefound by bothconfidence: highfix: days

What it is. When the desktop app launches, a single ~203-line block does almost everything: builds the tray icon, wires up callbacks, starts the web servers, handles “port already in use,” and kicks off the update checker — all inline, in one function.

What gets harder to change. To add or reorder one startup step, you have to read and safely edit a 200-line function that’s juggling about eight shared things at once (and re-cloning them repeatedly). Easy to disturb something unrelated.

The fix. Split it into named steps — build_tray, wire_callbacks, spawn_servers, spawn_update_poller — so the launch function just lists the steps in order.

Technical trace

Instance: the .setup closure at src-tauri/src/main.rs:260-462 — esp. 266-305, 316-367, 396-441 (inline AddrInUse / redundant-instance policy), 448-459. Extract Function into bootstrap phases leaves .setup as orchestration only and removes most clone-stutter.

Med-highF-04 One file does eight different jobs around the prover binary

What it is. A single module owns everything about the bb prover binary: deciding the version, naming the platform, building download URLs, downloading, hash-checking, installing, and cleaning up old versions — about 18 functions.

What gets harder to change. A tweak to any one concern (say, how the cache is laid out) means reopening a file full of unrelated concerns and re-testing all of them together.

MedF-07 Three certificate file paths always travel together as loose arguments
Data Clumpsfound by bothconfidence: medfix: hours · cheap

What it is. Three file paths — the CA cert, the leaf cert, and the leaf key — always move together through the TLS code, passed as three separate arguments. They’re all the same type, so accidentally swapping two of them compiles fine and silently breaks.

What gets harder to change. Rename a cert file or add a fourth artifact, and you must edit every place that spells out the trio (existence checks, writers, loaders, cleanup, promotion).

The fix. Bundle them into one CertPaths object with live() / staged() constructors. Swaps become impossible and the filenames live in one place.

Technical trace

Instances: src-tauri/src/certs.rs:20-33, 90-128, 201-205, 261-286 — all &Path (compiler-invisible arg swaps); rotate() re-declares the trio as staged *.new files. Introduce Parameter Object also folds in the (otherwise over-asserted) rotate() ordering coupling.

MedF-06 The SDK talks to one server using two different HTTP libraries
Divergent Change · Temporal Couplingfound by bothconfidence: medfix: days

What it is. To reach the same local accelerator, the SDK uses two different HTTP tools — the browser’s native fetch for the health check, and a library called ky for the actual proof — each with its own timeout, retry, and error behavior.

What gets harder to change. Want to add an auth header, a proxy, a path prefix, or smarter retries? You now have to do it twice, in two different styles, and keep them consistent.

The fix. Pull all server communication into one AcceleratorTransport class that owns URLs, protocol choice, the status cache, and a single error model.

Technical trace

Instances: sdk/src/lib/accelerator-prover.ts:163-168, 204-229, 235-374, 414-423 — /health via fetch+Promise.any+manual retry; /prove via ky; #acceleratorProtocol + #statusCache mutated in several spots. Extract Class shrinks the 122-line #probeAndParseHealth substantially.

MedF-08 Two functions share a hidden handshake about proving status
Temporal Couplingfound by claudeconfidence: medfix: hours

What it is. While making a proof, the server reports a status (“Proving”, “Downloading”). Two functions share that job: one sets “Proving”, the other may overwrite it with “Downloading” and then has to remember to set it back to “Proving” before returning — an unwritten handshake.

What gets harder to change. Add a new phase or move the download step and you must preserve an ordering rule that neither function’s signature mentions. Easy to miss.

The fix. Let the main prove() function own the whole status sequence (it already sets up a status guard); make the helper just return data, no side effects.

Technical trace

Instances: core/src/server/prove.rs:64-96, 160-165; characterization test at core/src/server.rs:617-685. resolve_version() restores Proving at prove.rs:94-96 purely so the caller’s state machine still works.

LowF-09 The “start HTTPS” code is copy-pasted in two places

What it is. The code that starts the HTTPS server — load the cert config, then spawn it — is duplicated in two spots: once at launch, once when you toggle HTTPS in settings.

Patterns worth noticing

1. Rules kept alive by comments, not by types

The three highest-priority findings (F-01, F-02, F-05) are one habit in disguise: a critical guarantee — “this state is complete”, “this origin is tidied”, “the docs match the code” — is maintained by convention and prose instead of by a type or a single source of truth. The cure is the same each time: make the rule un-breakable by construction. Do all three and the top of this report retires together.

2. The recent refactor left a little parallel wiring

F-01 and F-09 are both “the same assembly spelled twice” across the freshly-split core / desktop / headless boundary. The extraction was clean in structure; it just left some shared construction wanting a shared home.

3. This codebase is well looked-after

The reviewers repeatedly found things that were already fixed (a proper version type, extracted download/verify/install helpers, the safe status union, dedup helpers) and correctly left them alone. Low finding-count, zero findings refuted on re-check — a healthy baseline.

What we deliberately did not flag

For context: a few things the finders raised were checked and dropped — a single-use config adapter (not real duplication), a tiny one-implementation trait (real but trivially minor), and a suspected “errors get hidden” bug in the SDK that turned out to be correct (the error cases are already handled; the catch-all is genuinely the “both probes failed” case). Honest negatives matter as much as the findings.
