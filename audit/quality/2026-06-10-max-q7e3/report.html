Cheapest fixes first (land these four — hours each, both models agree)

F-01 extract the Safari-HTTPS bring-up sequence · F-02 move SDK public types to types.ts · F-03 a ProveError type for the 11 hand-built error tuples · F-04 break up the 154-line .setup() blob. F-01 is the one to do today — its failure mode is the SEC-08 “M1” bug fixed earlier this session.

Ship together

F-05 + F-06 + F-11 all live in the one 440-line SDK hotspot file (accelerator-prover.ts) — refactor them in a single pass, not three. F-01 + F-04 both touch main.rs startup; sequence them so they don't collide.

The one theme worth internalizing

The dominant smell isn't size or copy-paste — it's invariants kept true by a // must… comment instead of by a type or structure. Five top findings (F-01, F-06, F-09, F-10, F-13) are that exact shape. It already cost you once: F-01 is the root cause of the M1 bug. The codebase's (genuinely good) habit of heavy SEC-xx/Q-xx why-comments is quietly doing the job a type should — so the highest-traffic of these are worth converting into real seams (guard objects, owned state transitions, template methods).

Findings

HighF-01Safari-HTTPS bring-up is a comment-enforced sequence, copied across two paths
both ×3Temporal Coupling + Duplicate Codestructural · hotfix: hours

What it is. Turning on Safari HTTPS needs a fixed order of cert steps — delete the old key → make certs → trust → load → start the server. That order is hand-written in two separate places: the app-launch path and the Settings toggle. They're kept identical only by a comment.

What gets harder to change. Change the sequence (add a step, reorder one) and you must edit both copies perfectly — and a missed copy still compiles. This is not hypothetical: the bug fixed earlier today was the Settings path missing the "delete old key" step the launch path had.

The fix. Put the sequence in one function (certs::prepare_https(mode)); both callers just call it. One place to change, impossible to half-update.

Technical trace

Instances: src-tauri/src/main.rs:55-94 (try_start_https), main.rs:424-428 (startup gate), commands.rs:151-180 (enable_safari_support), certs.rs:170-182,257-345,427-433 (stepwise API).

Refactoring: Form Template Method / Extract Function — HttpsBootstrap::{prepare_at_launch, enable_from_settings} owning the order. Verified by re-read; this is the SEC-08 M1 failure mode.

HighF-02SDK's public types live inside the file that changes every week
bothMisplaced-Shared-Types + latent Cyclic Dependencystructural · hotfix: hours

What it is. The six types npm consumers import are defined inside the 440-line implementation file (the SDK's #1 hotspot). The transport file reaches back into that file to borrow two of them.

What gets harder to change. Every internal tweak churns the same file that defines the public contract, so "did the public API change?" gets buried in implementation diffs — and the two files now point at each other.

The fix. Move the types to a small lib/types.ts and re-export them from index.ts. The public contract gets a stable home and the back-pointer disappears.

Technical trace

Instances: sdk/src/lib/accelerator-prover.ts:11-92 (6 types), accelerator-transport.ts:3 (type-only back-import), index.ts:1-9.

Refactoring: Move shared types to lib/types.ts; re-export unchanged. Runtime-acyclic today only because the reverse leg is import type.

HighF-03Every server error is hand-built as a (status, string) pair — 11×, with one odd shape
bothDuplicate Code / Data Clumpstructural · hotfix: hours–1 day

What it is. Each error the proving server returns is assembled by hand as a (status code, JSON string) pair — 11 times — and one spot (host.rs) builds a structurally different body.

What gets harder to change. Want to add a field to errors, or standardize the body shape? You edit all 11 spots and hope they match. The odd-one-out in host.rs silently won't.

The fix. One ProveError type that knows how to turn itself into a response; each handler returns the type. No new library needed — a plain enum fits the codebase's deliberate "no thiserror/anyhow" style.

Technical trace

Instances: def server.rs:326; 11 sites server/prove.rs:63,122,134,182,222 + server/auth.rs:41,67,79,99,105,132; divergent shape server/host.rs:69-72 (uses axum::Json, no message field — verified by re-read).

Refactoring: Replace Error Tuple with Type — enum ProveError impl IntoResponse, preserving the pinned body contract.

HighF-04App startup is one 154-line function doing nine jobs
bothLong Method + Divergent Changestructural · hotfix: hours

What it is. The desktop .setup() closure is ~154 lines doing about nine unrelated jobs (tray, crash recovery, HTTPS, diagnostics, update polling…) with ten near-identical "clone for X" bindings.

What gets harder to change. Any startup change means wading into the whole blob, and none of it is unit-testable — you can only exercise it by launching the entire app.

The fix. Split into named pieces — init_tray / build_state / bootstrap_https / start_background_tasks — each readable and testable.

Technical trace

Instance: src-tauri/src/main.rs:314-466 (clone/callback wiring concentrated at 362-417).

Refactoring: Extract Class DesktopBootstrap.

HighF-05One 96-line SDK method does everything about the health check
bothLong Method + missing Factorystructural · hotfix: hours

What it is. #probeAndParseHealth probes, parses JSON, picks legacy-vs-multi-version protocol, applies version policy, logs, mutates protocol state, caches, and hand-builds six different status objects — all in one method.

What gets harder to change. Add a field to the status object and you must touch six branches; miss one and it still type-checks (the field is optional).

The fix. Pull the parse/version steps into their own methods, and build the status objects through one factory function.

Technical trace

Instance: accelerator-prover.ts:233-328; manual union construction ×6 at 248,269,291,308,317,326.

Refactoring: Extract Method + Introduce Factory for AcceleratorStatus.

HighF-06SDK remembers HTTP-vs-HTTPS by scattered side effects
bothTemporal Couplingstructural · hotfix: hours

What it is. The SDK pins which protocol reached /health by setting a flag in some branches and clearing it in others — the "pin on success, clear on error/bad-JSON" rule is spread across three spots, then read later when sending a proof.

What gets harder to change. Anyone adding a new health-check branch has to remember the pin/clear rule, or the SDK silently keeps talking to a stale endpoint.

The fix. Let the transport own a single method that updates the protocol from the result — the rule lives in one place.

Technical trace

Instances: pin accelerator-prover.ts:256; clear 268, 325; consumed in accelerator-transport.ts:57-63,121-133 via baseUrl.

Refactoring: Move Method — transport-owned commitStatus(status) deriving the pin from the discriminant.

MediumF-08The "version" and "origin" wrapper types are abandoned past the front door
bothPrimitive Obsessionstructuralfix: ~1 day each

What it is. The code has proper AztecVersion and CanonicalOrigin wrapper types, but only uses them at the boundary — downstream, everything reverts to raw strings.

What gets harder to change. The wrapper's guarantees stop at the edge, so every raw-string spot must be re-audited whenever version or origin rules get stricter. Old "already canonical" comments still mark the spots the wrapper was meant to make unnecessary.

The fix. Thread the wrapper types (&AztecVersion, &CanonicalOrigin) through the hot paths instead of unwrapping them early.

MediumF-09Two lookup tables kept in sync by hand
bothTemporal Couplingstructuralfix: hours

What it is. The authorization manager keeps two indexes (by origin, by request id) that must be updated together on every change, with a defensive guard already hinting the sync is fragile.

What gets harder to change. Add cancellation or expiry and you must remember to touch both tables in every mutator; one miss leaves a stuck or unresolvable request.

The fix. Wrap the pair in one PendingAuthorizations type with methods that update both — or drop the second table entirely (it holds ≤10 entries; a scan is fine).

Technical trace

Instances: authorization.rs:172-176,213-217,224-231,240-241 (defensive desync at 214).

Refactoring: Extract Class / Introduce Bidirectional Map.

MediumF-10Updater and crash-recovery share a hand-coded disarm/rearm protocol (Windows)
bothTemporal Coupling + Inappropriate Intimacystructuralfix: hours

What it is. On Windows, installing an update must disarm crash-recovery, install, then re-arm it — and re-arm on every exit path. The updater hand-codes this and re-arms in three spots, guarded by a comment ("every path that leaves the app running must end armed").

What gets harder to change. Add a new early-return to the update flow and you'll forget to re-arm; the Windows recovery rule leaks into the updater.

The fix. A guard object that re-arms automatically when it drops — suspend_for_update(app, || install) — so forgetting becomes impossible.

Technical trace

Instances: updater.rs:146-189 (3 rearm sites ~159/169/177) ↔ crash_recovery.rs:12-18,26-29,322-349, quit at main.rs:338-347.

Refactoring: Introduce Guard Object (RAII CrashRecoverySuspendGuard + defuse() on restart).

LowF-13"Lock, change, save" config pattern copied three times — with three different failure behaviors
claude + map (codex disagreed → resolved)Duplicate Codelocal · cross-cratefix: hours

What it is. The lock→mutate→save pattern appears three times, and they behave differently when the save fails: one propagates the error, one logs a warning, one ignores it. A helper exists but two copies can't reach it (it lives in the wrong crate).

What gets harder to change. A developer reading one site can't assume the others behave the same on save failure; adding fsync or retry means finding all three.

The fix. Move one lock_mutate_save helper into core/config.rs so both crates share it and one save-failure policy.

LowF-15Config load/save hardcode the path, so the test re-implements save

What it is. load()/save() hardcode the config file path, so the roundtrip test has to re-implement save to point at a temp file — and the atomic write-tmp-rename block has no test of its own.

What gets harder to change. You can't test load/save against a temp path without env tricks, and a test that duplicates save can't catch a real save regression.

Refactoring: Parameterize the path (Extract / Introduce Parameter).

Cross-cutting observations

Comment-enforced invariants are the systemic theme

F-01, F-06, F-09, F-10, and F-13 share one root: an ordering or a two-place state held in sync by a // must… comment instead of by a type or structure. The heavy SEC-xx/Q-xx/F-xx why-comment habit is excellent documentation but is doubling as the enforcement mechanism — and F-01 proves that fails. Converting the highest-traffic of these into structural seams (guard objects, owned state transitions, template methods) removes a whole bug class.

Prior refactors genuinely paid down the big-ticket duplication

The expected "cross-crate AppState/gating duplication" came back mostly already resolved by the #325 core extraction — both models credited it. What remains cross-crate is a ~6-line tracing bootstrap (Low) and the stranded server.rs tests (F-12). The crate split is deliberately not a Cargo workspace and that's working — don't re-litigate it.

Parse-don't-validate is adopted but not finished

AztecVersion and CanonicalOrigin exist and are correct at their construction edge, then decay to &str downstream (F-08). The pattern is right; the follow-through is the remaining work.

Refuted leads (signal, not noise)

The agents rejected eight size-based or duplication leads outright — server.rs and versions/mod.rs "Large Class" (both ~60% tests), crash_recovery's three platform impls (cleanly trait-dispatched), the network-tier retention "switch" (already a table), the bb-resolution ladder, tls.rs, mutate_config as Middle Man, and the config "Data Clump." Refuting is the discipline that keeps the finding list honest.
