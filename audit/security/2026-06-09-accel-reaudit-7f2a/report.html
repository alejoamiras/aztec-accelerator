The one residual

Medium R7A-01 Updater preflight size cap is defeated by manifest tampering
Found by Codex + Claude confidence: high CWE-400 status: KNOWN DEFERRAL · #345

What it is. Before downloading an update, the app checks the size the update feed claims the file is, and refuses anything over 500 MB. The problem: that claimed size comes from the same feed an attacker would have to tamper with anyway, so it isn't a trustworthy number. The update library also loads the entire file into memory before it checks the cryptographic signature.

What an attacker would do. An attacker who can tamper the update feed (e.g. by compromising the file host) points the download at a giant file and either omits the size or lists a small fake one. The check passes, the app tries to load the giant file into memory, and runs out of memory and crashes — before the signature check would have caught the tampering. It can't install a fake update (the signature still protects that); it can only crash the updater.

The fix. The real fix is to cap the bytes actually read during the download (a streaming limit), not to trust a number from the feed. The update library doesn't offer that hook today, so it needs an upstream change — or replacing the download path, which was deliberately rejected earlier because hand-rolling the signature check is riskier than this crash. So it's tracked (#345) and deferred, not fixed in this pass. Note: "just make the size mandatory" does not fix it — a present size can still lie.

Technical trace

Trace: updater.rs:67-81,88-118 reads Update.raw_json.platforms.*.size (attacker-controlled feed) → preflight passes on omit/lie-low → tauri-plugin-updater-2.10.1/src/updater.rs:696-713 buffers the whole artifact into an unbounded Vec with a non-aborting callback, before verify_signature.

Fix: independent byte bound in the download path (streaming abort cap). Needs upstream plugin support or the audit-R3-rejected self-managed downloader. Availability-only; integrity still enforced by minisign on the bytes. See report.md R7A-01 + the in-code comment at updater.rs:88.
