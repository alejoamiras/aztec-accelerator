- [packages/accelerator/server/src/main.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/server/src/main.rs:43), [packages/accelerator/core/src/server/auth.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/server/auth.rs:18), [packages/accelerator/core/src/server/prove.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/server/prove.rs:116): the headless server ships fail-open. If `ALLOWED_ORIGINS` is unset, `auth_manager` is `None`, `authorize_origin()` returns `Ok(())`, and any browser origin can `POST /prove`. Because the router also sets wildcard CORS, a malicious page can drive proving, read responses, and trigger arbitrary `bb` version downloads with `x-aztec-version` without any approval step. - [packages/accelerator/core/src/versions/mod.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/versions/mod.rs:282), [packages/accelerator/core/src/versions/downloader.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/versions/downloader.rs:31): `bb` integrity is verified only against a SHA-256 digest fetched live from the GitHub API. That means trust collapses to the same online control plane and system TLS roots used to fetch the tarball. An attacker who can tamper with GitHub release metadata/API traffic or any TLS-terminating MITM to both endpoints can serve a malicious tarball plus matching digest and the app will install and execute it. **Defenses that hold**
- Desktop `/prove` auth is wired correctly: ingress reads `Origin`, canonicalizes it, rejects malformed/null-ish origins, and checks approval before body buffering.
- I did not find a concrete desktop `/prove` bypass via DNS rebinding in the traced code path. There is no `Host` header validation, but I did not find an `Origin` bypass to pair with it.
- TLS generation is materially hardened: the CA key is in-memory only, the CA is constrained to `localhost` / `127.0.0.1` / `::1`, and cert/key files are written atomically with owner-only perms.
- Auto-update is materially better than `bb` download: the feed is HTTPS and the updater uses a pinned minisign public key, with default version comparison preventing downgrades.
