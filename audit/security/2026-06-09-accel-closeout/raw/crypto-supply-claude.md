# Raw — crypto + supply chain (Claude agent)

Run: 2026-06-09 post-quality-fixes closeout security audit. Scope: TLS/cert gen, bb download, auto-update.

## Findings
1. **[MEDIUM] bb integrity check is circular against a compromised GitHub release** — versions/mod.rs:289 (+ downloader.rs:156-175 verify_digest, prove.rs:200 execute). Attacker who compromises the AztecProtocol GitHub release re-uploads a malicious `barretenberg-*.tar.gz` AND overwrites the asset `digest`; `fetch_github_asset_digest` reads that same attacker-controlled digest over api.github.com → verify passes → bb chmod 0o755 + executed. Pure network MITM IS blocked (api.github.com + download both HTTPS/rustls); supply-chain compromise of the release is NOT. macOS ad-hoc codesign = no provenance. Comment at mod.rs:285-288 already admits the gap (unshipped TODO). Missing control: verify against a publisher signature (minisign/cosign) pinned in the client, or a digest pinned in the shipped app.
2. **[LOW] Decompression bomb: 64MB cap on compressed download only, extraction unbounded** — downloader.rs:240 (extract) vs :129-149 (download). entry.unpack streams the decompressed `bb` to disk with no size ceiling → disk-exhaustion DoS. Only bites once the digest defense is already defeated (verify runs before extract). Missing control: cap cumulative decompressed bytes / check entry.header().size().
3. **[LOW] migrate_legacy_ca_key is best-effort: delete failure silently leaves the mint-any-cert primitive** — certs.rs:186-198 (called main.rs:422). Older install with readable on-disk ca.key + still-trusted legacy anchor: remove_file tried once; on failure logs error + proceeds (HTTPS still comes up). A local attacker reading ca.key can mint certs for any domain the legacy anchor validates (NameConstraints the only backstop, not enforced by all clients). Missing control: verify key gone / retry / user alert / drop orphaned anchor.
4. **[LOW] Rotation trust-check verifies staged CA anchor but not that the staged leaf chains to it** — certs.rs:311 (`security verify-cert -c ca.pem.new` only). swap_into order ca→leaf→key (non-atomic 3 renames); load_rustls_config loads only leaf+key, no chain check. Crash between renames / future staging divergence → live ca/leaf mismatch no validity check catches until ~expiry. Missing control: `verify-cert -c leaf -r ca` before swap; atomic staging-dir swap.
5. **[LOW] Locally-trusted CA relies on client-side NameConstraints enforcement; keylessness is the real backstop** — certs.rs:97-107 (unconstrained trustRoot at :93). RFC5280 name-constraints are client-dependent. IF the CA key leaked, a non-enforcing client accepts attacker-minted *.com certs. NOT Critical because the CA key is genuinely never persisted (stack-local, dropped; rcgen `zeroize` Cargo.toml:53; leaf SAN limited to localhost; tests pin ca.key never written certs.rs:532-554). Flagged as defense-in-depth: a future regression persisting the key → *.com-minting Critical.

## Defenses that HOLD
- Network MITM on bb download (HTTPS/rustls both hops). Path traversal in tarball extract (dest is a fixed path, entry path discarded). CA key non-persistence (zeroize + tests). Leaf SAN restricted to localhost.

NOTE: all pre-existing (not introduced by quality-fixes). F-07/F-09 reorganized certs/https without changing these properties.
