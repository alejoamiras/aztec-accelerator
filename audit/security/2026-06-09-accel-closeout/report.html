Every finding here is pre-existing — none was introduced by the quality-fixes refactors (those were behavior-preserving; F-02 actually hardened origin canonicalization). The audit looked past the diff at the standing posture because this app is a localhost server that arbitrary web pages talk to, plus a self-updater. The desktop consent flow, TLS keylessness, and the auto-updater are well-built and held up under both model families. Two High issues stand out.

Cheapest high-value fixes first (land these in one PR):
  • Add a Host-header allowlist at server ingress (reject Host whose host ∉ localhost/127.0.0.1/[::1]). ~hours. Single-handedly defeats DNS rebinding (SEC-01) on desktop and headless.
  • Restrict /health CORS / shrink its unauthenticated surface (SEC-05). ~hours.
  • Size-cap the updater download before buffering (SEC-03). ~hours.
Ship-together theme — the origin gate: SEC-01 (no Host check + fail-open), SEC-04 (any localhost page auto-approved), and SEC-05 (/health open + wildcard CORS) are facets of one root cause: a browser-reachable local service trusting only Origin + wildcard CORS, with no Host binding. Treat them as one hardening workstream.

High

HighSEC-01Localhost origin gate is bypassable — no Host check + fail-open
found by Claude + Codex confidence: high CWE-350 · DNS rebinding fix: hours · CHEAPEST

What it is. The app runs a little web server on your own machine (port 59833) that browser pages ask to do heavy proving work. It decides who's allowed by looking at one label the browser attaches — the Origin — and pops up a MetaMask-style "approve this site?" prompt. But it never checks the Host label, and if the Origin label is simply missing, it says "yes, approved." The server-only (headless) build goes further: if you don't set an allowlist, it approves everyone.

What an attacker would do. A malicious website uses a 30-year-old trick called "DNS rebinding": it serves its trap page on port 59833 of evil.com, then quietly re-points evil.com at 127.0.0.1. Now the page's own follow-up request counts as "same site," so the browser sends it with no Origin label — and your local server, seeing no label, approves it. No popup. The attacker can now burn your CPU on their proving jobs, read the results, and make the app download arbitrary bb binary versions. The headless case is even simpler — any page can hit a no-allowlist server directly.

The fix. Check the Host label and refuse anything that isn't literally localhost/127.0.0.1/[::1] — that one check kills the rebinding trick outright. Then stop treating a missing Origin as "approved" (fail closed), and make the headless build deny-by-default. The codebase already has all the pieces.

Technical trace

Instances: core/src/server/auth.rs:30 (None => return Ok(()), pinned by test prove_skips_auth_when_no_origin_header at server.rs:908) · auth.rs:18+prove.rs:116 (no Host read anywhere) · server/src/main.rs:43 (headless auth_manager: None when ALLOWED_ORIGINS unset) · server.rs:194 (wildcard CORS) · binds 127.0.0.1 only (server.rs:186).

Fix: (1) Host-allowlist at ingress; (2) fail-closed on absent Origin (flip the test contract); (3) headless: unset ALLOWED_ORIGINS ⇒ deny-all-non-localhost, or explicit --allow-all. Cross-model nuance: Codex didn't confirm the desktop same-origin-rebinding chain ("no Origin bypass to pair with the missing Host check"); Claude traced it via same-origin-omits-Origin + the pinning test. Both agree the Host allowlist is the root fix and the headless default-open is real.

HighSEC-02bb download integrity is circular against a compromised release
found by Claude + Codex confidence: high CWE-494 · A08 fix: days

What it is. The app downloads the bb proving binary from GitHub and checks it's genuine by comparing it to a checksum — but it fetches that checksum from the same GitHub release it's downloading the binary from. That's like verifying a £20 note against a photo of it printed by the same counterfeiter.

What an attacker would do. Whoever can tamper with the GitHub release (a compromised maintainer account, or a network attacker able to intercept both GitHub endpoints) swaps in a malicious binary and updates the checksum to match. The app sees a "valid" checksum and runs the attacker's code on your machine. (Plain network eavesdropping is already blocked — both downloads use HTTPS — so this is specifically a supply-chain/account-compromise risk.)

The fix. Verify the binary against a signature from a key baked into the app itself, not fetched from the network — exactly what the app's own auto-updater already does correctly with a pinned minisign key. Copy that pattern.

Technical trace

Instances: core/src/versions/mod.rs:282-326 (fetch_github_asset_digest reads expected SHA-256 from api.github.com) · downloader.rs:31,156-175 (verify → install → chmod 0o755 → execute at prove.rs:200) · admitted TODO at mod.rs:285-288.

Fix: offline-pinned publisher signature (minisign/cosign/TUF), independent of the download channel — mirror src-tauri/src/updater.rs (pinned minisign). Interim: ship a curated pinned-digest manifest rather than fetching digests at install time.

MediumSEC-03Updater buffers the whole artifact in memory before verifying (no size cap)
found by Codex CWE-400 fix: hours

What it is. When updating itself, the app downloads the new version fully into memory before checking its signature.

What an attacker would do. A tampered update feed or compromised CDN points the app at a multi-gigabyte file; the app keeps loading it into RAM until it runs out — a crash/denial-of-service. (The signature check itself is solid, so they can't install fake code this way — only exhaust memory.)

The fix. Enforce a hard size limit from the update manifest, and/or stream to a temp file on disk before verifying.

Technical trace

src-tauri/src/updater.rs:66 → tauri-plugin-updater-2.10.1/src/updater.rs:702 (unbounded Vec<u8>). Availability only; signature (pinned minisign) is sound. Partly upstream-dep — cap on the app side.

MediumSEC-04Any localhost-origin page is auto-approved (no per-app scoping)
found by Claude CWE-923 · A01 fix: hours–day

What it is. Any page served from localhost (any port) is approved with no prompt — the app assumes "anything local is trustworthy."

What an attacker would do. A second dev server you're running, a malicious npm install script that opens a local port, or any tool that got a foothold on your machine gets unlimited, silent access to the prover.

The fix. Scope the auto-approval to the app's own playground origin, or prompt for other localhost origins.

Technical trace

core/src/authorization.rs:213-223 (is_auto_approved matches any {localhost,127.0.0.1,[::1]}:any-port), via is_approved at :230.

MediumSEC-05/health is unauthenticated + wildcard CORS → cross-site fingerprinting
found by Codex CWE-200 fix: hours

What it is. The /health endpoint answers any website, with no gate, exposing app version, cached Aztec versions, whether bb is present, and HTTPS status.

What an attacker would do. Any site you visit silently checks "is the accelerator installed, which version?" — reconnaissance to target a version-specific exploit, plus a privacy/tracking signal.

The fix. Restrict /health CORS to approved origins, or return only a minimal liveness response unauthenticated and gate the details.

Technical trace

core/src/server.rs:194,215 (wildcard CORS + full health payload).

MediumSEC-06respond_auth uses origin-as-identity (no per-request token)
found by Claude CWE-362 fix: day

What it is. When you click Approve/Deny, the decision is matched to a pending request by origin string rather than a unique request id. A malformed-origin Deny silently misses (the request just times out after 60s), and two pending requests for the same origin share one decision.

What an attacker would do. Limited today — the approval command is only reachable from the local app UI (Tauri IPC). It's a design weakness to fix before any new surface exposes that path.

The fix. Give each pending request a server-issued opaque id and resolve on that.

Technical trace

src-tauri/src/commands.rs:131 (auth.resolve(&origin,…) on raw arg) vs canonical-keyed pending map at server/auth.rs:71.

LowSEC-08Legacy ca.key cleanup is best-effort (fails open)

What it is / fix. On upgraded installs, if deleting the old on-disk CA key fails, the app logs and proceeds — leaving a readable "mint any cert" key plus its trusted anchor. Verify the key is gone / retry / surface to the user; consider failing closed for Safari HTTPS until removed. Found by both models.

Technical trace

src-tauri/src/certs.rs:181-198, called main.rs:418/422.

LowSEC-09Cert rotation verifies the CA anchor but not that the leaf chains to it

What it is / fix. Rotation checks the new CA is trusted but not that the new leaf chains to it, and swaps three files non-atomically — a crash mid-swap could leave a mismatch unnoticed until expiry. Add a leaf-against-anchor verify before swap; use an atomic staging-dir swap.

Technical trace

src-tauri/src/certs.rs:298-327 (verify-cert -c ca.pem.new only); swap_into 3 renames at :71-76.

Cross-cutting observations

1 · The local server trusts network-position + Origin, never Host

SEC-01/04/05 are one theme: a browser-reachable local service needs a Host allowlist and per-origin consent as defense-in-depth, not Origin-only + wildcard CORS. The Host check (SEC-01) is the single highest-leverage change.

2 · Two "trust the network for integrity" instances, opposite maturity

The auto-updater does it right (pinned minisign, HTTPS, downgrade-prevention — confirmed by both models). The bb downloader does it wrong (SEC-02). The fix is literally to copy the updater's pattern.

3 · A recurring fail-open habit

Absent Origin → approve; ca.key delete fails → proceed; headless no-allowlist → open. A consistent fail-closed posture on security-critical paths closes SEC-01 and SEC-08 together.

Defenses that hold

  • Desktop present-Origin auth: canonicalizes (the F-02 work), rejects malformed/null-ish, checks approval before buffering the body.
  • TLS generation: CA key in-memory only (dropped + zeroize), constrained to localhost, cert/key written atomically with owner-only perms — tests pin the key is never written.
  • Auto-update: HTTPS feed + pinned minisign key + downgrade-prevention.
  • Network MITM on the bb download is blocked (HTTPS both hops); the residual is supply-chain compromise (SEC-02), not passive eavesdropping.
