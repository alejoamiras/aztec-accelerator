# Security audit — cluster: prove-witness-bb (witness lifecycle)

Files in scope: `packages/accelerator/core/src/server/prove.rs`, `packages/accelerator/core/src/bb.rs`

## Finding 1: Private ZK witness is written to a world-readable temp file in a world-traversable directory

1. **Title**: Witness (`ivc-inputs.msgpack`) written via `std::fs::write` inherits umask-default (world-readable) permissions inside an umask-default (world-traversable) `tempfile::tempdir()`, exposing the plaintext private-transaction inputs to any other local user for the duration of proving.

2. **Impact factors**:
   - Confidentiality: violated — the artifact disclosed is the raw private ZK witness itself (per this audit's threat model, the single most sensitive artifact in the system: "deanonymizes a private transaction").
   - Integrity/Availability: not affected.
   - Blast radius: any other local, unprivileged account on the same host as the running accelerator (desktop app or headless server) for every `/prove` request that occurs while the attacker is watching — i.e. all users of a shared/multi-tenant host are potential victims, one witness per successful capture.
   - Data sensitivity: maximal (private ZK inputs).
   - Exploitability: Attack vector = Local. Attack complexity = Low (list `/tmp`, `cat` the file — no race/guessing required). Privileges required = Low (ordinary unprivileged local shell account, not root). User interaction = None from the attacker; the only "trigger" is the victim's own ordinary use of any Aztec dApp that calls `/prove`, which is normal expected traffic, not an induced action.

3. **Evidence confidence**: High. The code path was read directly, and the exact permission behavior of the pinned crate version was verified against upstream primary source (not assumed from memory) — see step 5.

4. **OWASP / CWE**: CWE-379 (Creation of Temporary File in Directory with Insecure Permissions), secondarily CWE-732 (Incorrect Permission Assignment for Critical Resource) and CWE-200 (Exposure of Sensitive Information to an Unauthorized Actor). OWASP category: A01:2021 – Broken Access Control.

5. **Trace** (source → sink, file:line at every step):
   - `prove.rs:112` — `axum::body::to_bytes(raw_body, MAX_BODY_SIZE)` buffers the untrusted request body (the witness) after `authorize_origin` passes (`prove.rs:110`).
   - `prove.rs:192` — `bb::prove(&body, version_for_prove.as_ref(), threads).await` forwards the raw witness bytes.
   - `bb.rs:78-82` — `pub async fn prove(ivc_inputs: &[u8], ...)` receives them.
   - `bb.rs:86` — `let tmp_dir = tempfile::tempdir()?;` creates the temp directory. Verified against the pinned version (`tempfile = "3.27.0"`, confirmed in `packages/accelerator/core/Cargo.lock`): `tempfile`'s Unix `TempDir` creation (`src/dir/imp/unix.rs` at tag `v3.27.0`) builds the directory via `std::fs::DirBuilder::new()...create()` and only applies an explicit `mode()` **if the caller passes `Builder::permissions(...)`**, which this call site does not. With no explicit mode, `DirBuilder::create` uses the OS-default `mkdir` permission request (`0o777`) filtered by the process **umask** — under the standard Linux/most-distro default umask `022`, that yields `0o755` (`rwxr-xr-x`): world-readable **and** world-traversable. (The crate's own public docs state this plainly: temp dirs are created "with the default permissions unless otherwise specified via `Builder::permissions`… depending on your platform, this may make them world-readable.")
   - `bb.rs:90` — `std::fs::write(&input_path, ivc_inputs)?;` writes the witness. This is the plain `std::fs::write` API, **not** `tempfile`'s own `NamedTempFile`/`tempfile()` constructors (which *do* hardcode `mode(0o600)` in `src/file/imp/unix.rs`'s `create_named()`). `std::fs::write` opens the file with the OS-default `open()` mode `0o666` filtered by the same process umask, i.e. `0o644` (`rw-r--r--`) under the standard `022` default — world-readable.
   - Sink: any other local user on the host can `ls`/`inotifywait` the shared OS temp root (`/tmp`, mode `1777` on essentially all Linux distros — listing is not blocked by the sticky bit) to discover the freshly created `tmp_dir` (a per-request random-named subdirectory), then traverse into it (`0o755` permits `x` for `other`) and `cat` `ivc-inputs.msgpack` (`0o644` permits `r` for `other`) to read the full plaintext witness.
   - Contrast showing the team knows and applies the correct pattern elsewhere, but not here: `packages/accelerator/core/src/config.rs:151` explicitly sets the config **parent dir** to `0o700`, and `config.rs:167` explicitly opens the config file with `.mode(0o600)` — for `approved_origins`, data far less sensitive than a raw ZK witness. No equivalent call exists anywhere in `bb.rs`.

6. **Missing control**: `bb.rs:86-90` never (a) calls `Builder::new().permissions(Permissions::from_mode(0o700)).tempdir()` (or a post-creation `std::fs::set_permissions(tmp_dir.path(), Permissions::from_mode(0o700))`) for the directory, nor (b) opens `input_path` with an explicit `OpenOptions::new().mode(0o600)` (mirroring the exact pattern already used in `config.rs:159-168`) for the witness file. Both are one-line fixes using the same `std::os::unix::fs::{PermissionsExt, OpenOptionsExt}` machinery already imported elsewhere in this crate.

7. **Exploit/violation scenario**:
   1. Attacker controls unprivileged local account `B` on a shared Linux host that also runs the accelerator as account `A` (desktop app or headless `accelerator-server`), with the host's default umask (`022`, the near-universal Linux/macOS default — not overridden anywhere in this codebase).
   2. `B` runs `inotifywait -m /tmp` (or simply polls `ls -la /tmp` — `/tmp` is world-listable, mode `1777`) to catch new entries.
   3. `A`'s browser (any approved-origin dApp, including `A`'s own legitimate use) POSTs a private-transaction witness to `http://127.0.0.1:59833/prove`.
   4. `bb::prove` creates `/tmp/.tmpXXXXXX` (mode `0o755` per step 5) and writes `/tmp/.tmpXXXXXX/ivc-inputs.msgpack` (mode `0o644` per step 5).
   5. `B` sees the new directory appear, runs `cat /tmp/.tmpXXXXXX/ivc-inputs.msgpack`, and obtains the complete plaintext private ZK witness for `A`'s transaction — no guessing, no race condition, no privilege escalation needed.
   6. The exposure window is at minimum "however long `bb prove` takes" and at maximum the full `PROVE_TIMEOUT` of 300s (`bb.rs:7`) for a slow/large circuit — since `tmp_dir` (and its `Drop`-based cleanup) is held live for the whole `bb::prove` call. If the host process is killed with `SIGKILL`, panics past an abort point, or the machine loses power mid-proof, `TempDir::drop` never runs and the plaintext witness file is left behind in `/tmp` indefinitely (subject only to whatever external tmp-cleaning cron/service the OS may or may not run).

8. **Preconditions**: (a) attacker has any unprivileged local shell account on the same host as the running accelerator process — no root, no special group membership; (b) the host uses a shared, non-per-user temp root (true by default on Linux, where `TMPDIR` is typically unset and `tempfile::tempdir()` resolves to the shared `/tmp`); (c) default/unhardened umask (`022`), which is the out-of-the-box default on virtually every mainstream Linux distribution and is never adjusted by this process. This precondition set describes an ordinary shared Linux server/CI runner/VM — exactly the class of host the headless `accelerator-server` variant of this app is built to run on, and exactly the kind of multi-tenant machine this repo's own operational context (many concurrent agents/users on one box) assumes as normal. (Note: interactive macOS/Desktop launches often get a private, launchd-assigned per-user `TMPDIR` with `0700` perms, which would close this specific path on that platform/launch-mode — the finding is scoped to the Linux/headless-server deployment target where the shared `/tmp` model applies.)

9. **Why existing mitigations fail**: This app's documented SEC-0N controls (Host-header allowlist SEC-01a, per-origin authorization SEC-04, TLS/updater/digest controls) all gate the *network* ingress — which remote origins may cause a `/prove` call to happen at all. `authorize_origin` (`prove.rs:110`, `server/auth.rs`) runs before the body is even buffered, which correctly stops unapproved origins from reaching witness buffering — but that is an orthogonal boundary from this finding. None of the documented mitigations constrain the *local-filesystem* permissions of the artifact the server itself persists to disk once a proof is triggered by any legitimately-authorized request. There is no SEC-0N entry covering temp-file permissions, and the one place in this codebase that *does* correctly harden file/directory permissions for sensitive data (`config.rs`, 0o700/0o600) is a separate code path that does not share an implementation with `bb.rs`, so its fix does not propagate here. This is a genuinely unmitigated gap, not a documented residual.

10. **Instances**: Single root cause, one call site within the audited files: `packages/accelerator/core/src/bb.rs:86` (`tempfile::tempdir()` — directory perms) and `packages/accelerator/core/src/bb.rs:90` (`std::fs::write` — file perms). `output_dir` (`bb.rs:88-89`, holding the *proof*, not the witness) inherits the same directory-permission gap, but the proof is not independently sensitive here since it is already returned to the requesting origin in the HTTP response, so it is not counted as a separate instance of the *witness*-confidentiality root cause.

---

No other findings met the bar for a concrete, evidence-backed trace in this cluster:
- Version/argv injection via `x-aztec-version`: validated by `AztecVersion::parse` → `is_valid_version` (`versions/version_policy.rs:187-196`), a strict ASCII-alphanumeric+`.`/`-`/`_` allowlist rejecting `..`, leading/trailing `.`, length>128; the value is passed as a single `tokio::process::Command` argv element (no shell), so no path traversal or argument/command injection is reachable. Confirmed traversal-safe by construction.
- bb `stderr`/exit-status leakage to the HTTP client: `bb.rs:135-140` returns only `format!("bb prove failed (exit {})", output.status)` to the caller; full `stderr` is logged server-side only (`bb.rs:130-133`), never included in the `ProveError`/HTTP response (`server.rs:358`). No witness bytes, file paths, or bb internals reach the browser on this path.
- Witness bytes reaching argv/env/stdout: only the temp file *path* is passed via `--ivc_inputs_path` (`bb.rs:99-111`); `HARDWARE_CONCURRENCY` env only carries a thread count (`bb.rs:115`); captured subprocess `stdout` is discarded, never logged or returned. No witness content crosses into argv/env/logs via this path.
- Local log-file leakage of witness-derived `stderr` content: considered, but the Tauri desktop build explicitly hardens the log directory to `0o700` before any log file is written (`src-tauri/src/main.rs:390-395`, outside this cluster's file scope but directly relevant to whether the sink is reachable by another local user) — no bypass of that control was found, so this does not clear the bar for a concrete finding.
- TOCTOU/symlink pre-planting on the temp path: `tempfile::tempdir()` uses cryptographically-random unique names with `create_dir`-style exclusive creation and retry-on-collision; the name is not predictable/guessable by another local user ahead of creation, so a symlink pre-plant attack is not feasible. No concrete bypass found.
- 50MB body cap + single-permit prove semaphore as a DoS vector: bounded memory (`MAX_BODY_SIZE`, `prove.rs:99,112`) and single concurrency (`prove.rs:121-125`) are both origin-gated (only an already-authorized origin can invoke `/prove` at all), and no evidence was found (in-scope files) that malformed input causes `bb` to run for the full 300s timeout rather than failing fast — no concrete exploit trace, not flagged.
