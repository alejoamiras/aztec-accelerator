## 1. Summary

Make the accelerator's local HTTPS listener (`127.0.0.1:59834`) **default-on** — consented via a new **first-run onboarding wizard** — on macOS, Linux, and Windows, not just macOS/Safari. Rename the misleading `safari_support` config/UI/command surface to HTTPS naming. Add **user-level** browser-trust install on Linux (NSS DBs, no root) and Windows (CurrentUser Root store), reusing the existing keyless-CA cert machinery. Teach the SDK to **prefer HTTPS when it's trusted**, with a proven zero-latency fast path when it isn't, plus an opt-in strict `httpsOnly` mode. Ship rotation and uninstall stories per OS. Production validation: WebDriver E2E on 3 OSes for the wizard, **new cert-trust CI on 3 OSes** (Linux + Windows exercise the real store/chain; the macOS leg tests `security`/chain mechanics via an admin-domain path, NOT the production login-keychain/user-auth flow — see R5), Playwright mocks for the wizard, and clippy/biome/unit gates on every phase.

**The one-line framing of what this does and doesn't buy** (carried from the prior HTTPS investigation): default mode is UX/positioning parity — Chrome/Firefox already reach HTTP loopback fine; HTTPS-preferred only gains real impostor-resistance for dApps that set `httpsOnly`. The wizard's OS-native trust dialog **is** the consent ceremony. Copy must not overclaim ("your traffic is now safe") — the honest benefit is an encrypted, authenticated loopback channel that a *different non-admin local user* can no longer trivially impersonate; a same-user attacker is still game-over (recorded SEC-04 threat model).

## 2. Naming & headline decisions

- **Config key:** `safari_support` → **`https_enabled`** (bool), migrated via `#[serde(default, alias = "safari_support")]`; `CONFIG_VERSION` 1→2. It's what the launch gate already keys on (`main.rs:69-83`).
- **Onboarding marker:** `onboarding_version: u32` (`#[serde(default)]` = 0). Wizard shows when `< ONBOARDING_VERSION` (=1). Versioned int (not bool) so a future release can re-onboarding for a new consent surface.
- **Rotation throttle field:** `last_rotation_prompt_at: Option<i64>` (unix secs) to throttle the macOS/Windows renewal window (§7).
- **UI label:** **"Encrypted Connection (HTTPS)"** everywhere (wizard toggle + Settings row), with per-OS sub-copy from `get_system_info` ("Required for Safari" on macOS; "Installs a local certificate into your browsers" on Linux/Windows).
- **Tauri commands:** `enable_safari_support`/`disable_safari_support` → **`enable_https`/`disable_https`**, compiled on all desktop OSes (drop the `#[cfg(target_os="macos")]` gate at `commands.rs:153-211`). Clean rename, no shim: the IPC surface is bundled-content-only (`capabilities/default.json` grants nothing to remote content).
- **New module:** `src-tauri/src/trust/{mod.rs,macos.rs,linux.rs,windows.rs}` — per-OS trust-store machinery extracted from `certs.rs`, which keeps only cert generation / rotation / rustls loading.

### 2.1 Wizard prefill (A9 decision) — HTTPS pre-checked for everyone
The migration still preserves `safari_support:true → https_enabled:true` (an upgrader who already opted in keeps it). But the **wizard's HTTPS toggle is deliberately pre-checked for ALL users** — new installs and upgraders alike, regardless of prior `safari_support` — because the goal is to move the entire installed base onto the encrypted path. So `get_onboarding_state` returns the HTTPS toggle default as `true` (not merely echoing the migrated `https_enabled`); Start-on-Login and Auto-Update prefills still reflect current OS/config state. The user can still uncheck HTTPS (opt-out is preserved). This makes A9 a genuine safe-path migration, not just a settings echo.

### "Default-on" vs first-run friction — resolved
True silent default-on is impossible: macOS `security add-trusted-cert` prompts for the keychain password, Windows CurrentUser-Root raises a native dialog. So **"default-on" = the wizard's HTTPS toggle is pre-checked, and one "Start" click performs the install with the OS-native dialog as the actual consent**. `AcceleratorConfig::default().https_enabled` stays `false` on disk (the headless binary shares this struct — a silent default-flip would light up TLS surface nobody consented to); only wizard/Settings consent flips it. Launch stays prompt-free (preserves the "launch only *verifies* trust, never installs" invariant, `main.rs:66-68`).

## 3. Per-OS trust design

| | Install | Verify | Rotation re-trust | Uninstall |
|---|---|---|---|---|
| **macOS** | `security add-trusted-cert` → login keychain (exists, moves to `trust/macos.rs`) | `security verify-cert` (exit code) | renewal **consent window** (§7) → prompt → verify → swap → delete old by SHA-1 | Settings "Remove trust" → `security delete-certificate`; docs for drag-to-trash |
| **Linux** | `certutil -d sql:~/.pki/nssdb -A -t "C,,"` + every Firefox profile from `profiles.ini` (mkcert-style); create nssdb if absent | `certutil -L -n <nick>` (exit code) | **silent** (user DBs need no auth): add new nick → verify → swap → delete old nicks | Settings "Remove trust" + `aztec-accelerator --remove-ca-trust` CLI; docs |
| **Windows** | `<System32>\certutil.exe -user -addstore Root ca.pem` (System32 via `GetSystemDirectoryW`, **not** `%SystemRoot%` env) — **no OS dialog is guaranteed** (audit H/L2: re-adding an existing root is silent; a fresh add may or may not prompt), so the **wizard Start click is the consent ceremony**, copy must not promise a Windows dialog | `certutil.exe -user -verifystore Root <serial>` (exit code — treated as tool-dependent, CI-pinned, not a documented API) | consent window → add → verify → swap → **delete old by serial** (§Ledger D4) | NSIS `NSIS_HOOK_POSTUNINSTALL` **guarded by `${If} $UpdateMode <> 1`** (§12 R1 — must not fire on auto-update) → `certutil -user -delstore Root <serial>` + del certs dir |

- **Linux browser coverage is reported honestly per store**: NSS user DB (`~/.pki/nssdb` — .deb/vendor Chrome/Chromium/Brave/Edge), each Firefox profile by name, plus best-effort snap (`~/snap/firefox/common/.mozilla/firefox`) and flatpak (`~/.var/app/org.mozilla.firefox/.mozilla/firefox`) Firefox roots. **`trust_status` must also enumerate-or-disclaim sandboxed Chromium** (snap `chromium` on Ubuntu is strictly confined and may not read `~/.pki/nssdb`; flatpak Chrome/Brave/Edge similar) — do not imply coverage we can't verify (audit M-2). **Windows Firefox 120+** imports OS-store roots by default, so the CurrentUser Root install covers it (audit L-6). Missing `certutil` → status "certutil not found — install libnss3-tools", and on Linux this counts as an **HTTPS-step failure in the wizard** (trust landed in 0 stores → HTTPS reported off + Retry, per your constraint), even though the bare listener could still bind harmlessly (§12 R3).
- **`.deb`:** add `bundle.linux.deb.depends: ["libnss3-tools"]` in `tauri.conf.json`.
- **AppImage:** **detect + degrade, do NOT bundle certutil** (§Ledger D5).

## 4. SDK: HTTPS-preferred probe + strict mode

Replace "first responder wins" with **"HTTPS wins only if it's healthy, bounded":** (revised per audit — see §12 R2)
1. Fire `httpsP` and `httpP` in parallel (2s timeout, `retry:0`) — unchanged shape.
2. If `httpsP` fulfills **with `response.ok` AND a parseable `/health` body** → **HTTPS wins**, even if HTTP answered first. A fulfilled-but-non-OK HTTPS response (404/500 — possible via ky's `throwHttpErrors:false`, e.g. a foreign server squatting fixed port 59834) does **NOT** win; it's treated like a rejection for winner-selection.
3. Otherwise the HTTP result decides (OK → http; not-OK/reject → the existing both-failed retry path, `accelerator-transport.ts:130-137` and the `keep`/`clear` transitions).
4. **Stall bound:** if `httpP` fulfills OK while `httpsP` is still pending, wait at most `HTTPS_GRACE_MS = 250` for `httpsP`; if unsettled, HTTP wins (don't cancel `httpsP`; ignore a late result).

**Zero-regression proof:** with nothing on 59834, `httpsP` rejects with loopback connection-refused in ~0–1 ms (default host is the literal `127.0.0.1`, `accelerator-prover.ts:66`, so no happy-eyeballs/IPv6 asymmetry), so step 3 resolves at `max(≈0, http)` — identical to today. Trusted HTTPS bound → one loopback TLS handshake (few ms), the case we *want*. Bound-but-untrusted → TLS alert rejects fast → HTTP wins as today. **Honest cost note (audit correction):** the 250 ms grace is paid whenever *any* process on 59834 accepts TCP but stalls the handshake (not only our own listener) — re-charged each `STATUS_CACHE_TTL_MS` (10 s) re-probe. Bounded and rare, but not exclusively self-inflicted.

**Strict mode:** `AcceleratorProverOptions.accelerator.httpsOnly?: boolean` + `AZTEC_ACCELERATOR_HTTPS_ONLY=1`. Probe/POST HTTPS only; never build an `http://` URL; unreachable → `{available:false, reason:"offline"}`. dApp-side policy knob, no server change.

Rejected: driving preference off the `/health` `https_port` advertisement instead of dual-probing — Safari's mixed-content block means the HTTP probe never resolves there, so the dual probe must stay primary; the hint would add a second code path for no latency win.

## 6. Decision Ledger {#ledger}

- **D1 — Naming** (all 3 converged): `https_enabled` config, "Encrypted Connection (HTTPS)" UI, `enable_https`/`disable_https` commands. Alias migration, no back-compat shim (internal IPC). *Rejected:* `encrypted_connection` key (redundant w/ label); keeping `safari_support` (misleading now cross-OS).
- **D2 — "Default-on" is a wizard default, not a struct default** (main+fable explicit; codex compatible): `default().https_enabled=false`; wizard pre-check + OS dialog = consent. *Rejected:* flipping the struct default (headless shares it → silent TLS nobody consented to).
- **D3 — Windows trust via `certutil.exe` (absolute System32 path), NOT CryptoAPI/`windows-sys`** — **conditionally upheld after Round 1** (both auditors accept certutil for Windows IF behavior is CI-pinned and we stop claiming API robustness). Chosen for: zero direct new deps (certutil is a system binary; `windows-sys` is only transitive today, would become a direct dep + unsafe FFI); consistency with the established shell-out-by-absolute-path precedent (`schtasks_exe`, `crash_recovery.rs:245-253`). **Codex dissent (partially sustained):** certutil is explicitly "not recommended for production, no compatibility guarantee" (MS docs) — so the plan **no longer claims exit-code semantics are an API contract**; they're a **CI-pinned, tool-dependent behavior** (P4 add/verify/remove tests are the contract). **Corrections adopted:** (a) the "NSIS needs certutil regardless" justification was a **false fact — removed** (the uninstaller can call `aztec-accelerator --remove-ca-trust` in-process; certutil-in-NSIS is a convenience, and we keep it only *because* the Rust CLI path also uses certutil, not because NSIS forces it); (b) resolve System32 via `GetSystemDirectoryW` / hardcoded `C:\Windows\System32` fallback, **not** the environment-derived `%SystemRoot%` (codex Med). Rotation/uninstall identity is delete-by-**serial** (D4), never by localized stdout.
- **D4 — delete-old-anchor identity, per-OS** (Round-1 + final-pass correction). All our anchors share CN "Aztec Accelerator Local CA", so during **rotation** (old + new briefly coexist) a delete must target the OLD one specifically:
  - **Windows rotation:** delete by **serial** — parseable from the old `ca.pem` with **`x509-parser` (already a direct dep)**, locale-proof, no new `sha1` crate. `certutil -delstore Root <serial>` accepts a serial as certID (CI-pinned in P4, not treated as an API contract).
  - **Linux rotation:** delete by **NSS nickname** — `certutil -D -n <nickname> -d sql:<db>` (final-pass correction: NSS's delete model is nickname-based, *not* serial; the per-anchor nickname `aztec-accelerator-ca-<8hex>` already makes old vs new unambiguous). The earlier "delete by serial everywhere" wording was wrong.
  - **macOS rotation:** keeps its existing SHA-1 path (`certs.rs:397-423`, unchanged).
  - **Uninstall (all OSes):** delete by **CN**, NOT serial — the NSIS `POSTUNINSTALL` hook runs *after* app files are removed, so `x509-parser`/the Rust binary are gone and a serial can't be computed. This is safe because rotation has already removed prior anchors, so only the current "Aztec Accelerator Local CA" remains; delete-by-CN removing "all our anchors" is exactly the uninstall goal (and our CN is specific enough that a foreign collision is implausible). No sidecar bookkeeping needed.
  - **Rejected:** codex's "add `sha1` crate" (serial obviates on Windows); main's "persist certutil's printed thumbprint" (fragile-parse trap); a uniform "delete by serial" (wrong for NSS + impossible at uninstall). **Ask A5 resolved.**

## 7. Rotation without annoying the user

Keyless CA ⇒ every rotation mints a NEW anchor needing re-trust. Linux: fully silent (user NSS needs no auth) — background rotation as today. macOS/Windows: an OS dialog is unavoidable, but instead of a context-free background prompt, when the leaf is within the 30-day window (`certs.rs:294`) show a small **"Certificate renewal" window** (wizard styling): one paragraph + "Renew now" → `rotate()` (which raises the OS dialog *with context*). Decline → old cert keeps serving; window reappears next launch until expiry (throttled by `last_rotation_prompt_at`); post-expiry the existing `MissingCertsReset` recovery applies (`main.rs:96-99`). Converts a surprise password prompt into an explained, user-initiated one, ~once every 794 days.

## 8. Security & Adversarial Considerations

- **S1 — The anchor.** Load-bearing mitigation unchanged: the CA key never touches disk (`certs.rs:141-153`, pinned by `generation_writes_no_ca_key`), so a trusted anchor in ANY store cannot sign anything new; name constraints (loopback/localhost, `certs.rs:97-107`) are defense-in-depth, **not** the primary control. **Do not trust name-constraint enforcement uniformity** across NSS/CryptoAPI/Chrome-verifier (historically uneven for user-added roots, Inference I5) — rely on key-lessness; wizard "learn more" copy states the honest benefit, no overclaim.
- **S2 — Trust-install execution.** All shell-outs use `Command::new` + fixed argv (no shell, no interpolation): macOS `security` (existing), Linux `certutil` resolved to a **known absolute path**, accepting a `which`-found binary only if absolute AND neither it nor its parent dir is user/group/world-writable (audit/codex H — a `which` that resolves is not enough; a planted binary in a writable PATH dir must be rejected), Windows `certutil.exe` via **`GetSystemDirectoryW`** (not env-derived `%SystemRoot%`; user-writable-PATH hijack is the realistic Windows attack — same class as `schtasks_exe`). **Zero new Rust deps** — delete-by-serial uses the already-present `x509-parser`, no `sha1` crate needed (D4). certutil supply chain = distro archives via `.deb Depends`, not us.
- **S3 — `profiles.ini` as attacker-influenced input.** User-owned but a compromised-unprivileged same-user process could plant entries. Mitigations: canonicalize each candidate path, require it under `$HOME`, require an existing `cert9.db`, pass as single argv element. Residual (certutil writing to an attacker-chosen NSS DB *as the same user*) is within the attacker's existing privilege — no escalation.
- **S4 — Wizard IPC (key least-privilege rule).** `complete_onboarding` takes exactly three typed bools; `get_onboarding_state`/`get_trust_status` are read-only; **no command accepts a path or cert bytes from JS** — trust code only ever operates on `CertPaths::live()`/staged paths it derives itself. **Final-pass correction:** the current `capabilities/default.json` grants `core:default` to *all* local windows, so a bundled-content XSS in any window could reach the new trust commands. **P5 adds a window-scoped capability** limiting the onboarding window (and the trust/onboarding commands) to their own capability set — OR, if that's deferred, the plan explicitly accepts the bundled-content-XSS risk (all HTML is first-party, no remote content loaded). Chosen: scope it in P5 (cheap, correct).
- **S5 — Config migration.** Malformed/ambiguous config (incl. both-keys duplicate-field) → defaults = HTTPS off (`config.rs:100-108`) — fail-safe. Alias can't *enable* anything new: `safari_support:true` only existed via explicit macOS opt-in.
- **S6 — Rotation & lingering anchors.** Fail-closed stage→trust→verify→swap order preserved per-OS; old anchor removed post-swap. Uninstall: Windows NSIS hook; Linux `disable_https`/`--remove-ca-trust`+docs (a root postrm can't safely walk user NSS DBs — out); macOS Settings-first + docs. Worst-case residue is a keyless loopback-constrained cert — cosmetic, still cleaned for hygiene.
- **S7 — SDK preference is not a downgrade oracle.** `httpsOnly` gives strict dApps no-fallback; default mode's loopback MITM is already local-game-over. Leaf key stays 0600 (`certs.rs:222-254`) — unchanged exposure.
- **S8 — Direct attacker targets:** (a) fake `certutil` on PATH → S2 (absolute + non-writable checks); (b) racing wizard cert-gen to swap `ca.pem` pre-install → 0600 in 0700 dir, install reads the path it just wrote, same-user gains nothing; (c) a legit-looking "install a certificate" wizard normalizing consent → precise copy + the OS dialog *where one exists* stays the ceremony; **on Linux no OS dialog exists** (audit M-1) so the in-wizard click is the only consent — surfaced as Ask A8; (d) NSIS hook running attacker `certutil` from PATH → full System32 path there too, and the hook is `$UpdateMode`-guarded so it can't fire unexpectedly on update (§12 R1); (e) CI real-trust tests mutating a dev's real store → all `#[ignore]` + temp DB/keychain; Windows is the only real-store touch, cleans by **serial** in `always()`.
