HTTPS by Default + First-Run Onboarding Wizard

Right now the accelerator only speaks encrypted HTTPS to your browser if you're on a Mac and dig into Settings to switch on "Safari Support." This plan makes an encrypted connection the default on all three operating systems, and adds a friendly first-launch setup screen that walks you through three choices — turn on HTTPS, start the app on login, keep it auto-updated — with one Start button that does all of it. You stay in control: you can decline the certificate and everything else still applies. Under the hood it teaches the little proving app how to install its own local certificate into your browsers' trust stores on Windows and Linux (not just macOS), and teaches the SDK that dApps use to prefer the encrypted channel when it's actually working.

Why this tier was chosen (deep)

The blueprint rubric counts six risk dimensions. Three came back HIGH, which forces deep (three independent plans, double audit, contradiction-check, final pass):

Process run: 3 parallel planning legs (main + codex + fable) → consolidated into one plan with a decision ledger → a contradiction-check + a fresh hostile audit (both "conditional approve") → all High/Critical findings folded → a final fresh-context codex pass ("conditional approve", no High/Critical survived, its Med fixes folded too). Full trail in plan.md §6, §9, §12. All 13 audit resolutions are R1–R13.

The catch the audits caught (worth knowing): the Windows uninstaller hook that removes the certificate would have fired on every auto-update (Tauri runs the old uninstaller when installing over itself), silently breaking HTTPS after each update. It's now guarded so it only runs on a real uninstall — and that guard has to be correct in the very first release that ships it, because a release can't fix its own uninstaller after the fact.

What "HTTPS by default" really buys (no overclaim)

Honest framing, because it drove the design: Chrome and Firefox already reach the app fine over plain HTTP on localhost, so for them the encrypted channel is mostly about parity and positioning. The real security gain is narrow — it stops a different, non-admin user on the same machine from impersonating the accelerator and harvesting your transaction's private data, and only dApps that opt into the strict httpsOnly mode get that guarantee enforced. A same-user attacker is still game-over (they can read the key file directly). So the wizard copy promises an "encrypted, authenticated local channel," never "your traffic is now safe."

Phases

Phase 1 — Rename & config migration

safari_support becomes https_enabled; old configs load unchanged; a new marker remembers whether you've seen the wizard. No behavior change yet.

Proves itself by: migration unit tests (old key loads, new key saves) + lint + typecheck, all green.

Phase 2 — SDK prefers HTTPS (safely)

The SDK now prefers the encrypted channel — but only when it answers healthily — and adds a strict httpsOnly mode for dApps. Proven not to add any delay when HTTPS isn't there.

Proves itself by: SDK unit tests including a wall-clock assertion that the no-HTTPS path is as fast as today + a "broken HTTPS must not win over healthy HTTP" test.

Phase 3 — Linux trust + first real cert-trust CI

Install the certificate into Linux browser trust stores (system NSS + every Firefox profile), no root needed, honest per-browser status. First CI job that runs real certutil and validates a full certificate chain.

Proves itself by: a Linux CI job that installs, chain-validates, rotates, and removes a real cert in a throwaway store.

Phase 4 — Windows trust + renewal UI + 3-OS cert CI

Same for Windows (CurrentUser Root). Replaces today's context-free background password prompt with a friendly "renew your certificate" window. Cert-trust CI extends to Windows + macOS.

Proves itself by: real add/verify/remove cert tests green on all 3 OS runners (with a spike first for the macOS/Windows CI quirks).

Phase 5 — The onboarding wizard

The actual first-launch screen: three toggles, one Start, honest partial-failure handling (if the cert step fails, HTTPS stays off with a Retry and the other choices still apply). New installs and existing users both see it once.

Proves itself by: Playwright mock tests for every wizard state + a real WebDriver wizard test on all 3 OSes.

Phase 6 — Packaging & safe uninstall

The .deb pulls in the NSS tool; the Windows uninstaller cleans up the certificate (guarded so it never fires on update); AppImage degrades gracefully.

Proves itself by: build smokes green + the critical "certificate survives an auto-update" assertion.

Phase 7 — Docs & closeout

Fix the stale docs (they still claim Windows is unsupported), document the per-OS trust/uninstall story, update both READMEs and CLAUDE.md.

Proves itself by: full test + actionlint green; no stray "safari_support" left in code.

Human context

Browsers don't share one trust store. macOS has the Keychain; Windows has its own cert store; Linux browsers each read their own little database (and the tool to write it, certutil, isn't always installed). That's why "install a certificate everywhere" is genuinely a multi-week, three-codepath job and not a checkbox — and why Linux gets a totally silent install (no OS password dialog exists there) while macOS pops the familiar password prompt. The certificate we install is deliberately toothless: the signing key is destroyed the instant it's used, so even though browsers trust it, it can't be abused to forge certificates for other websites.
