Bringing the Accelerator to Windows

Plain-English companion to plan.md. Why it's finally possible, what's genuinely hard, and the one open question.

The app does its heavy lifting by shipping a little engine called bb (Barretenberg) — the thing that actually crunches the zero-knowledge proofs. Until now, bb didn't come in a Windows flavour, so the app couldn't either. Aztec just started publishing a Windows bb, so the door is open.

The lucky break 🍀

We downloaded and X-rayed the actual Windows bb.exe. The big fear with Windows is "DLL hell" — programs that need a pile of extra system libraries that may or may not be on the user's PC. Turns out bb.exe is a single, self-contained file that only needs libraries already built into every modern Windows. No extra installs, no redistributables. That kills the scariest risk before we start.

The plan, in eight steps

P0 Teach the build to grab the Windows bb.exe (it lives in a different place than the Mac/Linux one) and verify its checksum so nobody can swap in a tampered one.
P1 Make Tauri actually produce a Windows installer (a -setup.exe) + the auto-update package.
P2 Fill the Windows-shaped holes in the code. One is urgent: a piece of "restart-if-I-crash" code only exists for Mac/Linux, and the app literally won't compile for Windows until we add a Windows version.
P3 Wire up auto-update so Windows users get a "windows" entry in the update file.
P4 The hard part — robot tests that install the app, auto-update it, and a fake-website test that proves a tampered update is rejected. (More below.)
P5 Plug Windows into the release pipeline so a release builds + ships the Windows files.
P6 A dry-run "release-candidate" to shake out surprises, with the Windows tests running as advisory first (they report but don't block) — the same careful rollout we used for Linux.
P7 Ship it for real, with all the Windows tests now required.

The make-or-break: testing auto-update on a robot 🤖

The riskiest piece is proving the app can quietly update itself on a Windows machine with nobody clicking anything. The test installs an old version, stands up a fake copy of our website on the test machine, and watches the old app download + install + relaunch as the new version — then checks it's alive and reports the new version.

Why it can work with no clicks: we install "just for this user" (into the user's own folder, no admin password), which lets the update happen silently. Two things only this test can settle: does Windows really do it click-free, and does the app grab its network port back fast enough after the restart (the exact bug that bit us on Linux — and Windows is a bit worse at it). The test is the experiment that answers both.

The honest trade-off: unsigned for now

Properly "signing" a Windows app (so it doesn't show a scary "unknown publisher" warning) needs a certificate that costs money + takes time to obtain. We're shipping unsigned first. What that means:

  • First install: users see a SmartScreen "unknown publisher" warning and click "More info → Run anyway." Mild friction.
  • Updates are still cryptographically safe. This is the important bit: the app verifies every update with a separate cryptographic signature (minisign) before running it. So even unsigned, nobody can sneak a malicious update past it — and a test proves exactly that by trying (and failing) to push a tampered one.

Signing is a clearly-scoped follow-up, not a blocker.

The one open question 🤔

How should the app restart itself if it crashes on Windows? Mac and Linux have a built-in "babysitter" for this. Windows doesn't have a clean per-user equivalent. Two options: use Windows' Task Scheduler (built-in, no extra files — my pick), or ship a tiny separate "watchdog" program (more reliable at telling a crash from "the user quit on purpose," but it adds a second file to the bundle — the exact kind of thing that caused a past breakage). The plan picks Task Scheduler and proves it with a "kill the app, did it come back?" test — but this is the call I'd most like your gut-check on.
