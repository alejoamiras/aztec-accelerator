**Issues found.**

1. The new headless `AZTEC_BB_VERSION` path is not purely a `/health` reporting change; it now affects `/prove` behavior when the env is unset. In [main.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/server/src/main.rs:68), headless only injects `bundled_version` from `std::env::var("AZTEC_BB_VERSION")`. In [prove.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/server/prove.rs:59), that value is used for the “requested version == bundled version” fast-path that skips a download. If the env is missing, `DEFAULT_BB_VERSION = "unknown"` ([server.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/server.rs:32)) preserves the old fallback value from [build.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/build.rs:6), but not the old delivery mechanism: pre-extraction, headless inherited a compile-time value via `src-tauri/build.rs`; post-extraction, it does not. That makes the fallback materially more reachable in manual/release headless runs, and can force an unnecessary download or offline failure instead of using the local bb resolution chain in [bb.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/bb.rs:27). I’d count that as behavior beyond the two documented `/health` changes.

2. The new `/health.version` injection is wired correctly, but not regression-tested through the real binaries. GUI injects it in [src-tauri main.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/main.rs:345) and headless injects it in [server main.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/server/src/main.rs:67); core also tests the handler path. But the PR smoke in [.github/workflows/accelerator.yml](/Users/alejoamiras/Projects/aztec-accelerator/.github/workflows/accelerator.yml:159) only asserts `.status` and `.aztec_version`, not `.version`. So a future constructor omission would survive the split-PR CI and silently fall back to the core crate version again. This is a test gap, not a current defect.

Otherwise, the extraction looks clean. I don’t see a missing or shadowed export in the wrapper: [src-tauri/src/server.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/server.rs:1) is just `pub use accelerator_core::server::*` plus the GUI-local `start_https`, and the current consumers in [main.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/main.rs:9), [commands.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/commands.rs:24), and [windows.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/windows.rs:108) are all satisfied by that surface. The `https_bound` wiring also still looks correct: the adapter captures the shared flag before `router(state)` consumes the state and flips it only after a successful bind in [tls.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/server/tls.rs:19), and `/health` gates `https_port` on that same flag in [core server.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/server.rs:173).

Supply-chain-wise, I don’t see a new third-party dependency or widened hot path. `accelerator-core`’s dependency list in [core/Cargo.toml](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/Cargo.toml:17) is a carve-out of crates already present in the old `src-tauri` package, while headless drops the Tauri/rustls/rcgen tree in [server/Cargo.toml](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/server/Cargo.toml:16). The bb download+execute path itself is moved, not changed, in [bb.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/bb.rs:18) and `versions.rs`.
