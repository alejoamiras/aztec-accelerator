Core Extraction — splitting the "engine" out of the desktop app

The Aztec Accelerator ships two things from one Rust codebase: a desktop app (system-tray GUI, the thing you click) and a headless server (no GUI — other people run it in their CI to make Aztec proving fast). Today the headless server is built on top of the desktop app's code, so it drags in the entire desktop GUI toolkit (Tauri, a web-view engine, certificate libraries…) — 475 packages — even though it never shows a window. This plan carves out a clean, GUI-free engine crate (accelerator-core) that both the desktop app and the headless server build on. The headless server then stops compiling all that GUI machinery — smaller, faster, fewer places for a supply-chain attack to hide.

Why this tier was chosen (/blueprint deep)

Risk rubric: blast radius (HIGH — it touches the release pipeline of a code-signed, auto-updating app), external coupling (MED-HIGH — the headless artifact runs in other projects' CI), security sensitivity (MED — supply-chain surface of that artifact). Two-plus HIGH dimensions → deep. We ran three independent plans (me + codex + an opus architect), then a two-model contradiction-check that caught two real blocking bugs before a single line was written.

The one big decision (and the dissent that won)

How clean should the cut be? Two options: (a) move everything non-GUI into the engine and switch the GUI-only bits (HTTPS, certificates) on/off with build flags; or (b) move only what the headless server actually needs, leaving the certificate/HTTPS code in the desktop app. Two of three planners picked (a). Codex argued (b): "don't make leanness a convention (a flag someone can flip) when you can make it structural (the code simply isn't there)." We verified the desktop's HTTPS code already plugs into a shared router() — so (b) is cheap — and adopted it. The headless server now structurally cannot pull in the certificate libraries, which is strictly safer.

Phases (each a small, safe, separately-shippable PR)

Phase 0 — Make the version label explicit

Before moving anything: today the app's reported version is baked in by "whichever crate compiles this file." If we just moved the file, the headless server would silently report the wrong version. So first we pass the version in explicitly. (Both reviewers caught that this is required in two spots, not one — miss it and the new engine won't even compile.)

Phase 1 — Create the engine and point the desktop app at it (one safe step)

Make accelerator-core, move the GUI-free modules (HTTP server, proving, version resolution, config, auth) into it, and switch the desktop app to build on it (a thin "wrapper" keeps every existing import path working, so the GUI code barely changes). This must be one step — the instant the modules move out, the desktop app has to already point at the engine or it won't build. The engine needs no build script at all now.

Phase 2 — Point the headless server at the engine (the payoff)

The headless server drops its dependency on the desktop app and builds on the engine directly. We assert its dependency tree no longer contains Tauri / the GUI toolkit / certificate + TLS-serving libraries (it keeps the small HTTP client it needs to download bb), and measure how much smaller + faster it got.

Phase 3 — Speed up the headless CI build

Stop installing desktop-only stuff (Linux web-view libraries, the Tauri pre-build) on the headless build legs (there are three), wire up the headless version source, add a CI tripwire that fails if the GUI deps ever sneak back in, and report the measured before/after numbers.

Phase 4 — Prove nothing broke (release dry-run)

A release-candidate dry-run runs the real auto-update tests on macOS / Linux / Windows to prove the signed desktop app still updates correctly. This step dispatches a release, so it's yours to trigger — the autonomous run stops here.

Human context

The desktop app is code-signed and notarized (Apple's tamper-seal) and updates itself in place. A past release (1.0.1) broke that seal by accidentally bundling a second program into the app. We confirmed this plan doesn't repeat that: adding a shared library doesn't add a second program, so the signed bundle's shape is unchanged. That's why the cut is safe — but the release dry-run still proves it.

Open questions — now resolved

  • Where does the headless server get its "default bb version" from once we stop baking it in? → Inject it from the packaging step (not a stale compile-time string). Decided.
  • What should /health report as the version? → The injected release version, not the engine crate's internal version. Decided.
  • Could a future shared workspace re-introduce the GUI deps into headless? → Option (b) makes that structurally impossible; a CI tripwire guards against it anyway.
