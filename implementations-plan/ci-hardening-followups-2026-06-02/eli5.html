Three little CI clean-ups

Plain-English companion to plan.md. None of these change the app — they tidy up how the robots build & release it.

1. Throw away the old master key low risk

After a release, a few robots do chores for us — bump version numbers, pull in new @aztec library versions — and to do that they open and merge pull requests. They unlock the door with a personal key (a "PAT") that belongs to you and never expires.

We already made a better key for the release robot: a little key that only opens two specific doors and expires after an hour. This step hands that same small key to the remaining robots, then we shred the old never-expiring one.

Like: swapping the building master key that a contractor kept in their glovebox for a day-pass that only opens the two rooms they actually work in.

Why bother: if the old key ever leaked (they sometimes show up in logs), someone could unlock everything. The day-pass can't, and it's expired by the time anyone finds it. Bonus: these robots can be run on-demand, so we can test the swap for real before deleting the old key.

2. Stop running the full test suite for doc edits fiddly — test hard

Our robots skip the big 15-minute test run when a change clearly can't affect the code — like editing planning notes. We recently fixed a bug there. But there's a leftover gap: if you edit a read-me or doc that happens to live inside a code folder, the robots still run the whole suite, for nothing.

This teaches the filter: "run the tests if real code changed — but if it's only a doc, skip, even if that doc sits next to code."

Like: a smoke alarm that currently goes off when you light a candle in the garage. We want it to ignore the candle but still scream for an actual fire — and the wiring to tell those apart is delicate.

Why it's the risky one: if we wire the filter wrong, it could do the opposite — skip the tests for actual code changes, letting a real bug slip through. That's much worse than wasting 15 minutes on a doc. So before this ships, we run a little checklist: edit some code → tests must run; edit only a doc → tests must skip; and three more cases. It only merges when all of them behave.

3. Actually switch on the "no brand-new ingredients" rule low risk, real value

Your own security rules say: never install a software ingredient (dependency) that was published less than 7 days ago. The idea — if a popular package gets hijacked and a poisoned version is published, that bad version often gets noticed and pulled within days. Waiting a week means we never grab it during that danger window.

The catch we found: that rule was never actually turned on anywhere in this project. This step is a single line of config that switches it on for the whole repo.

Like: a kitchen policy of "don't cook with any ingredient delivered in the last week" — except the policy was written on the wall but the fridge wasn't enforcing it. Now it does.

Why it's safe to add: it only affects new or changed ingredients; everything already in our locked recipe keeps working untouched. One line, real protection, basically free.

The plan in one breath

Do the easy supply-chain line first (3), then the delicate test-filter rewrite with its safety checklist (2), then hand the robots their new keys and shred the old one (1). Each ships as its own small change, reviewed once by a second AI before it lands.
