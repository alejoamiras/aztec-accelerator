Making CI faster (and stop the hangs)

Plain-language companion to plan.md · 2026-05-29

Two annoyances: our test pipeline rebuilds something it shouldn't have to, and one test job keeps freezing. Both are small, low-risk fixes.

Problem 1: we throw away a cache we should keep

"Caching" in CI = saving the result of slow work so the next run reuses it instead of redoing it. Like keeping your groceries in the fridge instead of re-shopping every meal.

We compile a Rust program (the "headless server") in one of the test jobs. Compiling it from scratch takes minutes. We do have a cache for it — but it's pointing at the wrong shelf.

Back when we split the server into its own sub-project (the 1.0.2 fix), the place it gets built moved — from src-tauri/target to server/target. But this one test job's cache still points at the old location. So every run, it opens an empty fridge, finds nothing, and re-shops (re-compiles) the whole thing.

The cost, measured: the test job that builds the server takes 5m40s; the equivalent job that doesn't build it takes 1m56s. About 3.5 minutes wasted every run, on every PR, for weeks.

The fix

Point the cache at the right shelf: server → target (not src-tauri → target). One line. We double-checked every other build job — they all cache correctly; this was the only stray one.

Subtle point both reviewers debated: should we cache both shelves? No — in this particular job everything gets built into the server shelf, so caching the other one would just save an empty folder. Cache exactly what's used.

Problem 2: the test job that keeps freezing

One job (the "Local Network E2E") has hung ~5 times today. It freezes while installing the browser it needs for testing (Playwright/Chromium). We've had to cancel + rerun it over and over.

It's not a missing cache — the cache exists. The freeze is the install command itself stalling: partly an apt (system-package) step that runs every time, partly a slow download server. When it stalls, nothing times it out, so it just sits there until the 45-minute job limit.

The fix (three small, robust changes)

  • Time-box it: give the install step a 5-minute limit so a stall fails fast instead of hanging for 45 minutes.
  • Retry it: 3 attempts with backoff, so a one-off network blip self-heals.
  • Fix the cache key: today different jobs key the cache differently and coarsely. Key it on the exact browser version so the cache reliably hits (and we stop re-downloading).

We deliberately keep using Playwright's own browser (not a system one) because that's what the tests are written against — swapping it would change what we're actually testing.

What we're deliberately NOT doing

You asked for high-ROI, low-risk only — so no exotic build-cache tools (sccache), no restructuring the slow release build, no touching the release pipeline we just stabilized. Just the two fixes above, both pure CI-config (revert = delete the change, no impact on what we ship).

How we'll know it worked

We record job timings before and after (comparing "warm cache" runs, since the very first run after a cache change is always cold). Target: the server-building job drops ~3 minutes, and the Playwright hang basically disappears.

Reviewed by a second AI model (codex) and an independent reviewer agent; both signed off ("ship it"). One open tweak: confirm the exact one-liner that reads the installed Playwright version for the cache key.
