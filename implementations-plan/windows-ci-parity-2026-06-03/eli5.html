Windows CI Parity — the last three follow-ups

The Windows version of the app now works and proved itself end-to-end (a green dry-run release built the installer and did a real silent, signature-checked auto-update). This plan is the finish-the-job list: three small-but-careful tasks so Windows is held to the exact same standard as Mac and Linux in our automated checks. Two of these were re-aimed after an audit caught the original plan pointing at the wrong files.

Phases

Phase 1 — #95: fix a stale bookkeeping file

Every crate has a Cargo.lock that should record the current version number. The server one still says an old version (1.0.2-rc.1) while everything else says 1.0.4-rc.1. It's harmless today (nothing enforces it), but untidy. We sync it, and — more importantly — fix the release robot so it updates the lock whenever it bumps the version, so this can't drift again. Plain find-and-replace; no heavy tooling.

Phase 2 — #96: test updating while "auto-restart-on-crash" is ON

Windows recovers from crashes with a timer that relaunches the app if it dies. That timer has a tripwire: if it fires during an update (while files are being swapped), it could corrupt the update. We already built a guard that switches the timer off just before the update and back on after — and hardened it heavily. This phase finally tests that guard by running an update with the timer armed. Mac/Linux don't need this (their crash-recovery can't fire during a clean update), so this is exactly the kind of Windows-specific risk that "same assurance as Linux" actually requires us to cover.

Phase 3 — P5: make the Windows update-check REQUIRED (and the big decision)

Right now the Windows update-test is "advisory" — it runs but can't block a bad release. Mac/Linux update-tests are required gates. To match them, we make Windows' required too. But the audit caught something: the Windows test currently updates a stand-in app signed with a throwaway key — it never touches the real signed installer we actually ship. Mac/Linux test the real thing. So making the stand-in test "required" would be a gate that doesn't check what we release.

The decision (yours): Option A — make the stand-in test required now (quick, but it's a weaker gate), with the real-installer upgrade as a fast-follow. Or Option B (recommended) — first upgrade the test to use the real signed installer (checked against the real signing key), then make it required. More work, but it's the honest version of "same as Mac/Linux." Either way we also add a one-click "undo" runbook so a flaky Windows test can't jam the release line.

Human context

Why so careful about a "required check"? Our release pipeline runs one release at a time. If a required Windows check hangs or flakes, it can stall all releases until it clears. So "make it required" comes with "and make sure it can't wedge us" — a tight timeout plus a documented way to instantly demote it back to advisory.

Why not just reuse the Mac/Linux approach wholesale? Their test downloads the previous real release as the "before" version. Windows has no previous real release yet (this dry-run was the first, and it's marked pre-release, which their logic deliberately skips). So the full copy would actually break the very first real Windows release. Option B sidesteps that by keeping a synthetic "before" version while still testing the real "after" installer.

Open questions

  • P5: Option A (lean) or Option B (real-installer parity)? The recommendation is B — it's the only version that makes the required gate actually check the shipped artifact, the way Mac/Linux do. A is faster but ships a weaker gate.
  • Keep or delete the 1.0.4-rc.1 pre-release? Keeping it gives a future "before" baseline once a real Windows release exists.
  • Re-enable commit signing going forward (the long autonomous run committed unsigned; it's harmless because merges re-sign, but worth turning back on).
