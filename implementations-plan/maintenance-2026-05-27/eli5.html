PR 1 — the boring hygiene one

What

Combine all the small dependabot bumps (lint-staged, commitlint, vite-plugin-node-polyfills, axum, tracing-appender, tokio, tauri-build) into one PR. Then delete .github/dependabot.yml so dependabot stops opening weekly PRs.

Why

Most of these PRs sat for weeks. We don't auto-merge them. Instead of trickling them in one-at-a-time, do one batch every now and then. Dependabot's weekly schedule isn't earning its keep.

The catch

Dependabot also files PRs for security CVEs, not just version bumps. Deleting the config removes that signal too. Worse: the repo-level toggles for "Dependabot alerts" and "automated security fixes" are also off (checked via gh api). After this PR, we have zero automated security signal.

You explicitly accepted this trade-off in exchange for less noise. The plan documents it in the PR body so it's not silently regressing later.

PR 2 — Tauri 2.11

What

Bump tauri 2.10.3 → 2.11.0 alone.

Why on its own

Tauri ships the entire desktop-app runtime: the tray icon, the window manager, IPC, the updater plugin. A "minor" version isn't actually small. If something regresses, we want to roll back just Tauri without losing all the other bumps from PR 1.

The safety net

The WebDriver end-to-end tests on the PR-gate workflow will exercise the tray, the popup, and the update plugin against the new Tauri. If those go red, we know before merge.

PR 3 — the headless tarball

What you asked for

You're running E2E tests in another project that uses the accelerator. WASM proving is slow. If your CI could just install the accelerator and run it locally, your tests would speed up. Today, the only way is to clone our repo and cargo build — annoying.

What we already have

There's a binary called accelerator-server in our repo. It's the same HTTP server the desktop app runs, but with no tray and no window — pure background process. We use it on our own CI already. It just isn't published anywhere.

What PR 3 does

  • Adds 4 cross-compiled tarballs to every release: macOS Apple Silicon, macOS Intel, Linux x86_64, Linux ARM64.
  • Each tarball gets a SHA-256 sidecar file for integrity checking.
  • README + release notes get a section that says "here's how to install this on CI" with a one-liner.

The important caveat

This binary is for CI test acceleration only. Not production.

The "auth check" (ALLOWED_ORIGINS) only works against web browsers — they send an Origin header that the server inspects. Anything that isn't a browser (curl, another process on the same machine) can skip that header and the server lets it in.

On a GitHub-hosted CI runner that's fine: each job gets a fresh, single-tenant VM. Nothing else is on the machine. The "only listens on localhost" rule IS the security boundary.

On a shared or self-hosted CI runner, other jobs or processes could hit the accelerator. The release notes say this loudly, in the warning box at the top of the headless section. Don't deploy this as a service.

What we are deliberately NOT doing in PR 3

  • Not adding a real auth token (would need a follow-up plan).
  • Not signing the binary or adding GitHub artifact attestation.
  • Not making the server's port configurable (today: hardcoded 59833 — the README at line 62 currently lies about this and PR 3 fixes that one-liner).
  • Not adding a Windows tarball or an npm wrapper package.

What codex caught that I had wrong

Two embarrassing ones worth flagging so you know what changed in v2 of the plan:

  • I claimed there was a bunfig.toml with a 7-day npm min-age gate. There isn't. I pulled that from personal-default memory and didn't verify. Track A's "supply chain mitigation" section was hand-wavy as a result.
  • I claimed bun audit covers the CVE-PR loss when we delete dependabot.yml. It doesn't — bun audit doesn't open PRs, doesn't cover cargo, doesn't cover github-actions. The plan now says clearly that we're accepting zero automated CVE signal.

Codex also flagged: the ALLOWED_ORIGINS auth model is misleading, the port collision on multi-job hosts is real, reqwest in our Cargo pulls native-tls so Linux ARM64 needs libssl-dev, sha256sum isn't on macOS (use shasum -a 256), and the release workflow tags before building so a failed headless build leaves a tag pushed. All of these are now in the plan.
