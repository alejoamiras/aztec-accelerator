Aztec rc.1 → rc.2 bump + SDK release

Aztec's testnet upgraded to 5.0.0-rc.2 today. This plan moves our SDK and playground onto rc.2, re-publishes the SDK to npm (on the testnet tag, never latest), and keeps the live playground working — including the one thing that almost slipped through: the sponsored "free gas" contract has to be re-deployed on testnet because rc.2 changed its code (and therefore its address).

Why mid was chosen

At first glance this is a one-line dependency bump (you guessed "no breaking changes"). The rubric scored 0 high-risk dimensions up front — so mid looked slightly heavy. But the dual audit earned its keep: it found a real migration-cost hit (the fee contract's address moves, so the testnet deployment we funded last time is now orphaned) and a gap where our test looked like it proved production-safety but didn't. mid (two independent reviewers + a fresh final pass) was the right call.

The headline finding. Your "no breaking changes" is right about our TypeScript code (it compiles clean) — but the sponsored fee contract was recompiled in rc.2, which shifts its on-chain address. The playground derives that address and pays gas through it; if we don't re-deploy + re-fund it at the new address on testnet, the live playground's transactions fail. The plan now makes that a required step.

Phases

P1: Bump the versions + regenerate the lockfile

Change every @aztec/* version from rc.1 to rc.2 (24 lines across two files), bump a one-line "cache version" that prevents a known in-browser crash when the proving format changes, and regenerate the lockfile. rc.2 is ~1 day old and the repo blocks installing anything under 7 days old (a supply-chain safety gate) — so we override that gate locally only (never in CI), then double-check the lockfile didn't quietly pull in any other brand-new packages.

Proves itself by: both installs succeed, the type-checker + linter + unit tests + playground build all pass, and the lockfile diff shows only Aztec version changes.

P2: Land the bump (with the real proving tests) and auto-merge

Open a PR. CI runs the SDK end-to-end test with the accelerator serving the native rc.2 proving binary — proving the prove/serialize/proof format still works. We also add the missing piece: a test that actually downloads the rc.2 binary and checks its signature, because that's the path real installed users take (their app shipped with an older binary and fetches rc.2 on demand) — the normal test skips that path.

Proves itself by: all CI gates green (native rc.2 e2e asserts it used the fast path, not the fallback) + the download-and-verify test passes; PR auto-merges.

P3: Re-deploy the fee contract, publish the SDK, redeploy + smoke the playground

First, re-deploy and re-fund the rc.2 sponsored-fee contract on testnet (a small Ethereum/Sepolia bridge transaction with the disposable wallet — same flow as last time). Then publish the SDK to npm on the testnet tag and redeploy the playground. Finally, a human opens the live playground and runs a real transaction to confirm fees are paid through the new contract. This whole phase is outward-facing (real money on L1, an npm publish, a live deploy) so it's dispatched by you, not automatically.

Proves itself by: the rc.2 fee contract is live on testnet, npm shows testnet → rc.2 (and latest still 4.x), the live bundle is on rc.2, and a browser smoke transaction succeeds.

Human context

SDK-only model. The installed accelerator app downloads the matching proving binary at runtime, so a version bump normally ships as just an SDK re-publish — no new app release. We verified the app's proving code is byte-identical to the deployed 1.0.6, so that still holds here.

Why re-deploy the fee contract. On Aztec, a contract's address is derived from its compiled code. rc.2 recompiled the sponsored-fee contract, so the "same" contract now lives at a different address. The one we funded during the rc.1 work is now unreachable from rc.2 — hence the re-deploy.
