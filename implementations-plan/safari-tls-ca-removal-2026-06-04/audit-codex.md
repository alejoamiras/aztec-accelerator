**Verdict**

`high-critical`

**Findings**

- `HIGH` Rotation is not atomic as a *cert set* and can wedge HTTPS persistently. `rotate()` renames `ca.pem`, `localhost.pem`, and `localhost.key` one-by-one ([certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:285), [certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:286), [certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:287)). If the app crashes or a rename fails after only one of leaf/key swaps, disk can hold a new leaf with the old key, or vice versa. `certs_exist()` won’t catch that because it only checks file presence plus leaf expiry ([certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:90)), then startup skips regeneration and only fails later in `load_rustls_config()` ([certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:203), [main.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/main.rs:80)). Result: Safari support can stay enabled but HTTPS never starts until manual cleanup.

- `MEDIUM` I do not see an app runtime path that writes `ca.key` to disk anymore, but “discarded” is stronger than what the code actually guarantees. `write_new_cert_set()` never serializes the CA key ([certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:107)), and the cert objects written are only `ca_cert.pem()` / `leaf_cert.pem()` / `leaf_key.serialize_pem()` ([certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:112)). That closes the on-disk minting HIGH. But this crate depends on `rcgen` without enabling its optional `zeroize` feature ([Cargo.toml](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/Cargo.toml:49)), so the CA private key is logically dropped, not scrubbed. A post-compromise heap/core-dump attacker could still recover it from process memory.

- `MEDIUM` Rotation can block startup on the macOS trust prompt. `regenerate_leaf_if_expiring()` runs synchronously during startup ([main.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/main.rs:70), [main.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/main.rs:359)), and `add_trusted_cert()` waits on `security ... .output()` ([certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:317)). That is fail-closed, not fail-open, but on autostart/headless-ish launch it can hang the app until the auth flow resolves.

- `MINOR` Keychain cleanup is best-effort and name-based on discovery. `ca_keychain_sha1()` takes the first `find-certificate -Z -c "Aztec Accelerator Local CA"` match ([certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:348)). That should not delete the *new* anchor, because the SHA-1 is captured before adding it, and there is no shell injection risk because this uses `Command`, not a shell. But duplicate legacy anchors can remain.

**What’s Solid**

No app runtime path writes `ca.key`; the keyless-anchor model does prevent disk-only attackers from minting new certs. `migrate_legacy_ca_key()` is safe because HTTPS only loads `localhost.pem` and `localhost.key`, never `ca.key` ([certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:145), [certs.rs](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/certs.rs:205)). The staged `*.new` files and `write_pem_file()`’s `*.tmp` files do not collide.
