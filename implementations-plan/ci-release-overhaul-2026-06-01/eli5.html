Making releases faster — and the auto-update test actually count

Plain-language companion to plan.md. Two goals, one pipeline.

A "release" is an assembly line: check the app works → build & sign it for Mac/Linux → smoke-test → tag → publish. Right now it's slower than it needs to be, and the test that checks "will auto-updating brick people's apps?" is just a sticky note, not a stop sign. We're fixing both.

1. Inspect while you build, not before free · biggest win

Today a safety inspector (the "WebDriver gate") makes the whole factory wait before building starts — 5 minutes on a good day, 20 on a bad one. Only then does the slow part (building + Apple notarization) begin.

Fix: run the inspection at the same time as the build. If the inspector fails the app, a tiny watchdog pulls the alarm and stops the line so we don't tag a broken release. And as a safety net, tag/publish still refuse to run unless the inspection passed — so even if the alarm misfires, nothing bad ships.

Why: the build (~8 min) is usually faster than the cold inspection (~20 min), so doing them together basically hides the inspection's time entirely. Plus: if you push a fix twice, the old run now auto-cancels instead of finishing pointlessly.

2. Turn the auto-update test into a real stop sign the gating goal

Version 1.0.1 shipped a bug where the app froze while auto-updating — on real people's machines. We built a test that reproduces that exact moment (install old version → let it update → check it relaunches). But it's advisory: it can go red and the release ships anyway.

Now it blocks the release, and we widen the net:

Mac (Apple Silicon + Intel) — blocking immediately; also verify the Intel installer is properly notarized (today we only check Apple Silicon).

Linux (AppImage) — brand-new: a Linux version of the same test. It's the riskiest piece (Linux self-updates are finicky in CI), so it rides along as advisory until one real practice-release proves it works, then it becomes blocking too.

Why this won't slow releases down: the update test runs in parallel with the existing smoke test, finishing inside the time the build already takes. So we add a real safety gate and still come out faster overall (because of #1).

One judgment call (corrected after review): the "tampered file must be rejected" test (proving the signature check works) blocks both practice and real releases. We originally planned to skip it on real releases, but a real release is cut from whatever's in the repo at that moment — so a green practice-run doesn't guarantee the real one. It's safe to block because this test uses our own local test-server (no flaky internet download), so it won't get stuck.

3. Stop recompiling the same code from scratch free

Building the Rust app is the slow part. We have a "save your work" shelf (a cache) so we don't recompile every time — but the shelf labels don't match between jobs, so almost nothing ever gets reused. Every release and most PRs recompile from cold.

Fix: give the cache consistent labels so the same compiled pieces are reused across releases and PRs, and keep each shelf small and tidy (don't mix unrelated builds). Optionally, a small job re-stocks the shelf on the main branch weekly so it never goes stale — exactly the trick that just killed the Playwright flake.

Why: this is where "releases take a while" actually comes from — cold compiles. Right labels = warm compiles = minutes back, for free.

4. Don't run heavy tests for unrelated changes free

Right now, changing one unrelated file can trigger the slow Mac crash-tests and full Rust builds even when they can't possibly be affected. We'll teach the pipeline to tell "this touched the desktop app" apart from "this touched only the SDK", and skip the heavy Mac/Rust jobs when they're irrelevant.

Why: you wait on PR checks every day. This trims the everyday wait without losing any real coverage.

5. (Optional) Rent a faster Mac for the one slow step paid · only if needed

The slowest single piece is the Intel Mac build. GitHub will rent us a beefier Mac for it — a one-line change, billed per minute (trivial for the rare release). We'll only pull this lever if, after the free fixes, the Intel build is still the bottleneck. Measure first, spend second.

What we're being careful about

• The "pull the alarm" watchdog has powerful permissions, so it's a bare-minimum job — no code checkout, no secrets, nothing to hijack.
• The update test fakes being our website inside a throwaway machine and needs no signing key — it serves our already-signed release and the old app verifies it. We're hardening the fake-certificate cleanup so it can never leak.
• Shared caches are safe because GitHub keeps each PR's cache walled off from the main one — a malicious PR can't poison what everyone builds from.
• A flaky blocking gate could jam all releases, so the new Linux gate stays advisory until proven, and emergencies would use a separate, owner-only "break-glass" — never a quiet skip switch.

Open question we deferred

The Linux auto-update has two unknowns the review surfaced. First, we may be shipping the wrong file format for Linux updates — if so, Linux auto-update is already quietly broken, and this test's real job is to catch that and fix it. Second, the Linux self-update may simply not run in GitHub's Linux machines (a technical thing called FUSE). So before wiring Linux as a blocker, we do a quick proof-of-concept on both. If it genuinely can't work headlessly, Linux stays advisory and we note why — we won't fake a green check.
