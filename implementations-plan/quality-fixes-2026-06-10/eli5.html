Quality fixes — the 15 from the maintainability audit

Two days ago a maintainability audit found 15 "code smells" — places where the app works fine today but is expensive to change. This plan fixes all 15, in 3 pull requests, without changing what the app does. Every change is a refactor: move a thing, name a thing, give a type one job. The test suite proves behavior didn't shift.

Why "mid" ceremony

Rubric: novelty LOW (refactoring known code), blast-radius / irreversibility / migration all LOW (revertible, CI-gated, no data or API change). The one raised dimension is security-sensitivity — three fixes touch paths a security pass hardened last week (the Safari-HTTPS cert sequence, the origin-canonicalization, the Windows updater). So: mid — one plan + a competing outline, audited by two independent reviewers (codex + a Fable planner). That double-audit earned its keep: it caught five spots where a "harmless refactor" would have silently changed behavior.

What the audit caught (and the plan now prevents). Five refactors looked safe but weren't: (1) the Windows updater deliberately re-arms crash-recovery before restarting — a tidy "guard object" would have skipped that and left a failed update unprotected; (2) the host-guard's error reply is intentionally a different, minimal shape — merging it into the common error type would have changed the wire format the SDK parses; (3) unifying the "save config" helper to one error policy would have made a config-save failure fail a proof the user already approved; (4) the SDK's HTTP/HTTPS memory has a subtle three-way rule that a naive rewrite would flatten; (5) the cert "sequence" is actually two different sequences (launch vs. settings). All five are now explicit conditions in the plan, each with a test written before the change.

The three PRs

PR-1 — the SDK file (one package, no API change)

Four fixes all live in one 440-line file: move the public types into their own types.ts, break up two giant methods, and put the HTTP-vs-HTTPS memory behind one clear rule (the careful three-state one). The published API stays byte-identical — a contract test guards that. Low risk; can land first or in parallel.

PR-2 — the Rust "comment-enforced rule" fixes

The bulk. The audit's big theme: rules kept true by a // must… comment instead of by a type. This PR converts the high-traffic ones into real structure — one function that owns the cert bring-up order (the fix for the bug we hit last week), a typed error channel, a guard object for the updater, a shared config helper, plus moving ~715 lines of misplaced tests. Ordered safest-first; tests written before the risky ones.

PR-3 — the two day-scale ones (cuttable last)

Split the 1000-line "versions" module into three, and finish threading the "version" and "origin" wrapper types through the code instead of dropping back to raw strings. Sequenced last so it can be cut without stranding anything. One risky sub-item (a magic "unknown" string the SDK reads) is carved out and deferred.

Human context

"Behavior-preserving" is the whole game here. The app and the published SDK are in production; the security surface was just audited. So the bar is: the existing tests — especially the security characterization tests and the SDK contract test — must pass unchanged, or the refactor is wrong. Where a fix touches a spot with no test, we write the test first (that's why the audit's "no test pins this" findings matter — they're exactly where a silent regression could slip through).
