Filing down the quality debt — without changing what users see

The /harden quality audit found 15 maintainability problems (plus a bag of small ones) in the accelerator app and its SDK. None are bugs — the app works and just shipped 1.0.4. This plan turns those findings into a careful, step-by-step refactor that makes the code easier to change while keeping every observable behavior identical: same HTTP responses, same tray, same auto-update flow, same proof output. Because the app auto-updates real users, "don't break anything" is the prime directive.

Why this tier (mega-deep)

By the rubric this was borderline deep (we know this code well — low novelty), but you asked for the deepest pass and the scope justifies it: everything including the two riskiest changes (splitting the giant server.rs and its central AppState), a public SDK type-break, and a shipping app that auto-updates. Three independent planners (me + codex + opus) drafted in parallel after four research agents mapped each module. That fan-out already paid off — it caught a wrong assumption in the brief (see Open Questions).

How it's sequenced

0. Safety net first (characterization tests)

Before touching anything risky, write "golden" tests that lock down current behavior — exact error messages, the proving flow, the version logic. Trick: a fake bb binary lets us test the success path without a real prover. If a later refactor changes behavior, these tests go red.

1. Cheap, safe wins

Share the duplicated 60-second timeout; replace 19 hand-built error blobs with one typed object (same wire output); sweep tiny duplications. Low risk, quick.

2–6. The "value objects" + mid refactors

Give versions a real type (AztecVersion) instead of raw strings; make the tray status a typed enum instead of matching on the word "Proving"; split the 149-line downloader into named steps; unify the per-platform crash-recovery code behind one interface. Each is one small PR with its tests.

7. The risky splits (last)

Split AppState into "server core" vs "desktop UI bits", then carve the /prove + authorization workflow out of the 1,335-line server.rs. Done last, behind a release-candidate dry-run, because this is the hot path.

8. The SDK type-break

Make the SDK's status/phase types precise (discriminated unions). This is the one intentionally-breaking change; the in-repo playground updates in the same PR. (See the Open Question about how "breaking" actually works here.)

Human context

The safest way to refactor a live app is many tiny, independently-revertible PRs, each proven behavior-identical by tests written before the change. Three findings look like "cleanups" but are actually behavior changes in disguise (a swallowed error, an origin-matching tweak, a missing recovery path) — the plan keeps today's behavior and splits the real fixes into separate, clearly-labeled PRs you opt into.
