reject (blocking: the plan misstates current `/prove` ordering, understates observable SDK/runtime changes, and leaves key safety/security behaviors under-characterized)

[critical] The plan’s model of `/prove` is already wrong. Current order is auth → body buffer → semaphore → initial `Proving` status → `resolve_version`/download → `bb::prove` [packages/accelerator/src-tauri/src/server.rs:493-537]. The plan’s Phase 0 says `auth→resolve→semaphore`, which is false. If implemented that way, two concurrent requests for the same uncached `x-aztec-version` can race into `download_bb()` before the semaphore; that function uses a fixed `.{version}.tmp` dir plus `remove_dir_all`/rename [packages/accelerator/src-tauri/src/versions.rs:354-370]. That is not behavior-preserving. The harness must pin `body-before-semaphore` and `semaphore-before-resolve`, not just `auth-before-body`.

[high] Q8 is not as wire-safe as claimed. Today `/prove` errors are JSON strings returned as `(StatusCode, String)`, so they go out as `text/plain` [packages/accelerator/src-tauri/src/server.rs:280-285]. If Q8 switches to `Json`/typed DTO, `Content-Type` becomes `application/json`. In `ky`, that changes `HTTPError.data` from plain text/undefined to parsed object [node_modules/ky/distribution/core/Ky.js:478-486]. For 400/500 paths that the SDK rethrows, that is an observable SDK runtime change even if the body bytes stay `{error,message}`. “No coordinated release” is too strong unless `text/plain` is preserved or the SDK change is accepted.

[high] Q10’s “tray timing preserved” claim is under-tested. On an uncached-version prove, current callbacks are `Proving` [server.rs:530-532] → `Downloading bb...` [436-438] → `Proving...` [464-466] → `Idle` via `StatusGuard` [266-276]. The tray animates from substring matching on those exact texts [packages/accelerator/src-tauri/src/main.rs:348-357]. An enum refactor that “cleans this up” to `Downloading` first, or removes the duplicate `Proving`, silently changes observable tray behavior.

[high] Ask A is overstated/misstated. The SDK does have publish-time semver/revision logic and release tags: `resolvePublishVersion()` emits `4.2.0-revision.N` / prerelease revisions, and `_publish-sdk.yml` publishes that exact version [scripts/get-sdk-publish-version.ts], [.github/workflows/_publish-sdk.yml]. The real constraint is narrower: the base version is tied to `@aztec/stdlib`, so you cannot invent an unrelated major without changing release engineering or the Aztec dependency baseline.

[medium] Security coverage is incomplete. Q14 can widen auto-approval: `http://LOCALHOST:5173` is denied today by the manual parser [packages/accelerator/src-tauri/src/authorization.rs:126-147], but a canonicalize-then-match refactor would lowercase it and auto-approve [authorization.rs:21-57]. Q4/Q6 also need the Windows abort-path pinned, not just success ordering: `disable_crash_recovery()==false` must block install and rearm [packages/accelerator/src-tauri/src/updater.rs:91-100]. Q7 must preserve current fail-closed asymmetry: startup resets on missing/invalid certs but only skips on untrusted CA [packages/accelerator/src-tauri/src/main.rs:59-79], while Settings enable has no recovery path [packages/accelerator/src-tauri/src/commands.rs:141-160].

[medium] Q5+Q12 should split. Pairing a private extraction with a public type break kills rollback granularity and muddies review. Land Q5 under existing SDK types first; land Q12 as the isolated breaking PR with consumer migration. Q3-vs-Q8 ordering is not the material dispute; preserving the real `/prove` ordering is.

What’s solid: the plan does account for the audit findings, `Q1≺Q2` and `Q3≺Q11` are correct, and it correctly surfaces Q7/Q9/Q14 as behavior-change traps. The “characterization-first” instinct is right; the problem is that the characterization target is currently missing the highest-risk orderings and a couple of real observable deltas.
