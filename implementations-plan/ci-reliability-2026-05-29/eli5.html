What broke CI, and how we're fixing it

Plain-language companion to plan.md · evidence in diagnosis.md · 2026-05-29

Short version: shipping version 1.0.2 accidentally broke our own test pipeline. The fix is small and safe. This page explains why it happened and what each change does, without jargon.

The story: how releasing 1.0.2 broke the tests

The desktop app checks for updates in the background. When it finds a newer version, it pops up a little "Update available" window.

Think of our automated UI tests like a robot clicking through the app. The robot is told: "look at the Settings window and check the speed label is there." It works fine — until a surprise pop-up window jumps in front. Now the robot is staring at the pop-up, not Settings, and reports "I can't find the speed label!" — and the test fails.

Here's the twist that made it happen now:

  • Our test builds carry the in-development version number 1.0.2-rc.1.
  • For weeks, the newest published release was 1.0.1 — older than our test build — so the update check found nothing and stayed quiet.
  • The moment we published 1.0.2, our test build (1.0.2-rc.1) suddenly saw "1.0.2 is newer than me!" and started popping the update window mid-test.

That's why nothing in our code changed but the tests started failing: the trigger was external — a new release appearing on the update server. Same test runners, same browser engine, same code. Only the outside world changed.

We confirmed this exactly from the test machine's own log: it printed Update available 1.0.2-rc.1 → 1.0.2, then Showing update prompt, and right after, asset not found: settings.html — the robot lost the Settings window.

The fixes, one at a time

THE FIX · PR-A

WS1 — Stop test/dev builds from checking for updates

A throwaway test build has no business phoning the real update server or popping a window. We switch the update check off for two kinds of builds: the ones used by the robot tests, and the ones developers run on their own machines while coding.

Why this also un-jams everything: the robot test build is exactly one of the builds we're switching off. So the moment this fix runs, the pop-up can't appear, the test passes, and the whole pipeline goes green again — no special admin override needed.

There are two off-switches kept as safety valves: an emergency "never update" flag, and an opt-in "yes, actually check" flag for the rare developer who wants to test updates locally. The normal shipped app is untouched — it still auto-updates exactly as before.

SAFETY NET · PR-B

WS2 — Make the robot re-find the Settings window itself

Even with pop-ups gone, we teach the test robot to explicitly look for the Settings window before checking it — instead of assuming it's already looking at the right one. And if Settings genuinely isn't there, the test fails loudly rather than papering over it.

This is insurance: any future stray window can't silently break the tests the same way.

SUPPLY CHAIN · PR-B

WS3 — Pin the headless server's ingredients list

The standalone "headless server" we ship for other teams' CI didn't have a locked-down ingredients list (a Cargo.lock). That means its exact dependency versions were decided fresh at build time instead of from a reviewed, committed file. We commit that list and build strictly against it.

Reviewed + reproducible builds = a smaller window for a supply-chain surprise to sneak in.

GUARD RAIL · PR-B

WS5 — Make the "what's inside the app bundle" check stricter

Remember the 1.0.1 disaster? An extra binary sneaked into the macOS app bundle and broke auto-update. We already added a guard that checks the bundle's contents — but it only looked at regular files and could miss, say, a sneaky shortcut or folder. We tighten it to check everything at that level, exactly.

This guard is also why we're comfortable not building a giant automated update test right now — it already catches that whole class of bug for free (see below).

DOCUMENT NOW, AUTOMATE LATER

WS6 — How we'll test the actual update path

The dream is an automated test that installs the old version, lets it update, and confirms it still launches. It turns out that's genuinely hard to automate safely: the updater cryptographically checks every download against a built-in key, so a fake test-server download would either need the real production signing key (too risky for CI) or a fake key (then you're not testing the real thing).

So for now we write a short manual checklist to run before each release, and leave the automated version as its own future project. The stricter bundle check (WS5) already guards the specific failure that bit us in 1.0.1.

DROPPED

WS4 — A fix we discovered we didn't need

We thought the headless server might report the wrong version number. On closer inspection (and a second opinion that corrected an earlier one), the version it reports actually comes from a shared component we already stamp correctly at release time. So there was no real bug. We dropped it, keeping only an optional tiny nicety if we ever want a --version command.

Two things we deliberately are NOT doing

  • No release. We're landing fixes onto the main branch and cutting a version another day. No rush.
  • No big architectural refactor yet. The headless server still quietly drags the desktop UI framework along for the ride. Cleaning that up ("a shared core library both can use") is real work and gets its own plan later — not bundled into this firefight.

How we checked our own thinking

Per our usual process, three independent plans were drafted — ours, one from a different AI model family (codex), and one from a separate reviewer agent — then merged. They caught real things we'd have gotten wrong:

  • Our first instinct (switch off updates only for robot builds) would have left an unused chunk of code that trips the linter. Corrected.
  • We almost used a too-strict build flag (--frozen) that fails on cold caches; switched to the right one (--locked).
  • The "wrong version number" fix (WS4) was based on a misunderstanding; dropped.

Final verdict from the cross-check: "ship-to-approval: the plan is sound."

Open questions for the human

1. The dropped fix (WS4) — really drop it, or add a small accelerator-server --version command?

Context: nothing is broken either way. A --version command is a minor convenience for people running the headless server. Low value. The recommendation is to drop it.

2. Do we trust the "self-unblocking" claim enough to leave branch protection untouched?

Context: we believe the fix makes its own pull-request pass the checks normally, so we never have to weaken the merge rules. If that prediction is somehow wrong, the fallback is a one-time manual override on that single PR — never a standing change. Recommendation: proceed as planned, keep the rules intact.

3. Where should the manual update-testing checklist live, and should the release workflow link to it?

Context: a small doc-placement choice. Leaning: a UPDATER_TESTING.md next to the app, with a one-line reminder in the release pipeline.

Sequence: PR-A (the fix, un-jams the pipeline) → PR-B (the three safety/hardening changes) → the checklist doc. Nothing ships to users until a future, deliberate release.
