- Medium — [authorization.rs:129](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/authorization.rs:129), [authorization.rs:470](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/authorization.rs:470). The “strict newtype `Deserialize`” split is not implemented. `Deserialize` delegates to `TryFrom<String>`, and `TryFrom<String>` canonicalizes any canonicalizable string, so `"HTTPS://NULO.SH:443"` deserializes successfully instead of being rejected. That still preserves the storage invariant, but it defeats the intended strict-vs-lenient boundary: future direct serde use of `CanonicalOrigin` will silently normalize input instead of failing. Concrete fix: make `Deserialize` accept only already-canonical strings (`canonicalize_origin(&s)` must return `Some(canon)` with `canon == s`), keep `de_approved_origins` as the only lenient canonicalizing/drop-invalid path, and flip the test to assert that non-canonical-but-fixable strings fail direct `CanonicalOrigin` deserialization.

- Low — [commands.rs:115](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/commands.rs:115), [authorization.rs:159](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/core/src/authorization.rs:159), [windows.rs:107](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/src/windows.rs:107), [authorize.html:33](/Users/alejoamiras/Projects/aztec-accelerator/packages/accelerator/src-tauri/frontend/authorize.html:33). A malformed `respond_auth` payload cannot be upgraded to `Allow`, which is good, but it also does not actually “deny immediately” the real pending request. In the `None` branch it calls `auth.resolve(&origin, Deny)` on the raw user-provided string; if that string was tampered in the popup, it will not match the real pending key, so the request waits for the 60s timeout cleanup. This is not an auth bypass, but it misses the decided behavior and shows the pending-key invariant is still only conventional because `AuthorizationManager.pending` remains `HashMap<String, ...>`. Concrete fix: stop routing popup decisions by the user-editable origin string. Use a server-issued opaque request id (or, second-best, type `pending`/`request`/`resolve` to `CanonicalOrigin` and make popup responses carry a canonical key produced by the server).
