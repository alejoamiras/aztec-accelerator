Quality Fixes — aztec-accelerator

Blueprint deep · 9 audit findings · 4 PRs · plain-language companion to plan.md

We're fixing the 9 maintainability findings from the recent quality audit — making the code cheaper to change later, and adding tests that prove each change is safe. None of these are bugs users hit today; it's about future-change cost. Behavior stays identical everywhere except one finding (F-02), which also closes a small real gap in how website addresses get approved.

The shape of the work: 4 pull requests, grouped so each hits one CI gate cleanly. Three are pure tidy-ups proven safe by the compiler + the existing test suite + the real browser end-to-end tests. One (F-02) is a genuine improvement to a security-adjacent boundary, so it gets new tests and a security review afterward.

Why "deep" (the heaviest planning tier)

We scored the risk rubric: blast radius = HIGH (this touches the server's core state, the origin-approval trust boundary, release-critical download/cert code, and the published SDK package) and security sensitivity = MED-HIGH (F-02 is literally the "who's allowed to ask for a proof" check). Two high dimensions → deep. Novelty is low (we just refactored this codebase), so not the even-heavier mega-deep. Three independent plans were drafted, cross-checked, and stress-tested by two adversarial audits.

What each PR does (plain language)

PR-1 — Typed invariants F-01 + F-02 · the important one

F-01: the server's shared settings are an "everything's optional, fill-in-by-hand" form copied in 3 places. We add proper constructors so a new setting is one edit the compiler checks — no more silently-empty fields.

F-02: an approved website address is stored as plain text, and the rule "it must be tidied into one standard spelling first" lives only in comments — one entry point skips it. We make a real CanonicalOrigin type whose only way to exist is already tidied, so the rule can't be bypassed anywhere. This one changes behavior slightly: a malformed address in the CI server's config now fails loudly at startup instead of silently never-matching.

PR-2 — Break up two giant files F-03 + F-04 · pure reorganization

F-03: the desktop app's startup is one 203-line do-everything block. We split it into named steps (build tray, wire callbacks, start servers…) so the launch reads like a checklist. F-04: the "everything about the prover binary" file (download, cache, platform, install…) splits into focused modules. Both are pure moves — the public names stay the same, the compiler + existing tests prove nothing broke.

PR-3 — Three small tidy-ups F-07 + F-08 + F-09

F-07: bundle the 3 cert file-paths (that always travel together, and could be swapped by accident) into one object. F-08: give one function clear ownership of the proving-status messages instead of two functions sharing a hidden handshake. F-09: de-duplicate the copy-pasted "start HTTPS" code.

PR-4 — SDK docs + transport F-05 + F-06 · the published package

F-05: the SDK's README currently documents the old, retired status shape (teaching the exact mistake a recent change fixed). We fix the docs, export a type that was promised-but-missing, and add a test so docs can't silently drift again. F-06: the SDK talks to one server using two different HTTP libraries — we unify them into one internal transport. No change to the public package surface (the compiler enforces that).

Human context

This codebase just went through a deep quality refactor and a clean "core extraction" (splitting the GUI-free engine into its own crate). It's healthy — this work is polish, not rescue. The one part that deserves care is F-02, because it touches the security boundary that decides which web pages may request proofs; that's why it gets new tests and a /harden security pass afterward. We deliberately skipped writing "characterization" (golden-master) tests — the existing 90 Rust + 96 TS unit tests, the 9 real-browser end-to-end tests, and Rust's compiler already catch refactor breakage; we add tests only where there's genuinely new code or new behavior.

  • (Open) Resave removal (F-02): deleting an old migration step also stops a one-time rewrite of the config file. We back this with a round-trip test proving existing configs still load correctly. Rec: safe (the file is app-private) — confirm.
