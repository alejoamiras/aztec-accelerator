Aztec 5.0.0 stable bump + SDK release

Aztec shipped its final 5.0.0 release this morning, and the test network is already running it. This plan updates our code to 5.0.0, fixes the handful of things the update genuinely breaks, re-deploys the fee-sponsor contract whose address moved (again), publishes our SDK to npm, and — for the first time — makes it the default version people get when they install our package. The desktop accelerator app does not need a new release: it downloads the proving engine at runtime, and we re-verified its code hasn't changed since the version people already have.

Why this tier was chosen

mid — third time running this exact kind of bump (low novelty), but it couples to lots of things outside our control (npm packages hours old, a live test network, a contract deployment) and involves an irreversible npm publish. Rubric: 1 HIGH (external coupling), several MODERATEs. The dual audit earned its keep: the first codex pass rejected the draft and both auditors caught real problems (see "What the audits changed").

Phases

Phase 1: Update the packages + fix what breaks

Run the bump tool (it updates all 24 version pins, plus two safety values automatically). Fix the one real code break: account creation now requires a signing key — 5 places in our code pass only 2 of 3 arguments (the draft said 3 places; both auditors caught the 2 missed ones in our deployment scripts). Switch the playground's wallet to in-memory storage — the new version changed how browsers store wallet data in a way that breaks our error recovery and locks out second tabs; since the playground creates throwaway accounts anyway, keeping nothing on disk sidesteps all of it. Also: wire up a type-checker for the playground's source code, which — surprise — nothing was actually checking. Finally, refresh the dependency lockfile using a one-time local bypass of our 7-day new-package quarantine (same documented exception as the last two bumps), then verify nothing unexpected snuck into the lockfile.

Proves itself by: install + frozen-install both clean; every pin exactly 5.0.0; lint, type-checks (now including the playground), unit tests, playground build, and the 28 UI mock tests all green.

Phase 2: Publishing safety rails + land the change

Add a tiny new workflow whose only job is pointing npm's "default version" label at an already-published version (it never publishes anything — that separation is what lets us promote only after real-world testing). Harden the publish workflow: don't let a queued run cancel a half-finished publish, run the full native proving test at publish time (previously it only tested the slow fallback path), and close a back-door input on the reusable publish workflow that could have flipped the npm default with no testing at all (the final audit's catch). Then open the PR and let CI prove the important thing: a real transaction proves via the native engine on a 5.0.0 network ("transmit", not "fallback"), plus a local test that downloads and integrity-checks the 5.0.0 proving engine exactly the way the deployed app will.

Proves itself by: workflow linting green; all four CI pipelines green including the native-proving e2e; the download+checksum test passes; PR auto-merges.

Phase 3: Deploy the fee contract → test → publish → test → promote

First, re-deploy the fee-sponsor contract on the test network (its compiled code changed, so its address moved off the funded one) — with careful pre-checks so a network hiccup can't trick us into a wrong deployment, using a throwaway wallet that only needs a little Sepolia ETH for gas. Then a pre-publish smoke test in a real browser against the live network — deploy an account and send a token transfer (this exercises the one behavioral change no automated test can see: how recipients discover private messages). Only if that passes: publish the SDK to npm's testnet tag and deploy the playground. Then the acceptance smoke: the already-released desktop app (which will genuinely download the new proving engine at runtime — the one path nothing else tests end-to-end), plus a Safari pass. Only after all of that: flip npm's default to 5.0.0, on the day you choose (open question 3).

Proves itself by: fee contract deployed + funded on the live network; pre-publish transfer smoke green BEFORE publishing; npm tags correct; live site on 5.0.0; released-app + Safari + WASM smokes green.

Human context

Why no desktop app release? The installed app doesn't bundle the proving engine — it downloads whichever version a website asks for, verifying its checksum. We re-checked that the app's entire core hasn't changed a byte since the released version, and Phase 2's tests prove the released behavior works with 5.0.0. If those tests failed, the plan stops and escalates to a full app release.

Why does a contract need re-deploying? On Aztec, a contract's address is derived from its compiled code. Aztec recompiled the fee-sponsor contract for 5.0.0, so "the same contract" now lives at a new address, which starts out empty and unfunded. Same thing happened on the previous bump. Anyone can drain a public fee sponsor by design — that's what it's for — so we fund it modestly and consciously.

Why the quarantine bypass? We normally refuse to install npm packages younger than 7 days (supply-chain protection). 5.0.0 is hours old. The bypass is a single local command, never committed, never in CI — and we hand-audit the lockfile diff afterward, checking the publish date of anything unexpected. This is the third time using this documented exception.

What's the "latest" flip about? Until now our npm package's default install pointed at an old 4.x version (which can't even talk to the current network). Promoting 5.0.0 to the default is the highest-visibility pointer we own, so the plan makes it a separate, test-gated, deliberately-timed step — with an honest note that if 5.0.0 turned out bad, there's no good version to roll back to; recovery would be publishing a fix forward.

What the audits changed

Codex round 1 rejected the draft; the independent Fable audit said conditional approve with 5 conditions. Every finding was re-verified against the code and folded (or explicitly rejected with reasons in the decision ledger): 5 breakage sites not 3; the playground's source code was type-checked by nothing; the wallet-storage change silently broke our error recovery (fixed by going in-memory); publish and promote had to be separate steps; the publish workflow could cancel itself mid-publish; the pre-publish transfer smoke was missing; the rollback story was wishful thinking. A fresh final codex session then attacked the consolidated plan and caught four more (a back-door workflow path that could flip the npm default ungated; the published package itself never being install-tested; a funding "proof" that didn't actually exist; an internal contradiction in the new promote workflow) — all folded, and on resume it gave conditional approve with its last condition (the back-door's second input) also folded. One question the auditors disagreed on is left for you (open question 3).

Audit verdicts

  • Codex round 1 (fresh session, xhigh): reject → all 7 blockers folded or scope-rejected with reasons.
  • Fable (independent context): conditional approve → all 5 conditions folded.
  • Final fresh-context codex pass: reject → 4 findings folded → resumed: conditional approve → final condition folded. Verdict on record: conditional approve, all conditions folded.
