Hardening the bb downloader + version checks

When you ask the accelerator to prove for an Aztec version it doesn't have, it downloads the matching bb prover binary at runtime. Our full-codebase audit found three small-but-real bugs in that path, plus two tidy-ups. None are exploitable by a random website (they're gated behind your "allow this site" approval), but they're the kind of sharp edge worth filing down. This fixes them.

Phases

Phase 1 — Don't let a lying server blow up memory

The downloader trusted the server's claimed size, then loaded the whole file into memory at once. A malicious/broken server that doesn't state a size could stream gigabytes and crash the app (out of memory). Fix: read the download in small chunks, counting as we go, and stop the moment it passes a sane ceiling (64 MB — the downloaded platform tarball is ~17 MB; the cap was bumped from 32 MB post-implementation when code-review measured the real size).

Phase 2 — Block the ".." path trick

A version name is used to build a folder path. The check that validates version names accidentally allowed .. ("go up a folder"), which in the worst case could point a "delete this folder" step at the wrong directory. It's currently unreachable (another check blocks it first), but we close it properly — reject .. at the front door and right before the risky delete.

Phase 3 — Don't crash on weird error text

When bb fails, we log the first 500 bytes of its error. If a character (like an emoji) straddles byte 500, that crashes. Fix: cut at 500 characters instead — never mid-character.

Phase 4 — Two small cleanups

Replace two hand-written "bytes → hex" loops with the standard one-liner, and simplify the cache-eviction code's loop (keeping its tricky-but-correct "never evict the bundled version" rule intact — the audit caught that this part is load-bearing, not redundant).

Human context

This came out of a whole-codebase audit (two reviewers) after the bigger security work. The accelerator is otherwise high-quality; these are the genuinely new findings. The plan itself was then reviewed by two more reviewers (codex + opus), who caught that an earlier draft of two fixes wouldn't have compiled / would have broken a test — those are now corrected.

Open questions

  • Resolved post-impl. The one judgement call — a 32 MB vs 64 MB download ceiling. You originally chose 32 MB (tighter), but that rested on a wrong "bb is ~5 MB" assumption; code-review measured the real downloaded tarball at ~17 MB (avm-class ~30 MB), so 32 MB had too little headroom (a future bump would silently fall back to slow WASM). Bumped to 64 MB — matching the sibling copy-bb.ts limit for the same artifact.
