A; holds-with-changes.

- High: the rc gate is not a “dry-run”; it is a real prerelease publication path. On prerelease versions, `latest.json`/S3/live-feed/`bump-source` are skipped (`.github/workflows/release-accelerator.yml:637-641`, `784-857`), but the workflow still pushes a git tag (`539-560`) and creates a GitHub Release with uploaded assets (`686-781`). That means `gh workflow run release-accelerator.yml --ref <branch> -f version=1.0.5-rc.N` is the right mechanism to exercise `build-headless` and the updater smokes, but only if everyone treats it as a real rc publish, not a harmless validation run.

- High: reusing an `rc.N` is unsafe. The tag step skips creation if the tag already exists (`release-accelerator.yml:554-559`), while the release step deletes and recreates the Release for that tag (`705-709`). So a second run on a different commit can attach fresh artifacts to an old tag target. If Part B uses rc-gating, the assumption must be “fresh rc number every run,” not “rerunnable dry-run.”

- Medium: Approach A is the right e2e move. `_e2e.yml` already gets Bun, bun-cache, and `bun install --frozen-lockfile` from `setup-aztec` (`.github/actions/setup-aztec/action.yml:13-24`). Routing `_e2e` through `setup-accelerator` would layer a second overlapping bootstrap on top of that, plus another rust/cache setup, inside a reusable workflow that already has Aztec-specific ordering. The current e2e accelerator leg only needs one surgical change: trim the inline apt list at `.github/workflows/_e2e.yml:45-53` and keep the prebuild plus `BB_BINARY_PATH`/`AZTEC_BB_VERSION` export (`51-65`). B is DRY in theory, but in this repo it is not a pure dedupe; it is a broader refactor.

- Medium: the `src-tauri` version patch in `build-headless` is dead for this job. `build-headless` builds from `packages/accelerator/server` (`release-accelerator.yml:281-291`), and that crate depends only on `accelerator-core`, not `src-tauri` (`packages/accelerator/server/Cargo.toml:16-24`). `/health.version` is injected from `server` via `app_version: Some(env!("CARGO_PKG_VERSION"))` in `packages/accelerator/server/src/main.rs:62-72`, and the shared `/health` handler uses that injected value first (`packages/accelerator/core/src/server.rs:167`). `accelerator-server --version` also prints the `server` crate version directly (`server/src/main.rs:21-29`). So removing the `sed` against `packages/accelerator/src-tauri/Cargo.toml` at `release-accelerator.yml:277` does not break `/health.version` or `--version`. What does need cleanup is the stale build comment at `287-289`: after removing the `src-tauri` patch, `--locked` is still incompatible because you still patch `server/Cargo.toml` (`278`), not because of a `src-tauri` path-dep.

- Medium: skipping the host-assert with `run-prebuild: false` is safe for the current checked-in matrix, but the plan overstates why. The assert in `setup-accelerator` is explicitly there to protect host-selected bb copying (`.github/actions/setup-accelerator/action.yml:55-77`), and `build-headless`’s matrix currently pairs each target with a matching host (`release-accelerator.yml:247-259`). So disabling it for this job is fine today. But “`cargo build --target` would fail on a genuine mismatch anyway” is not a robust invariant, especially on macOS. After this change, safety comes from matrix discipline, not from a guard in the composite.

- Medium: dropping prebuild from `build-headless` does not remove anything the shipped tarball currently contains. Prebuild copies `bb` into `src-tauri/binaries` and writes `src-tauri/AZTEC_VERSION` (`packages/accelerator/scripts/copy-bb.ts:167-195`); the headless tarball contains only `accelerator-server` plus its checksum (`release-accelerator.yml:304-321`, `603-605`). The headless server reads bb version from runtime `AZTEC_BB_VERSION`, not from `src-tauri` build artifacts (`server/src/main.rs:67-71`). So no, removing prebuild does not break the tarball’s `--version` stamp or `/health.version`. The caveat is narrower: the rc gate proves `accelerator-server --version` before packaging (`293-302`), but it does not prove `/health` behavior from an extracted tarball, and it never did.

- Security/assumptions: the code-change blast radius is genuinely smaller because you are removing packages and skipping prebuild, but the validation blast radius is not. RC validation still uses real signing/notary credentials in the desktop `build` job and real `contents: write` publication authority in `tag`/`release` (`release-accelerator.yml:544-569`). That is acceptable, but the plan’s Security/Assumptions sections should say “real prerelease publish, no S3/bump-source” rather than implying a non-publishing dry-run.
