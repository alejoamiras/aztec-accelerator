Headless Slim Follow-ups — finishing the job

Two leftover spots from the core-extraction arc still install desktop GUI libraries (and run a desktop pre-step) on CI jobs that only build the headless server. This trims them. It's split into two pull requests because they prove themselves very differently: one is checked by the normal PR gate; the other touches the release pipeline, which can only be proven by cutting a release-candidate.

Why this tier was chosen (/blueprint mid)

Small CI edits, but one of them lives in the release workflow (blast radius: HIGH — the tarball ships to other teams' CI). That one high-risk dimension earns the dual audit. Both codex and an opus architect reviewed it → both picked the same approach and verified the trickiest claim (that a leftover version-patch is truly dead code) by checking the lockfile directly.

Phases

Part A — the e2e job (one-line, merges normally)

The SDK end-to-end test job installs the desktop GUI libraries but only compiles the headless server. Drop the GUI libraries (keep the one the HTTP client needs, and keep the step that fetches the real bb the test proves against). One line. Its own PR gate proves it.

Part B — the release build (rc-gated, separate PR)

The job that builds the 4-platform release headless tarballs does the same desktop busywork. Flip the two switches we already added in Phase 3b to skip it. While we're in there: a leftover step still patches a file (src-tauri) this job doesn't even compile anymore — both auditors confirmed it's dead — so we remove it and fix two now-wrong comments.

Part B validation — you cut a release-candidate

The release build can't be checked by a PR. You dispatch a release-candidate (rc.N) on the branch; it builds the slimmed tarballs and runs the real auto-update tests against them. Merge once it's green.

Human context — the rc is a real publish, not a sandbox

Important: a release-candidate run is not a harmless dry-run. It pushes a public accelerator-v1.0.5-rc.N tag and creates a GitHub pre-release with the assets. It does not update the production auto-update feed (that's skipped for pre-releases), so your real 1.0.x users are untouched — but it is a real publish. Two rules: treat it as such, and use a fresh rc.N every run (reusing a number re-points an old tag at new files).
