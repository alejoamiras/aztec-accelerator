Aztec 5.0.1 security patch + standards-token migration

Aztec shipped 5.0.1 yesterday — a security patch that closes several ways a hostile sender could break note delivery for anyone on 5.0.0, which is what our live playground and our npm users run today. This plan updates everything to 5.0.1, swaps the playground's demo token to the newly-official @aztec-foundation/aztec-standards package (the successor to your own package), re-deploys the fee-sponsor contract (its address moved again), publishes SDK 5.0.1, and finally marks your old npm package as deprecated with a signpost to the new one. Everything runs on the machinery the 5.0.0 release built and proved.

Why this tier was chosen

mid — fourth time on these rails (low novelty), but it leans on lots of outside state: two freshly-published npm packages (one from a scope that was hours old at planning), a live test network, and a contract deployment. Rubric: 1 HIGH (external coupling), the rest MODERATE. The dual audit again earned its keep: codex initially rejected the draft, the independent Fable audit dug up facts nobody had (including reading the new token's actual source code to verify a parameter the docs don't document).

Phases

Phase 1: Update packages + swap the demo token + teach the bump tool

Bump the 24 Aztec pins to 5.0.1. Add the new standards package (recording its cryptographic build-attestation as evidence at install time) and rewrite the token demo to use it — as its own commit, so it can be surgically undone if it misbehaves (that's the safety valve below). Teach the version-bump tool about the new package so future bumps can't silently strand it. Proves itself by: install + frozen-install clean; every pin exactly 5.0.1; lint, all typechecks, unit tests (incl. the extended bump tool's), UI mock tests, and the production-build smoke all green.

Phase 2: Land the change through CI

Open the PR (bump and token-swap kept as separate commits) and let CI prove the important thing: a real transaction proves natively against a 5.0.1 network, plus the local test that downloads and integrity-checks the 5.0.1 proving engine exactly the way the released desktop app will.

Proves itself by: all four CI pipelines green including the native-proving e2e; the download+checksum test passes; PR auto-merges.

Phase 3: Fee contract → test → publish → test on the real site → promote

Re-deploy and fund the fee-sponsor contract (1000 test tokens, same wallet). Then a browser test against the live network before publishing anything — including the new token demo end-to-end. Safety valve: if the new token misbehaves there and isn't a quick fix, we revert just the token commit and ship the security patch on schedule — the demo swap re-lands separately later. After publishing and deploying: fresh-install test of the exact npm artifact, the released desktop app genuinely downloading the new proving engine, a Safari check (you), and — new this cycle — the full token demo run once more on the actual deployed website before npm's default version flips to 5.0.1.

Proves itself by: fee contract live + funded (balance recorded); pre-publish demo green; publish + deploy green; all acceptance checks incl. the on-site token run; npm testnet = latest = 5.0.1.

Phase 4: Deprecate your old package

Mark every version of @alejoamiras/aztec-standards as deprecated with a one-line pointer to the new official package. Runs under your personal npm login (never the CI token), expects your 2FA prompt, and verifies every single version shows the notice. Fully reversible.

Proves itself by: every published version returns the deprecation message; the new package still installs cleanly.

Human context

Why move fast on a patch? Unlike the last two cycles, the test network hasn't changed — nothing is drifting. But 5.0.1 fixes bugs where a malicious counterparty can permanently freeze someone's note syncing or send them a note they can never decrypt. Those affect people using our live playground and SDK right now, which is why the release notes say the update is "required" for apps even though node operators can wait.

How much do we trust the new package? We verified what's verifiable: it's published from the official AztecProtocol GitHub organization with a cryptographic attestation tying the artifact to a specific commit and build workflow, its code descends from the long-standing Wonderland implementation, and it contains zero third-party dependencies. What we deliberately do NOT claim: that it's been audited (no evidence exists), or that the attestation proves anyone reviewed the code. The audits made us state this honestly and record the evidence at install time. One auditor even read the token's source to confirm the "no authorization contract" mode we use is genuinely supported — the shipped docs don't say.

Why the safety valve? The one real disagreement between the auditors: codex wanted the security patch and the token swap shipped as two separate releases so a demo problem could never delay a security fix; Fable showed a cheaper way to get the same protection — keep one release, but make the token swap trivially revertible and pre-authorize the fallback. The plan adopts that hybrid.
