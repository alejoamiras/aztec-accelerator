A test that asks: "if we ship this, can people actually update to it?"

Plain-language companion to plan.md · 2026-05-29

1.0.1 was a disaster because a freshly-built app installed fine but broke when an older version auto-updated into it. Our tests only ever checked fresh installs. This plan adds the missing test: actually perform the update, before we publish.

Why 1.0.1 slipped through

Think of it like a door. We tested "does a brand-new door open?" — yes. We never tested "if you swap this door into an existing frame while someone's standing there, does it still open?" — and that's the one that jammed.

The auto-updater swaps the new app in on top of the running old one, then relaunches. In 1.0.1 that swap changed the app's internal shape just enough that macOS's security check (amfid) hung forever on relaunch. A fresh install never does that swap, so our fresh-install test stayed green while real users got a frozen app.

The new test, in plain terms

During a release, after we've built and signed the new version (call it N), and before we publish it, a new CI job does exactly what a real user would:

1. Download the previous released version (N-1) and install it for real (into /Applications, the writable spot — not a read-only disk image, or the swap couldn't happen and we'd be fooling ourselves).
2. Quietly point that N-1 app at a pretend update server running on the CI machine, which offers our just-built N.
3. Let N-1 auto-update to N, swap, and relaunch — untouched, exactly as a user's would.
4. Check the app came back to life and reports the new version (/health says "I'm N"). If it hangs (the 1.0.1 symptom) → the release is blocked.

Two clever bits worth knowing

We need no secret keys

The updater only trusts downloads signed with our private key. Our build already signs N with it. So the test just serves the already-signed N — the old app checks the signature against the public key baked inside it, and it matches. The test job never touches the secret signing key. Cleanest possible security posture.

How we fake the update server without changing the app

The app's update address is hard-wired to aztec-accelerator.dev. On the CI machine we (a) make that name resolve to "this machine," and (b) run a little local server there with a trusted certificate for that name. The app can't tell the difference — so we test the exact binary we're about to ship, with zero test-only code added to it.

We confirmed this works by checking how the app validates certificates: it uses the operating system's trust store, so a certificate we install on the CI machine is honored. (The alternative — shipping special "test mode" code in the real app — was rejected.)

Rollout: careful on purpose

  • macOS blocks releases from day one — both Apple Silicon and Intel (we ship both; testing only one would be a blind spot). This is where the real risk lives.
  • Linux starts as advisory (runs, but doesn't block) until we've seen it pass cleanly on a pre-release once. Linux's AppImage self-update is the fiddliest part to run on CI machines, and we don't want an untested setup blocking real releases. Promote it to blocking once proven.

The part that makes it trustworthy

A test that always passes is worthless. So there's a permanent negative check: we feed it a deliberately-corrupted download and confirm it rejects it. And "success" only counts if the updated app reports the new version — not merely "I'm alive" (the old version is also alive). That way the gate can't silently rot into always-green.

Open questions for the human

1. On macOS CI, will a freshly-downloaded notarized N-1 launch headlessly without a Gatekeeper prompt? (It's notarized, so it should — we'll confirm on the first dry-run.)

2. Linux AppImage: will it run natively on the CI runner, or need an extract-and-run workaround? If the latter, that changes the very update path we're testing — so we validate it early and keep Linux advisory until it's clean.

Reviewed by codex + an independent reviewer agent. Verdict: design sound; ship with macOS blocking, Linux advisory until proven. We'll prove it on a 1.0.3-rc dry-run before it ever gates a real release.
