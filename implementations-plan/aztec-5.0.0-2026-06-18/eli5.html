Aztec 5.0.0-rc.1 bump + release

We're moving the whole project from Aztec 4.3.1 to 5.0.0-rc.1. Aztec 5.0 is a hard fork — the cryptography and on-chain rules changed, so a 4.x app and a 5.x network can't talk. The key realization (yours): we probably do not need to re-release the desktop app. It already downloads whatever prover (bb) version the SDK asks for, at runtime, with its own integrity check. So if the prover's command-line interface didn't change, the already-installed app just fetches and runs the 5.0 prover by itself. That turns this from "migrate + rehearse + ship a desktop release + auto-update everyone" into "migrate + publish the SDK to npm" — far smaller and far less risky.

Why this tier was chosen

Planned at deep (hard fork = 5 of 6 risk dimensions HIGH) — three independent plans, two adversarial audits, a fresh-context final pass. The audits earned their keep: they caught that the test "proving" the prover works actually asserted nothing, and that the SDK would've shipped without real evidence. After your steer the execution shrinks to roughly a mid-sized job (SDK-only), but the migration + parity rigor from the deep audit still stands behind it.

Phases (SDK-only path)

P0 — Recon (no code)

Read the real 5.0 type definitions; write down exactly what each thing we import changed. Also check the live v5 testnet for a ready-to-use, already-funded fee-payer contract (so we don't deploy our own).

Proves itself by: a complete signature-diff table + the fee-payer answer.

P1 — Bump the versions

Point every @aztec/* at exactly 5.0.0-rc.1. Our 7-day supply-chain gate would block this, so we use a one-command, local-only, documented override.

Proves itself by: no 4.3.1 left, clean frozen install, lint green.

P2 — Fix the SDK

Our published library extends Aztec's prover. Fix whatever 5.0 changed. (A scary-looking "the app rejects the version string" risk turned out to be false — verified against the actual code + its passing test.)

Proves itself by: SDK type-check + unit tests + build green.

P3 — Fix the playground

Handle "mined but reverted" correctly (a new 5.0 state), point at the v5 testnet, and use the network's default fee-payer (no deploy/funding needed).

Proves itself by: repo lint + tests + the 28 mocked UI tests green.

P4 — The decision gate: does the deployed app handle 5.0?

The crux. Run the existing test that drives the real native prover, now against a local 5.0 network. If the fast native prover mines a 5.0 transaction — and we prove it actually used the native path, not a silent fallback — then the deployed app handles 5.0 on its own and SDK-only is confirmed. If not, we fall back to rebuilding+releasing the app (the contingency appendix).

Proves itself by: a 5.0 tx mines through the verified native path.

P5 — Full local check + manual v5 smoke

Run everything (lint, tests, the 9 desktop UI tests), then you/we manually open the playground against the live v5 testnet and confirm a real deploy mines. That's the live-network acceptance — done by hand, not built as a new CI gate.

Proves itself by: every command green + the manual v5 deploy mines.

P6 — Merge the bump

Open the PR, CI green, auto-merge.

Proves itself by: all PR checks green.

P7 — Publish the SDK (you dispatch)

Publish the SDK to npm under the testnet tag (not latest, since the deps are a release-candidate). This is the release. Human-triggered.

Proves itself by: package on npm with provenance, testnet tag.

Human context

The "accelerator" is a desktop app that makes Aztec proving fast via a native binary instead of slow in-browser WASM. It already keeps a multi-version cache and downloads new prover versions on demand — which is exactly why an SDK-only release can work. The one thing that must hold: the prover's command-line + input/output interface didn't change between 4.3.1 and 5.0. We don't assume that — P4 proves it by actually running it.

Decisions made (2026-06-18)

  • Release shape: SDK-only; rebuild+release the desktop app only if P4 shows the prover interface broke (contingency). ✅

Implementation seeds

Two options for the autonomous implementation session. /goal survives claude --resume and stops when a condition holds; /loop fires on an interval until cleared (7-day expiry). The SDK publish (P7) is human-dispatched, so the seed drives P0–P6 and stops at the release boundary.

Use exactly ONE per session. They don't compose — setting one replaces the other.
