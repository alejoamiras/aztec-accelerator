What v1 got wrong

Bug 1: v1 said "publish SDK at 4.2.0". But 4.2.0 is already on npm, and the publish script appends a revision suffix — producing 4.2.0.1, which is not valid semver (we verified with node-semver). npm would have rejected the publish.

v2 fix: Fix the script so stable bases use 4.2.0-revision.1 (valid semver) instead. Small (~10 line) change folded into PR A.

Bug 2: v1 said "trigger a -rc.2 dry-run to validate the headless build matrix." But the current release workflow has no concept of a dry-run — it ALWAYS generates production latest.json, uploads to prod S3, and opens a bump-source PR. An -rc.2 would have auto-updated existing users to an rc release.

v2 fix: Restructure the workflow in PR A to detect prerelease versions (anything with a -) and skip the latest.json, S3 upload, and bump-source for them. The workflow also gets fixed so the tag job runs AFTER the build succeeds (previously: tag pushed even on failed builds).

Bug 3: v1's branch-cleanup script would delete branches whose PRs were just CLOSED (not merged). If you reused a branch name after a PR was abandoned, you'd lose the new commits.

v2 fix: Only delete branches whose tip SHA matches a MERGED PR's merge commit. Plus, the actual branch count is 11, not 129 (I miscounted local tracking refs).
