The reviews also caught a factual error in my first draft: I’d called a file “the CA private key.” It isn’t — the certificate authority key is never written to disk. The unprotected file is the localhost TLS key. Corrected, and the plan re-grounded on what’s actually at risk.

The three fixes

The Windows lock

F-003 · hardening

On Mac and Linux, the scratch folder your private transaction data briefly passes through is already locked to just you. On Windows that lock does nothing — the folder inherits whatever permissions it’s given. We add a real owner-only lock, applied at the moment the folder/file is created (so there’s no unlocked instant), and we cover the localhost TLS key and the settings file too.

Honest limit: this separates different Windows users. It does not wall off programs run by the same user — exactly like the Mac/Linux lock. If all your agents run as one Windows user, this is defense-in-depth, not a wall between them.

Proves itself by: a Windows-CI test that reads the folder’s actual permissions back and asserts owner-only, plus a check that a planted shortcut/junction can’t redirect the lock.

The “allow this site?” prompt

C9 · MED + 2×LOW

Three sharpenings of the approval popup a dApp triggers:

  • Click-steal (MED): today a second prompt can pop up under your cursor and catch a click you meant for the first. Fix: only one prompt is clickable at a time, and buttons ignore a click for a beat after the prompt appears — so a click can’t be stolen.
  • Look-alike extension IDs (LOW): reject browser-extension origins with weird hidden/look-alike characters, using each browser’s real ID format.
  • Show the real name (LOW): the popup now displays the site name the server actually recorded, not one handed in through the web address.

Proves itself by: Rust tests for the one-at-a-time queue + the click delay + rejected IDs; a two-site browser test in CI that a click can’t jump prompts.

The startup-launch undo

C8 · robustness

Switching on “launch at startup” does two things. If the second half fails, today the first half is left switched on — a half-on state. Fix: undo the first half if the second fails — and if the undo itself fails, say so loudly instead of pretending it worked. Care is taken not to treat an already-on state as a failure (that would wrongly switch a working setup back off).

Proves itself by: Rust tests that force each failure point and confirm the undo runs completely, in order, and reports honestly.
