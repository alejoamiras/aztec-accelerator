C6 — Trusting the bb prover binary the cache hands us

Cluster C6 · Finding F-007 · mid-tier blueprint · branch sechard/bb-cache-integrity

The accelerator runs a downloaded native binary called bb to build zero-knowledge proofs. That binary reads your private witness data. Today, two things download bb into a trusted cache folder and neither checks that the bytes are the real ones — and the runtime that later runs bb trusts the cache purely because the file exists. So a tampered or man-in-the-middled download becomes a trusted binary that runs over your secrets.

The core bug in one sentence. scripts/download-bb.ts does fetch → untar → chmod with zero digest verification, and Rust's find_bb returns a cached bb on exists() with no re-check — and if it ever did reject one, it would silently fall through to a different bb from ~/.bb or $PATH.

Why mid tier

Rubric: security-sensitivity HIGH (it's the binary that touches witnesses) and external-coupling HIGH (GitHub release downloads). Blast radius and irreversibility are moderate (per-user cache, re-downloadable). Two HIGH but not novel and not architectural → mid (plan + competing outline + codex + fable dual audit + final codex pass).

What we're building

The fix is the master plan's own F-007 recipe, applied to both download paths and the runtime:

1. Verify at download. Fetch the tarball, compare its SHA-256 against GitHub's published release-asset digest (the same fail-closed check the Rust side already does for its own path). No digest, wrong digest, or a network error ⇒ refuse.
2. Extract safely into a private staging folder. Accept only a single regular file named bb — reject symlinks, .. paths, extra members, and anything over a sane size cap. This is the "don't let a malicious archive write outside where you expect" rule.
3. Sign, then fingerprint, then publish atomically. On macOS we ad-hoc re-sign bb (which changes its bytes), so we compute the fingerprint after signing. We write a small bb.sha256.json marker (archive digest + final-binary digest) and then rename the finished folder into place in one atomic step — a crash never leaves a half-written trusted entry.
4. Re-check every time we run it. Before running a cached bb, re-hash it and compare to the marker. Missing/corrupt/legacy/mismatched marker ⇒ treat the cache as a miss, re-download the verified copy, and — crucially — never silently substitute a different bb for a specifically requested version.
Defense in depth, on purpose. We keep BOTH the download-time check (catches the network attacker at the point of attack) and the runtime marker re-check (catches an at-rest tampered cache). Both prior reviewers agreed a marker-only design was too weak — it would make bb:download silently "succeed" on an unusable cache and only fail much later at the first proof.

Phases & how each proves itself

Phase 1 — TypeScript downloader (scripts/download-bb.ts)

Add digest verification, private staging, tar-safety, the marker, and atomic publish; skip re-download only when an existing marker validates.

Proves itself by: root bun run test (new download-bb.test.ts: mismatch throws + nothing published; unsafe archive member rejected; marker written post-codesign; skip only on a valid marker) + bun run lint.

Phase 2 — Rust marker helpers + downloader

Add verify_cached_bb, marker read/write, stage-and-atomic-publish in downloader.rs; make listing use cheap marker-existence only.

Proves itself by: cargo test (marker round-trip; accept valid / reject missing+mismatch; marker computed post-finalize) + cargo clippy -D warnings + cargo fmt --check.

Phase 3 — Rust runtime: fail-closed, no wrong-version fallback

find_bb re-verifies the cache and hard-errors on a bad requested version (no sidecar/~/.bb/$PATH fallthrough); prove.rs decides "needs download" via verification, not exists().

Proves itself by: cargo test (end-to-end: legacy/tampered cache → exactly one verified re-download → correct path; download failure → no execution; requested bad version → Err, not a different binary) + clippy + fmt.

Phase 4 — CI wiring + docs

Make sure accelerator.yml runs both the root TS test and the core tests; confirm the pinned bb versions actually expose a GitHub digest; document the fail-closed/offline behavior.

Proves itself by: bun run lint:actions + full bun run test + full cargo test.

Human context you need

  • GitHub release-asset digests are a real feature (since June 2025): each asset exposes an immutable sha256:… generated at upload. We mirror the exact fetch the Rust side already uses.
  • macOS ad-hoc signing changes the binary, which is why the fingerprint must be taken last. Getting that order wrong would make every Mac cache-hit falsely fail.
  • The listing path feeds /health and the tray, which get polled — so listing must stay a cheap folder stat, never a "hash every cached binary" loop (that could read gigabytes on each poll).

Open questions (defaults chosen)

  • Legacy caches (downloaded before this change, no marker): default = fail-closed + re-download on first use. Offline machines with a legacy cache will need one online re-download.
  • Same-UID local attacker who can already write the cache can rewrite both binary and marker — this is out of scope for F-007 (it's a network-integrity finding; that attacker already has a higher privilege). Closing it needs a publisher signature or a folder ACL — noted as future work.
