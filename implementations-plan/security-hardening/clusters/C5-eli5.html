C5 · F-005 — Locking down who can deploy

Right now a single AWS role lets four different CI pipelines write anywhere in the website's storage bucket — including the file that tells every installed app "here's the latest version to auto-update to." If any one pipeline (or its token) is compromised, an attacker could overwrite that update file (a rollback attack) or deface the site. This plan splits that one key into three narrow keys — one per pipeline — so each can only touch its own corner, and only the release pipeline can touch the update file. It also tightens the rules on the main branch and fixes a lurking bug where a routine site deploy was deleting the update file. We only write and check the config here — you (a human) run the actual AWS + GitHub changes via a precise, reversible runbook.

Why "deep" tier

Rubric: 3 of 6 dimensions HIGH — security-sensitivity (IAM/OIDC deploy trust), irreversibility (production cloud roles + a live update feed), blast-radius (all deploy pipelines). Deep means three independent plans (me + Codex + Fable) plus a double audit. That paid off: the two auditors disagreed on a load-bearing AWS detail, which forced a safer choice (below).

What we'll change (phases)

Phase 1 — Split the one role into three (Terraform) + harden the main-branch rules

Three per-pipeline roles: landing (can't touch the update file — explicit deny + a fix so its sync stops deleting the feed), release-feed (can write only latest.json), playground. Each role only trusts its own workflow running on main. Main-branch rules add: no force-push, no branch deletion, linear history, conversation-resolution, required infra-validation check (kept at 0 approvals — GitHub won't let a solo owner approve their own PR).

Proves itself by: tofu fmt -check + tofu validate + the existing OpenTofu CI job + a ruleset-JSON assertion, all green.

Phase 2 — Point each pipeline at its own key + safety asserts

Each workflow uses its own role secret; the release workflow gets an early "can I even reach AWS?" check before it pushes a git tag (so a mis-wire can't leave a dangling release); the unused nightlies workflow is explicitly switched off (so it can't half-run and publish to npm).

Proves itself by: bun run lint:actions green + a grep confirming no live workflow still uses the old shared secret.

Phase 3 — Remove the old over-broad role (separate, later commit)

Deleted only AFTER the new roles are proven live — kept as its own post-smoke PR so it can't be applied too early.

Proves itself by: tofu validate green + the commit states "2 destroy, 0 add".

Phase 4/5 — Human-applies runbook

An exact, ordered, reversible script you run: apply roles → set 3 secrets → land the workflow cutover on main → smoke each pipeline (incl. a negative test that a stolen token is denied the wrong role, and AWS policy simulations) → delete the old role → apply the ruleset. Every wrong order fails closed (a visible red run, never a silent privilege grant).

Human context

This is a single-owner public repo. The realistic threat isn't a random internet attacker running code — it's blast-radius: a compromised CI token or an owner slip overwriting the update feed or defacing the sites. The in-app update verification from the previous cluster (C4/F-004) already rejects a forged or rolled-back feed on the client side; this cluster removes the ability to write that feed from the wrong place in the first place.

The one thing the auditors disagreed on

To stop pipeline A's token from assuming pipeline B's role (they all run on main), each role is bound to one specific workflow. Fable said the strong "workflow file path" claim (job_workflow_ref) works for these jobs; Codex said it's only emitted for reusable workflows and would make every login fail. Rather than gamble on a total deploy outage, the plan uses the workflow-name claim that both agree is supported and always present. Its only weakness — renaming a workflow to impersonate another — requires merging that rename to main, which is already the "malicious code on main" risk that no claim can stop. The stronger file-binding is offered as optional upgrade A7.
