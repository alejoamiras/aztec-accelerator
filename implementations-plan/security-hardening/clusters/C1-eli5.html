C1 — workflow-input-hardening (F-006)

The SDK publish workflow (_publish-sdk.yml) takes a text input, dist_tag, and pastes it straight into shell commands that hold the npm and GitHub tokens. Because GitHub substitutes the text before the shell runs, whoever can start that workflow could type a command instead of a tag and steal the tokens (or publish a fake package). The fix: check dist_tag against a strict allowlist and pass it as a quoted environment variable so it can only ever be data, never a command.

Why this tier was chosen

light. One file, one finding, a well-known fix. Only one risk dimension is high (it touches the token-bearing publish pipeline); everything else — novelty, reversibility, migration, external coupling — is low. So: one plan, one Codex audit, then implement.

Phase 1 — validate + quote dist_tag

Add an early step that rejects any dist_tag not matching ^[a-z0-9._-]+$ (real tags like testnet/nightlies/latest pass; spaces, ;, backticks, $, quotes don't). Then pass dist_tag into the publish and release-notes steps as a quoted env var ("$DIST_TAG") instead of raw ${{ … }} in the shell.

Proves itself by: bun run lint:actions (actionlint) green, no ${{ inputs.dist_tag }} left inside any run: block, and the validate step running before any token-bearing step — plus actionlint.yml green in CI.
