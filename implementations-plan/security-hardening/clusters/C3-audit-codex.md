**Verdict: conditional approve** — conditions: correct the Rust-action pinning method, use explicit tested tool versions, replace the actionlint installer safely, fix shellcheck without masking failures, and add an update/coverage process.

- **Remote `uses:` survey:** Complete. I found 121 mutable remote uses plus three already-pinned `create-github-app-token` uses, including dependencies inside all four composite actions. No remote reusable workflows, Docker actions, or expression-generated refs were missed. The 20 local actions and 15 local reusable-workflow calls beginning `./` must remain unpinned; local reusable workflows already resolve from the caller’s commit. [`actions/cache`, `/save`, and `/restore` can safely share one repository SHA.](https://docs.github.com/en/actions/how-tos/reuse-automations/reuse-workflows#calling-a-reusable-workflow)

- **SHA pinning:** Yes, include GitHub-owned actions; full SHA is GitHub’s only immutable action reference. However, the resolver is misstated. `^{}` is Git syntax, not a REST dereference. For annotated tags, fetch `git/ref/tags/<tag>`, inspect `object.type`, then follow `/git/tags/<object.sha>` until reaching a commit. Lightweight tags already return a commit. Record the exact associated release, e.g. `# v6.x.y`, not merely `# v6`. [GitHub tag-object API](https://docs.github.com/en/rest/git/tags), [secure-use guidance](https://docs.github.com/en/actions/reference/security/secure-use#using-third-party-actions).

- **Rust:** There is no `rust-toolchain.toml`. The three `rust-version = "1.77.2"` declarations are MSRV claims, not evidence that CI currently uses or passes on 1.77.2. Pin the action code to a SHA from `dtolnay/rust-toolchain`’s **master history**, then set an explicit `toolchain:`. Do not pin the SHA currently behind `@stable`: upstream warns those generated commits can be garbage-collected. Current stable is Rust 1.97.0; use it only after the Rust/matrix gates prove green, otherwise pin the last demonstrated-green version. Floating `stable` risks surprise clippy/rustfmt/compiler failures; exact pinning risks stale compiler security fixes and future dependencies requiring newer Rust. [Upstream SHA warning](https://github.com/dtolnay/rust-toolchain#choice-of-full-length-commit-sha), [Rust 1.97.0](https://blog.rust-lang.org/2026/07/09/Rust-1.97.0/).

- **Bun:** The repo declares no runtime Bun version. `@types/bun@1.3.11` is irrelevant. Local Bun and the current official release are 1.3.14, making that the defensible candidate after CI validation. Prefer adding `"packageManager": "bun@1.3.14"` centrally and letting `setup-bun` read it, rather than duplicating 22 inputs. [`setup-bun` supports this](https://github.com/oven-sh/setup-bun#usage). Phase 2 also misses two floating `node-version: 24` selectors; pin 24.18.0 or explicitly declare them out of scope.

- **actionlint:** Do not vendor a platform binary and do not continue executing the remote installer. Directly download an exact release archive, verify a hardcoded reviewed SHA-256 or GitHub artifact attestation, then extract. Remove the tiny binary cache, or verify cached bytes too. Current CI is additionally mislabeled: the `v1.7.11` installer script defaults to actionlint **1.7.10**, while the cache key says 1.7.11. Prefer current 1.7.12, already used by the local passing gate. v1.7.11+ publishes attestations. [actionlint release verification](https://github.com/rhysd/actionlint/releases/tag/v1.7.11).

- **Shellcheck:** The proposed `test && shellcheck || echo` form is blocking: a real shellcheck failure executes the `echo` and returns success. Use an actual `if`, or `find infra -type f -name '*.sh' -print0 | xargs -0r shellcheck`. Recursive `find` matches the existing `infra/**/*.sh` filter; `infra/*.sh` does not. There are currently no infra scripts, but there are three real scripts under `packages/**/scripts`; decide explicitly whether CI is infra-only or change both the filter and command to lint repository-wide scripts.

- **Maintenance:** There is no `.github/dependabot.yml`. SHA pinning without updates trades retagging risk for staleness, and GitHub notes that vulnerability alerts are not generated for SHA-pinned actions. Dependabot version updates can update SHA pins and same-line comments, but documented discovery covers workflows and a root action file—not these nested `.github/actions/*/action.yml` files. Require a weekly updater plus explicit nested-composite coverage, or a documented audited bump cadence. [Dependabot behavior and limitations](https://docs.github.com/en/actions/reference/security/secure-use#keeping-the-actions-in-your-workflows-secure-and-up-to-date).

- **Least privilege/residual supply chain:** Pinning is not fully orthogonal here. `publish-nightlies.yml` and `publish-testnet.yml` grant `contents: write` and `id-token: write` globally, exposing OIDC capability to checkout, paths-filter, setup-bun, and cache steps. Scope these per job or explicitly depend on C5. Also record the unverified Aztec `curl | bash` installer in `setup-aztec` as residual risk.

- **Gate correction:** The proposed grep only detects `@vN`; it misses `@stable`, `@main`, short SHAs, and other mutable refs. Parse every `uses:` and require either `./...`, `docker://...@sha256:...`, or `owner/repo[/path]@[0-9a-f]{40}`.
