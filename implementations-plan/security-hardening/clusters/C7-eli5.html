C7 — Trusting the Windows bb.exe we ship

Cluster C7 · Finding F-008 · mid-tier blueprint · branch sechard/bb-windows-provenance

On macOS/Linux the prover binary bb comes from an npm package (with a 7-day quarantine + a locked integrity hash). But Windows has no npm build, so we fetch bb.exe from a GitHub release and pin its SHA-256 in the repo as a supply-chain check. The problem: the way that pin gets created is circular, and the way version-bump PRs can merge is too eager.

The bug in one breath. update-aztec-version.ts auto-creates the Windows pin by downloading the file and recording whatever hash came back (that's not verification — it's "trust whatever arrived"). And _aztec-update.yml on auto_merge:false ran gh pr merge --squash — an immediate merge that lands the bump before CI even checks it.

What we found (and confirmed by hand)

  • No independent proof exists today. We checked: AztecProtocol publishes no build attestation (gh attestation verify → 404) and no signed checksum file. The current rc.2 pin is byte-for-byte "the file we downloaded." So a pin can only ever be a human-reviewed change-detector right now — honest labeling matters.
  • The auto-pin isn't even committed by the bump workflow (it only git adds package.json + lockfile) — so it's mostly a footgun for someone running the script locally. But it's the mechanism by which a circular pin would ever enter the tree, so it goes.

Why mid tier

Security-sensitivity HIGH (supply chain of a shipped binary) + external-coupling HIGH (GitHub release + npm). Contained to a few files. Both the codex and fable audits rejected the first draft and a third pass rejected the second — each time sharpening it, not reversing direction. That's the dual audit doing its job.

The fix

1. Stop auto-creating pins. Delete the download-and-write. A new Windows version simply has no pin until a human adds a reviewed one — and the Windows CI gate stays red until they do.
2. Make the pin's provenance machine-enforced, not a comment. Each pin becomes { sha256, provenance, note }. The resolver only accepts manual-review today and throws on anything it doesn't recognize. attestation is reserved for the future — when Aztec signs releases, we wire up gh attestation verify.
3. Leave the bump PR open. Replace the confusing auto_merge boolean with merge_mode: none|auto (default none): none creates the PR and stops — a human reviews before merge. No more immediate merge.
4. Delete the ungated path. The nightlies bump targeted a branch no CI gate watches (and the branch is unused). Deleting aztec-nightlies.yml removes the only caller that could dodge the Windows gate — only the gated stable → main path remains.
The honest part. This does NOT make bb.exe cryptographically verifiable — that needs Aztec to sign their releases (the same upstream-signing gap F-007 hit, tracked as SEC-02). What it DOES do: stop the automation from silently accepting + merging an unverified binary, and make a human look at every new Windows pin. That's the achievable win today.

Phases & how each proves itself

Phase 1 — structured provenance + remove auto-pin

Proves itself by: bun test packages/accelerator/scripts/ (resolver accepts manual-review; rejects missing / reserved-attestation / unrecognized / malformed) + bun test scripts/update-aztec-version.test.ts (no pin logic, no network) + lint.

Phase 2 — leave-PR-open + post-install pin check + least-privilege + delete nightlies

Proves itself by: check-windows-bb-pin test (present vs required, no network) + lint:actions + a static check that the none path never calls gh pr merge + an invalid-forced-input guard.

Phase 3 — provenance policy docs + ruleset runbook

Proves itself by: full bun run test + lint:actions.

Open questions (decided)

  • Block all Windows, or keep current pins usable? Keep them usable as honestly-labeled change-detectors — hard-blocking would break Windows with no path forward until Aztec signs. The block applies to the automation, not to existing reviewed pins.
  • Is the CI gate actually enforced? It depends on the C5 main ruleset being live (human- applied) with the release-bot not on the bypass list — a documented human-gated dependency + a runbook readback, not something the repo can prove by itself.
