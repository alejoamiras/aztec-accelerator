C8 — Two desktop hardening fixes (both defense-in-depth)

F-010 (autostart path injection). When the app enables "start on login", it writes the path to its own binary into OS launcher files (a Linux systemd unit, a macOS plist, a Windows Run key). Today the systemd unit does ExecStart="{path}" with no proper escaping — a binary installed at a weird path containing a newline could inject extra launcher directives. The audits showed systemd actually rejects paths with quotes/backslashes/globs, so the fix is: build a strictly-escaped ExecStart=":/path%%" and reject (fail closed) any path systemd can't safely run; a preflight refuses to enable autostart at all for a newline/control/non-UTF-8 path (which would break every OS launcher), and we remove the raw webview permission so the only enable path goes through that gated check. Honest scope: the third-party autostart crate's per-platform formatting quirks (space-splitting) are a documented residual we can't fix without patching the crate; the injection vector (newlines) is closed everywhere.

F-016 (CA key not scrubbed). To serve HTTPS on localhost (Safari), the app mints a short-lived CA, signs a leaf cert, and throws the CA key away — but a plain drop leaves the key bytes in freed memory. Fix: wrap the CA key in Zeroizing (rcgen already supports it) and drop it right after signing. Honest residual: this wipes only rcgen's copy of the key, not the underlying crypto-backend's copy, and can't touch swap or a core dump — best-effort, not "the key is gone". The CA is loopback-name-constrained and recovering it needs process-memory access (an attacker who already won).

Why mid→deep: both findings are honestly LOW/defense-in-depth, but the F-010 fix, done right, touches the systemd serializer + a cross-platform preflight + a capability permission + fail-closed propagation through 3 callers with rollback. Three audit rounds (codex + fable dual + 2 finals) vetted it.
