Locking the desktop app's "trust boundary"

The Accelerator desktop app shows a few small windows — the site-approval popup, the settings panel, the update prompt. Each can talk to the app's Rust core through a set of 12 privileged commands. Right now any window can call any command — the approval popup could, in principle, silently turn on Safari support or forget an approved site. This plan closes that: each window may call only its own commands, hostile injected scripts are blocked, and the whole thing is proven by tests that fail loudly if the wall ever has a hole.

The one thing to understand

Tauri (the framework) has a permission system, but we discovered — and then proved in Tauri's own source code — that it does not guard your app's own commands unless you explicitly declare them. Today we don't, so all 12 commands are wide open to every window. The fix is a switch: once we declare the commands in the build script, Tauri flips to "deny unless this window was explicitly granted this command." We add three independent layers so the wall holds even if one brick is loose:

  • Per-window permissions (the framework's own wall) — proven real, not decorative.
  • A Rust identity check on each command ("are you the window I expect?") — a second wall the framework can't bypass, and it also ties the approval popup to the exact request it's answering.
  • A strict Content-Security-Policy — blocks injected scripts, inline code, eval, and sneaky ways of leaking data out (network calls, page navigation, form posts).

What we'll do, step by step

P0 — Add the dependency. Pull in the official @tauri-apps/api at an exact pinned version so the frontend can call commands without the global back-door.
Proves itself by: a clean frozen-lockfile install + lint.
P1 — Move all code out of the HTML. Today each popup has its script written inline in the page. We bundle those into real files and delete the inline blocks and the old bridge.
Proves itself by: the bundle builds, the mock UI tests pass, a static test confirms zero inline scripts/styles remain.
P2 — Turn on the strict security policy. Remove the global back-door and switch on the Content-Security-Policy plus Rust guards that block navigating or posting a form to an attacker.
Proves itself by: real-browser tests where injected scripts, eval, off-site fetches, navigations and form posts are all blocked — but normal app calls still work.
P3 — Flip the permission switch (the big one). Declare the 12 commands and give each window only its own. This is all-or-nothing, so we run the full positive test suite (every real flow must still work) plus a test that proves a window is refused another window's command — and that no side effect happened.
Proves itself by: a real-build ("custom-protocol") test lane on Linux + the 3-OS suite + an exact per-window permission-matrix static check.
P4 — Add the Rust identity check. Each command confirms it was called by the window it belongs to; the approval popup additionally confirms it's answering the exact request it was opened for.
Proves itself by: Rust unit tests for the check + the cross-window refusal still holding.
P5 — Lock it in. Wire the static drift-detector tests into the normal lint/test run so a future change that weakens any of this fails CI.
Proves itself by: the whole 3-OS pipeline green.

Two judgment calls the audits disagreed on (and how we settled them)

Should each window keep the framework's "core" default permissions? → No, drop them. It turns out those grants currently apply to no window at all (they're dormant), so keeping them while adding the new per-window rules would actually hand out new powers the popups never had. The popups are closed by Rust and use no such APIs, so we drop them and add back only a specific one if a test ever proves it's needed.
Is the normal test mode enough to prove the shipped app is safe? → No — add a real-build test lane. We found the development test mode never runs the exact code path that ships to users. So the key security test also runs against a true production-style build (on Linux, to keep CI cheap), plus a guard that stops anyone silently weakening the policy in dev.
Open question for you: the real-build test lane costs extra CI time. The plan runs it on Linux only (the trust-boundary test), keeping the full 3-OS matrix on the fast dev mode. If you'd rather run it on all three OSes for maximum parity, say so — otherwise Linux-only is the default.

What could still go wrong (and the guards)

  • A future change quietly weakens the wall (a new command not listed, a relaxed policy, a new window). → Static "drift-detector" tests fail CI.
  • The webview engine itself has a bug. → Out of scope; no CSP can stop a native flaw. Documented as residual.
  • A bundled script file goes stale or ships unreviewed code. → Clean-before-build, a staleness guard in the build script, and a scan of the emitted bundles.
