Security hardening — the localhost gate, the updater, and the certs

The accelerator runs a tiny web server on your own machine that web pages talk to, so they can offload heavy proving to it. Our security audit found that a malicious website could sneak past the "who's allowed?" check, and a few other sharp edges around downloading the proving binary and managing certificates. This plan fixes 8 of the 9 findings (the 9th needs the Aztec team to sign their releases first — deferred with a note). It ships as 5 small pull requests, safest change first.

Why this tier was chosen (deep)

Two risk dimensions scored HIGH: blast radius (we're changing the security gate on an app that auto-updates real users — a mistake locks out legit sites or opens the door) and security sensitivity (it's literally a security plan on the trust boundary). Two HIGH ⇒ deep. That earned its keep: three independent planners agreed on the shape, then two adversarial auditors (codex + a fresh Opus reviewer) each found real problems — including one Critical hole the planners had completely missed — which we folded in before asking for your approval.

Phases (the 5 PRs)

PR-1 — The keystone: check the "Host" label

The attack ("DNS rebinding") works because the server only ever looked at one label on incoming requests (the "Origin") and never the "Host". We add a check that rejects any request whose Host isn't literally your own machine (127.0.0.1/localhost/[::1]) on the exact port. This one check kills the main attack, and — importantly — it breaks nothing, because every real client already sends the right Host. Ships first, alone, low-risk.

PR-2 — Close the gate properly: Origin, headless default, /health, and localhost

Build on PR-1: stop the headless server from approving everyone when it's started with no allowlist (you opt back into that with --allow-all); shrink what the public /health endpoint tells random websites (version, cache) while still letting your own tools read the details; and — the Critical the audit caught — stop silently trusting any page running on localhost. A bad local program shouldn't get free proving with no prompt.

PR-3 — Give each approval its own ticket

Today an Approve/Deny is matched to a request by the site's name. We switch to a unique, unguessable ticket per request, so a malformed reply can't silently miss and two requests from the same site can't share one decision.

PR-4 — Don't let the updater eat all your memory

The self-updater loads the whole download into memory before checking its signature. We add a size limit up front so a tampered feed can't point it at a giant file. (The audit talked us out of a fancier rewrite here — re-implementing the signature check ourselves would have been riskier than the problem it solved.)

PR-5 — Tidy the download and certificate edges

Cap how much the proving-binary archive can expand to (anti "zip bomb"); if an old leftover certificate key can't be deleted, refuse to start Safari HTTPS rather than limping on; and verify a new certificate properly chains before swapping it in. Plus a code note + tracking issue for the deferred bb-signature finding (the real fix needs Aztec to sign their nightly releases).

Human context

"DNS rebinding" is the classic way a website attacks a service running on your own computer: it loads a page from evil.com, then quietly re-points evil.com at 127.0.0.1 so its next request lands on your local server while the browser still thinks it's the same site. The fix is to check the Host label, which still says evil.com — so we reject it. Everything else in this plan is defense-in-depth layered behind that one check. None of these were caused by the recent quality-fixes work; they're standing issues the audit surfaced.

To close the "any local page is silently trusted" hole, the default becomes: a localhost page gets one approval prompt (then remembered), instead of silent auto-approval.
