Headless CI Slim — the core-extraction follow-up

We already made the headless server build 56% lighter (no desktop GUI code). But the CI robots that build it still act like they're building the full desktop app — they install the Linux desktop GUI libraries (WebKit/GTK) and run the desktop "prebuild" step. This plan tells the **two CI jobs that only build + health-check the server** (not the proving ones) to skip that desktop busywork, so they're leaner and faster. Plus two small correctness tidies the post-extraction audit flagged.

Why this tier was chosen (/blueprint mid)

It's a contained CI-config change, but it brushes the release-adjacent jobs (one high dimension: blast radius), so it earns the dual audit — codex + an opus architect both reviewed it and both picked the same approach, then caught real defects (a CI step that would've started failing, a wrong file path, a test we wrongly assumed existed). All fixed before approval.

Phases

Phase 1 — A "just the version" helper

The desktop prebuild does two jobs: copy the `bb` binary AND figure out its version. The slimmed jobs only need the version. So we pull the version-figuring bit into its own tiny, tested helper the lean jobs can call without copying anything.

Phase 2 — Add on/off switches to the shared setup

The CI "setup recipe" is shared by everyone. We add two switches (default = "on", so nothing changes for anyone) that let a job say "skip the desktop GUI libraries" and "skip the prebuild."

Phase 3 — Flip the switches on the two lean jobs (+ the tidies)

Smoke and Release-Smoke turn both switches off. Smoke gets its version from the new helper instead of the (now-absent) prebuild file. The audit's two correctness fixes land here: the proving e2e test gets the right version so it doesn't do a needless download, and we add a sanity check on the reported version.

Phase 4 — Notes

Document the leftover cleanup (the same GUI-library list still lives in a couple of other places) and mark this done.

Human context

The proof here is the CI run: this very change makes the Smoke / Release-Smoke / e2e jobs run on its own pull request, so a green check literally proves the slim works — no separate release rehearsal needed. One expected, harmless change: the lean Smoke job's health check will now report bb_available: false (because we stopped copying `bb` there) — that's correct, and nothing depends on it.
